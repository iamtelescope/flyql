O FlyQL suporta **placeholders de parâmetros** — `$name` (nomeado) e `$1` (posicional) — que podem substituir valores em qualquer lugar onde um literal normalmente apareceria. Os parâmetros são resolvidos em tempo de execução chamando `bindParams()` antes de gerar o SQL ou avaliar a consulta em memória.

Esta é a forma segura de criar templates de consultas: você mantém o texto da consulta estático e faz o bind de valores concretos depois. Não há concatenação de strings, nem escape manual, nem risco de injeção.

## Por Que Parâmetros

Um caso de uso típico é uma ferramenta ou dashboard que executa a mesma forma de consulta com diferentes entradas:

<FlyqlCode code={`status = $code and env = $env`} />

```python
from flyql import parse, bind_params

ast = parse("status = $code and env = $env")
bind_params(ast.root, {"code": 200, "env": "prod"})
```

Os valores dos parâmetros podem vir de campos de formulário, requisições de API ou qualquer outra fonte em tempo de execução — sem nunca reconstruir a string da consulta.

## Sintaxe

### Parâmetros Nomeados

Um parâmetro nomeado é `$` seguido por uma letra ou underscore, e então qualquer combinação de letras, dígitos e underscores:

<FlyqlCode code={`status = $code`} />
<FlyqlCode code={`env = $my_env`} />
<FlyqlCode code={`user = $_internal`} />

O nome do parâmetro **não** inclui o prefixo `$` — `$code` referencia um parâmetro chamado `code`.

### Parâmetros Posicionais

Um parâmetro posicional é `$` seguido apenas por dígitos. Parâmetros posicionais são **indexados a partir de 1** (como `$1` no SQL):

<FlyqlCode code={`a = $1 and b = $2`} />

`$0` não é permitido. `$1abc` (dígitos seguidos de letras) não é permitido.

## Onde os Parâmetros Podem Aparecer

Parâmetros funcionam em todas as posições onde um valor literal é válido:

### Valores de Comparação

<FlyqlCode code={`status = $code`} />
<FlyqlCode code={`age >= $min_age`} />
<FlyqlCode code={`name != $banned`} />
<FlyqlCode code={`score > $1`} />

### `has`, `like`, `ilike`

<FlyqlCode code={`message has $needle`} />
<FlyqlCode code={`path like $pattern`} />
<FlyqlCode code={`title ilike $search`} />
<FlyqlCode code={`tags not has $excluded`} />

### Listas IN

Parâmetros podem ser misturados com valores literais dentro de listas `in` e `not in`:

<FlyqlCode code={`status in [$x, $y]`} />
<FlyqlCode code={`env in [$current, 'staging', 'dev']`} />
<FlyqlCode code={`code not in [$banned, 401, 403]`} />

### Argumentos de Funções Temporais

Parâmetros podem substituir valores de duração dentro de `ago()`, argumentos de fuso horário dentro de `today()` e `startOf()`:

<FlyqlCode code={`created > ago($duration)`} />
<FlyqlCode code={`date = today($tz)`} />
<FlyqlCode code={`created > startOf('day', $tz)`} />

O valor ligado a um parâmetro de `ago()` deve ser uma string de duração como `"5m"`, `"1h"` ou `"7d"`.

### Argumentos de Transformadores

Parâmetros podem aparecer dentro de posições de argumentos de transformadores:

<FlyqlCode code={`tags|split($sep) has $needle`} />

## bindParams()

Após o parsing, chame `bindParams()` para substituir os placeholders de parâmetros por valores concretos. A função percorre a AST in-place e substitui todos os nós `Parameter` que encontra.

### Python

```python
from flyql import parse, bind_params

ast = parse("status = $code and env = $env")
bind_params(ast.root, {"code": 200, "env": "prod"})
```

### JavaScript

```javascript
import { parse, bindParams } from 'flyql'

const ast = parse('status = $code and env = $env')
bindParams(ast.root, { code: 200, env: 'prod' })
```

### Go

```go
import flyql "github.com/iamtelescope/flyql/golang"

ast, _ := flyql.Parse(`status = $code and env = $env`)
err := flyql.BindParams(ast.Root, map[string]any{
    "code": 200,
    "env":  "prod",
})
```

### API

| Linguagem | Assinatura |
|----------|-----------|
| Python | `bind_params(node, params: dict) -> Node` |
| JavaScript | `bindParams(node, params: object): Node` |
| Go | `BindParams(node *Node, params map[string]any) error` |

- **`node`** — o `Node` raiz retornado por `parse()`.
- **`params`** — um dict/objeto/map mapeando nomes de parâmetros a valores concretos. Parâmetros nomeados usam o nome como chave (ex.: `"code"`). Parâmetros posicionais usam chaves de string com dígitos (ex.: `"1"`, `"2"`). Dicts mistos são permitidos.
- **Valor de retorno** — Python e JS retornam o mesmo `Node` (mutado in-place) para encadeamento. Go retorna um `error`.

### Mapeamento de Tipos

O tipo nativo do valor ligado determina o `value_type` resultante na AST:

| Tipo nativo | `value_type` resultante |
|-------------|------------------------|
| `string` | `"string"` |
| `int` (dentro do intervalo int64) | `"integer"` |
| `int` (fora do intervalo int64) | `"bigint"` |
| `float` | `"float"` |
| `bool` | `"boolean"` |
| `null` / `None` / `nil` | `"null"` |

O tipo é determinado pelo **tipo em tempo de execução do valor ligado**, não por parsing do valor como string. Fazer bind de `"200"` produz uma string, não um inteiro:

```python
bind_params(parse("a = $x").root, {"x": "200"})
# expression.value      → "200"
# expression.value_type → "string"

bind_params(parse("a = $x").root, {"x": 200})
# expression.value      → 200
# expression.value_type → "integer"
```

Tipos não suportados (ex.: `Decimal`, `BigInt`, objetos customizados, `undefined`) geram um erro:

```
unsupported parameter value type: Decimal
```

## Validação Estrita

`bindParams()` é **estrito** tanto com parâmetros ausentes quanto com parâmetros extras.

### Parâmetros Ausentes

Se a consulta referencia um parâmetro que não está no dict fornecido, um erro é gerado:

```python
bind_params(parse("a = $x").root, {})
# FlyqlError: unbound parameter: $x
```

### Parâmetros Extras

Se o dict contém um parâmetro que não é referenciado na consulta, um erro é gerado:

```python
bind_params(parse("a = $x").root, {"x": 1, "y": 2})
# FlyqlError: unused parameter: y
```

Isso é mais estrito que drivers SQL típicos, que frequentemente postergam a validação de parâmetros para o momento da execução da consulta. Capturar esses erros no momento do bind mantém os enganos visíveis.

### Parâmetros Posicionais

Para parâmetros posicionais, todo índice de `1` até o índice máximo referenciado deve ser fornecido. Lacunas não são permitidas:

```python
bind_params(parse("a = $1 and b = $3").root, {"1": 1, "3": 3})
# FlyqlError: unbound parameter: $2
```

## Matching em Memória

Parâmetros funcionam da mesma forma com o matcher. Faça o bind primeiro, depois avalie:

```python
from flyql import parse, bind_params
from flyql.matcher.evaluator import Evaluator

ast = parse("status = $code")
bind_params(ast.root, {"code": 200})

result = Evaluator().evaluate(ast.root, {"status": 200})
```

Se você esquecer de chamar `bind_params()` antes de avaliar, o matcher gera um erro claro:

```
unbound parameter '$code' — call bind_params() before evaluating
```

A mesma proteção existe em cada gerador SQL, então você não pode acidentalmente produzir SQL com parâmetros não resolvidos.

## Representação na AST

Um parâmetro é representado na AST como uma `Expression` com `value_type: "parameter"` e um objeto de valor `Parameter`:

Consulta: <FlyqlCode code="status = $code" inline />

```
Node (leaf)
  expression:
    key: { segments: ["status"], raw: "status" }
    operator: "="
    value: { name: "code", positional: false }
    value_type: "parameter"
```

Para parâmetros em listas IN, o objeto `Parameter` aparece dentro de `expression.values`:

Consulta: <FlyqlCode code="status in [$x, 'ok']" inline />

```
Node (leaf)
  expression:
    key: { segments: ["status"], raw: "status" }
    operator: "in"
    values: [{ name: "x", positional: false }, "ok"]
    values_types: ["parameter", "string"]
```

Para parâmetros em funções temporais, eles aparecem em `FunctionCall.parameter_args`:

Consulta: <FlyqlCode code="created > ago($d)" inline />

```
Node (leaf)
  expression:
    key: { segments: ["created"], raw: "created" }
    operator: ">"
    value:
      name: "ago"
      duration_args: []
      parameter_args: [{ name: "d", positional: false }]
    value_type: "function"
```

Após `bindParams()` resolvê-los, esses placeholders `Parameter` são substituídos por valores concretos e a AST fica parecida com uma consulta normalmente parseada.

## Erros

| Erro | Errno | Quando |
|-------|-------|------|
| `empty parameter name` | 74 | Consulta contém `$` sem nome (ex.: `status = $`) |
| `invalid parameter name` | 75 | Nome começa com dígito mas contém não-dígitos (ex.: `$1abc`) |
| `positional parameters are 1-indexed` | 76 | Posicional `$0` (ex.: `status = $0`) |

`bindParams()` gera `FlyqlError` (ou retorna um erro em Go) para:

- `unbound parameter: $name` — referenciado na consulta, ausente do dict
- `unused parameter: name` — fornecido no dict, não referenciado na consulta
- `unsupported parameter value type: <type>` — o valor ligado não é um dos tipos suportados
