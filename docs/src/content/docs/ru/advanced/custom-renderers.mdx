description: Регистрация определений рендереров и хуков валидации для вашего приложения
---

:::note
Рендереры никогда не выполняются внутри flyql -- ваше приложение отвечает за
интерпретацию `parsed_column.renderers` и отображение значения.
:::

:::note
`metadata` -- произвольный объект, определяемый разработчиком. Ядро flyql и редактор
никогда не читают из него конкретные ключи. Если вам нужны стандартизированные поля,
определите их в вашем приложении и задокументируйте рядом с зарегистрированными рендерерами.
:::

См. [страницу синтаксиса рендереров](/syntax/renderers/) для грамматики и
семантики. Эта страница описывает API регистрации для добавления собственных
рендереров в валидатор и редактор.

## Интерфейс Renderer

`Renderer` объявляет:

- **`name`** -- идентификатор, который пользователи вводят после пайпа алиаса
- **`arg_schema`** -- кортеж записей `ArgSpec(type, required)`, управляющих
  валидацией количества и типов аргументов
- **`metadata`** -- необязательный произвольный dict/map/объект для использования
  приложением; flyql никогда не инспектирует его содержимое
- **`diagnose(args, parsed_column)`** -- необязательный хук для каждого рендерера,
  возвращающий `Diagnostic[]` для пользовательской валидации

## Пример на Python

```python
from flyql.flyql_type import Type
from flyql.renderers import ArgSpec, Renderer, RendererRegistry
from flyql.columns import parse, diagnose
from flyql.core.validator import Diagnostic
from flyql.core.range import Range


class HrefRenderer(Renderer):
    arg_schema = (ArgSpec(type=Type.String, required=True),)

    @property
    def name(self) -> str:
        return "href"

    @property
    def metadata(self):
        return {"html_safe": True}

    def diagnose(self, arguments, parsed_column):
        if arguments and "{{value}}" not in arguments[0]:
            return [Diagnostic(
                range=Range(0, 1),
                message="href template should contain {{value}}",
                severity="warning",
                code="custom_href_no_placeholder",
            )]
        return []


registry = RendererRegistry()
registry.register(HrefRenderer())

cols = parse('url as link|href("/users/{{value}}")',
             capabilities={"transformers": True, "renderers": True})

# ...затем запустите валидатор с обоими реестрами:
diags = diagnose(cols, schema, renderer_registry=registry)
```

## Хук цепочки на уровне реестра

Для правил на уровне цепочки (например, "запретить `href` после `truncate`") используйте
`set_diagnose`:

```python
def chain_rule(parsed_column, chain):
    names = [r["name"] for r in chain]
    if "truncate" in names and "href" in names:
        if names.index("truncate") < names.index("href"):
            return [Diagnostic(
                range=Range(0, 1),
                message="href cannot follow truncate",
                severity="error",
                code="chain_forbidden",
            )]
    return []

registry.set_diagnose(chain_rule)
```

Хук цепочки выполняется после хуков `diagnose` каждого рендерера и после
встроенных проверок неизвестного имени / количества аргументов / типа аргумента.

## Пример на Go

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    "github.com/iamtelescope/flyql/golang/columns"
    "github.com/iamtelescope/flyql/golang/flyqltype"
    "github.com/iamtelescope/flyql/golang/transformers"
)

type HrefRenderer struct{ columns.BaseRenderer }

func (HrefRenderer) Name() string { return "href" }
func (HrefRenderer) ArgSchema() []transformers.ArgSpec {
    return []transformers.ArgSpec{{Type: flyqltype.String, Required: true}}
}

reg := columns.DefaultRendererRegistry()
_ = reg.Register(HrefRenderer{})

opts := columns.DiagnoseOptions{RendererRegistry: reg}
diags := columns.DiagnoseWithOptions(parsedColumns, schema, opts)
```

## Пример на JavaScript

```javascript
import { Renderer, RendererRegistry, ArgSpec } from 'flyql/renderers'
import { Type } from 'flyql'
import { parse, diagnose } from 'flyql/columns'

class HrefRenderer extends Renderer {
    get name() { return 'href' }
    get argSchema() { return [new ArgSpec(Type.String, true)] }
    diagnose(args, col) {
        if (args && args[0] && !args[0].includes('{{value}}')) {
            return [{
                range: { start: 0, end: 1 },
                message: 'href template should contain {{value}}',
                severity: 'warning',
                code: 'custom_href_no_placeholder',
            }]
        }
        return []
    }
}

const registry = new RendererRegistry()
registry.register(new HrefRenderer())

const cols = parse('url as link|href("/users/{{value}}")',
    { transformers: true, renderers: true })

const diags = diagnose(cols, schema, null, registry)
```

## Подключение к редактору flyql-vue

Vue-компонент `FlyqlColumns` принимает проп `rendererRegistry`. При его
наличии редактор включает разбор рендереров после алиаса, выдаёт подсказки
для рендереров и отображает диагностики рендереров:

```vue
<template>
  <FlyqlColumns
    v-model="query"
    :columns="schema"
    :registry="transformerRegistry"
    :renderer-registry="rendererRegistry"
  />
</template>
```

Когда `rendererRegistry` опущен или равен `null`, разбор рендереров полностью
отключён -- пользователи, вводящие `|` после алиаса, увидят отклонение на уровне
парсера, а не диагностики `unknown_renderer` при каждом нажатии клавиши. См.
[Компонент Columns](/editor/columns-component/) для полного описания пропов.
