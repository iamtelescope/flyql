## Boolean Operators

FlyQL uses three boolean operators to combine conditions:

| Operator | Meaning | Example |
|----------|---------|---------|
| <FlyqlToken type="operator">and</FlyqlToken> | Both conditions must be true | <FlyqlCode code={`status = 200 and active`} inline /> |
| <FlyqlToken type="operator">or</FlyqlToken> | Either condition can be true | <FlyqlCode code={`level = 'error' or level = 'critical'`} inline /> |
| <FlyqlToken type="operator">not</FlyqlToken> | Negates the immediately following expression or group | <FlyqlCode code={`not archived`} inline /> |

### Combined Negation

<FlyqlToken type="operator">not</FlyqlToken> can be combined with <FlyqlToken type="operator">and</FlyqlToken> and <FlyqlToken type="operator">or</FlyqlToken>:

<FlyqlCode code={`active and not archived`} />
<FlyqlCode code={`level = 'error' or not healthy`} />

## Operator Precedence

FlyQL follows **standard operator precedence** — the same rules as SQL, Python, JavaScript, Go, and every mainstream query language:

| Precedence | Operator | Binds |
|-----------:|----------|-------|
| 1 (tightest) | <FlyqlToken type="operator">not</FlyqlToken> | To its immediate operand — the following expression or parenthesized group |
| 2 | <FlyqlToken type="operator">and</FlyqlToken> | Tighter than <FlyqlToken type="operator">or</FlyqlToken> |
| 3 (loosest) | <FlyqlToken type="operator">or</FlyqlToken> | Loosest — combines whole AND-chains |

Concretely:

<FlyqlCode code={`a = 1 or b = 2 and c = 3`} />

is parsed as:

<FlyqlCode code={`a = 1 or (b = 2 and c = 3)`} />

because <FlyqlToken type="operator">and</FlyqlToken> binds tighter than <FlyqlToken type="operator">or</FlyqlToken>. Same-precedence chains are left-associative, so <FlyqlCode code={`a = 1 and b = 2 and c = 3`} inline /> parses as <FlyqlCode code={`(a = 1 and b = 2) and c = 3`} inline />.

<FlyqlToken type="operator">not</FlyqlToken> binds tightest and applies only to the expression or group that follows it — never the entire preceding expression.

## Parenthesized Grouping

Use parentheses to override the default precedence when you want <FlyqlToken type="operator">or</FlyqlToken> evaluated before <FlyqlToken type="operator">and</FlyqlToken>:

<FlyqlCode code={`(status >= 400 or level = 'error') and service = 'api'`} />

Without the parentheses, the same source

<FlyqlCode code={`status >= 400 or level = 'error' and service = 'api'`} />

parses as

<FlyqlCode code={`status >= 400 or (level = 'error' and service = 'api')`} />

which is a different query. When you want <FlyqlToken type="operator">or</FlyqlToken> to dominate, reach for parentheses explicitly.

### Nested Grouping

Parentheses can be nested:

<FlyqlCode code={`(a = 1 or b = 2) and not (c = 3 and d = 4)`} />

### Negating Groups

<FlyqlToken type="operator">not</FlyqlToken> applies to the entire group:

<FlyqlCode code={`not (status = 200 and healthy)`} />

This negates the combined condition — true only when either <FlyqlToken type="key">status</FlyqlToken> is not 200 or <FlyqlToken type="key">healthy</FlyqlToken> is false.

## Negation

<FlyqlToken type="operator">not</FlyqlToken> negates any expression — a truthy check, a comparison, or a group:

<FlyqlCode code={`not active`} />
<FlyqlCode code={`not status = 200`} />
<FlyqlCode code={`not (a = 1 and b = 2)`} />
<FlyqlCode code={`active and not archived`} />

Double negation cancels out: <FlyqlCode code={`not not active`} inline /> is equivalent to <FlyqlCode code={`active`} inline />.

## Examples

<FlyqlCode code={`status = 200 and service = 'api'`} />
<FlyqlCode code={`level = 'error' or level = 'critical'`} />
<FlyqlCode code={`status >= 400 or level = 'error' and service = 'api'`} />
<FlyqlCode code={`(status >= 400 or level = 'error') and service = 'api'`} />
<FlyqlCode code={`(status >= 500 and host = 'prod') or (level = 'error' and not service = 'healthcheck')`} />
