## Pertenencia a listas

Usa <FlyqlToken type="operator">in</FlyqlToken> para comprobar si un valor está en una lista y <FlyqlToken type="operator">not in</FlyqlToken> para comprobar que no lo está:

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['prod', 'staging']`} />

## Reglas de sintaxis

- Los valores van entre corchetes <FlyqlToken type="operator">[ ]</FlyqlToken> y se separan con comas
- Los valores pueden mezclar tipos: <FlyqlCode code={`[200, 'ok', true, null]`} inline />
- Los strings deben ir entre comillas: <FlyqlCode code={`['a', 'b', 'c']`} inline />
- Los valores numéricos no llevan comillas: <FlyqlCode code={`[1, 2, 3]`} inline />
- Se permiten espacios alrededor de comas y corchetes

### Listas de strings

<FlyqlCode code={`method in ['GET', 'POST', 'PUT']`} />
<FlyqlCode code={`env not in ['dev', 'test', 'local']`} />

Se pueden usar comillas simples o dobles para los strings.

### Listas numéricas

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`port not in [80, 443]`} />

### Listas vacías

Se permiten listas vacías:

<FlyqlCode code={`status in []`} />

Una lista <FlyqlToken type="operator">in</FlyqlToken> vacía es siempre false — nada coincide con una lista vacía.

<FlyqlCode code={`status not in []`} />

Una lista <FlyqlToken type="operator">not in</FlyqlToken> vacía es siempre true — todo está "not in" en una lista vacía.

## Tipos mezclados

Las listas admiten valores heterogéneos: puedes mezclar strings, números, booleanos y null:

<FlyqlCode code={`field in [200, 'ok', true, null]`} />

Cada elemento lleva su propio tipo en el array `valuesTypes` de la expresión parseada (por ejemplo `["int", "string", "bool", "null"]`). En listas vacías, `valuesTypes` es `null`.

Cuando un generador SQL procesa una lista, valida que los tipos de los valores de la lista sean compatibles con el tipo de la columna de destino. Por ejemplo, pasar valores de tipo string a una columna numérica lanzará un error de validación durante la generación de SQL.

## Consistencia de tipos en listas IN

El parser **siempre** acepta listas heterogéneas — la validación de tipos vive en los generadores SQL, no en el parser. Cada generador (ClickHouse, PostgreSQL, StarRocks) aplica la misma lógica de salto de dos capas antes de lanzar un error.

### Capa 1 — control en el sitio de llamada

Antes de invocar al validador, el generador comprueba tres condiciones:

1. La columna tiene un **tipo declarado** (`column.flyqlType()` no está vacío y no es `Unknown`).
2. La clave **no es segmentada** (un único segmento como `category`, no `meta.category`).
3. La lista es **homogénea** — todos los elementos tienen el mismo tipo de valor (`len(set(values_types)) == 1` en Python; `new Set(valuesTypes).size === 1` en JavaScript; equivalente en Go).

Si **alguna** de estas condiciones es falsa, el validador se omite por completo y el SQL se genera tal cual. Las listas heterogéneas pasan. Las listas contra columnas sin tipo pasan. Las listas contra claves segmentadas (rutas JSON, índices de array) pasan.

### Capa 2 — comprobación de elementos

Cuando el control en el sitio de llamada se cumple, el validador itera por cada elemento y lanza error en el primer valor incompatible, según este mapa de compatibilidad:

| Tipo de columna | Tipos de elemento permitidos |
|-----------------|------------------------------|
| `String`        | `string`                     |
| `Int`           | `int`, `float`               |
| `Float`         | `int`, `float`               |
| `Bool`          | `bool`, `int`                |
| `Date`          | `string`                     |

Otros tipos de columna (`Array`, `Map`, `JSON`, `JSONString`, `Struct`, `Duration`, `Unknown`) no están en el mapa y omiten silenciosamente la comprobación de elementos.

### El texto del error

Cuando la validación falla, las tres implementaciones lanzan el mensaje de error **idéntico**:

<FlyqlCode code={`type mismatch in IN list: {column_type} column cannot contain {value_type} values`} />

Concretamente:

- Python: `FlyqlError("type mismatch in IN list: int column cannot contain string values")`
- Go: `errors.New("type mismatch in IN list: int column cannot contain string values")`
- JavaScript: `new Error("type mismatch in IN list: int column cannot contain string values")`

El subtexto `type mismatch in IN list:` es estable en los tres lenguajes y los tres dialectos SQL, así que es seguro hacer grep por él.

Para una demostración ejecutable end-to-end de los tres casos (coincidencia homogénea, bypass heterogéneo, error de tipo incorrecto homogéneo), ver [Recetas — Consistencia de tipos en listas IN](/es/syntax/recipes/#consistencia-de-tipos-en-listas-in).

## Parámetros en listas

Los marcadores de parámetros pueden aparecer dentro de listas IN, mezclados libremente con valores literales:

<FlyqlCode code={`status in [$x, $y, 200]`} />
<FlyqlCode code={`env not in [$current, 'dev']`} />

Los objetos `Parameter` permanecen en el array `values` hasta que `bindParams()` los resuelve. Consulta [Parámetros](/es/syntax/parameters/) para ver la referencia completa.

## Combinación con otros operadores

Los operadores de lista se combinan con los operadores booleanos como cualquier otra condición:

<FlyqlCode code={`status in [200, 201] and method in ['GET', 'POST']`} />
<FlyqlCode code={`level = 'error' or status not in [200, 201, 204]`} />
<FlyqlCode code={`(status in [500, 502, 503] or level = 'critical') and env='prod'`} />

## Ejemplos

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['dev', 'test', 'staging']`} />
<FlyqlCode code={`method in ['GET', 'POST'] and status not in [401, 403, 404]`} />
