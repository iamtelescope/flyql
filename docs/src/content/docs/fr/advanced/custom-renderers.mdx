:::note
Les renderers ne s'exécutent jamais à l'intérieur de flyql — votre application est responsable de l'interprétation de `parsed_column.renderers` et du rendu de la valeur.
:::

:::note
`metadata` est libre et défini par le développeur. Le cœur de flyql et l'éditeur ne lisent jamais de clé spécifique. Si vous voulez des champs standardisés, définissez-les dans votre propre application et documentez-les avec vos renderers enregistrés.
:::

Consultez la [page de syntaxe des Renderers](/syntax/renderers/) pour la grammaire et la sémantique. Cette page couvre l'API d'enregistrement pour ajouter vos propres renderers au validateur et à l'éditeur.

## L'interface Renderer

Un `Renderer` déclare :

- **`name`** — l'identifiant que les utilisateurs tapent après le pipe de l'alias
- **`arg_schema`** — un tuple d'entrées `ArgSpec(type, required)` qui pilotent la validation du nombre et du type d'arguments
- **`metadata`** — un dict/map/objet optionnel libre pour l'usage applicatif ; flyql n'inspecte jamais son contenu
- **`diagnose(args, parsed_column)`** — un hook optionnel par renderer retournant `Diagnostic[]` pour la validation personnalisée

## Exemple Python

```python
from flyql.flyql_type import Type
from flyql.renderers import ArgSpec, Renderer, RendererRegistry
from flyql.columns import parse, diagnose
from flyql.core.validator import Diagnostic
from flyql.core.range import Range


class HrefRenderer(Renderer):
    arg_schema = (ArgSpec(type=Type.String, required=True),)

    @property
    def name(self) -> str:
        return "href"

    @property
    def metadata(self):
        return {"html_safe": True}

    def diagnose(self, arguments, parsed_column):
        if arguments and "{{value}}" not in arguments[0]:
            return [Diagnostic(
                range=Range(0, 1),
                message="href template should contain {{value}}",
                severity="warning",
                code="custom_href_no_placeholder",
            )]
        return []


registry = RendererRegistry()
registry.register(HrefRenderer())

cols = parse('url as link|href("/users/{{value}}")',
             capabilities={"transformers": True, "renderers": True})

# ...puis lancez le validateur avec les deux registres :
diags = diagnose(cols, schema, renderer_registry=registry)
```

## Hook de chaîne au niveau du registre

Pour les règles au niveau de la chaîne (par ex. « interdire `href` après `truncate` ») utilisez `set_diagnose` :

```python
def chain_rule(parsed_column, chain):
    names = [r["name"] for r in chain]
    if "truncate" in names and "href" in names:
        if names.index("truncate") < names.index("href"):
            return [Diagnostic(
                range=Range(0, 1),
                message="href cannot follow truncate",
                severity="error",
                code="chain_forbidden",
            )]
    return []

registry.set_diagnose(chain_rule)
```

Le hook de chaîne s'exécute après les hooks `diagnose` par renderer et après les vérifications intégrées de nom inconnu / nombre d'arguments / type d'arguments.

## Exemple Go

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    "github.com/iamtelescope/flyql/golang/columns"
    "github.com/iamtelescope/flyql/golang/flyqltype"
    "github.com/iamtelescope/flyql/golang/transformers"
)

type HrefRenderer struct{ columns.BaseRenderer }

func (HrefRenderer) Name() string { return "href" }
func (HrefRenderer) ArgSchema() []transformers.ArgSpec {
    return []transformers.ArgSpec{{Type: flyqltype.String, Required: true}}
}

reg := columns.DefaultRendererRegistry()
_ = reg.Register(HrefRenderer{})

opts := columns.DiagnoseOptions{RendererRegistry: reg}
diags := columns.DiagnoseWithOptions(parsedColumns, schema, opts)
```

## Exemple JavaScript

```javascript
import { Renderer, RendererRegistry, ArgSpec } from 'flyql/renderers'
import { Type } from 'flyql'
import { parse, diagnose } from 'flyql/columns'

class HrefRenderer extends Renderer {
    get name() { return 'href' }
    get argSchema() { return [new ArgSpec(Type.String, true)] }
    diagnose(args, col) {
        if (args && args[0] && !args[0].includes('{{value}}')) {
            return [{
                range: { start: 0, end: 1 },
                message: 'href template should contain {{value}}',
                severity: 'warning',
                code: 'custom_href_no_placeholder',
            }]
        }
        return []
    }
}

const registry = new RendererRegistry()
registry.register(new HrefRenderer())

const cols = parse('url as link|href("/users/{{value}}")',
    { transformers: true, renderers: true })

const diags = diagnose(cols, schema, null, registry)
```

## Intégration dans l'éditeur flyql-vue

Le composant Vue `FlyqlColumns` accepte une prop `rendererRegistry`. Lorsqu'elle est fournie, l'éditeur active l'analyse des renderers post-alias, émet des suggestions de renderers et affiche les diagnostics de renderers :

```vue
<template>
  <FlyqlColumns
    v-model="query"
    :columns="schema"
    :registry="transformerRegistry"
    :renderer-registry="rendererRegistry"
  />
</template>
```

Quand `rendererRegistry` est omis ou `null`, l'analyse des renderers est entièrement désactivée — les utilisateurs tapant `|` post-alias voient un rejet au niveau du parser plutôt que des diagnostics `unknown_renderer` à chaque frappe. Consultez [Composant Columns](/editor/columns-component/) pour la référence complète des props.
