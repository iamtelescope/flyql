:::note
Os renderers nunca executam dentro do flyql — a sua aplicação é responsável por interpretar `parsed_column.renderers` e renderizar o valor.
:::

:::note
`metadata` é livre e definido pelo programador. O núcleo do flyql e o editor nunca leem nenhuma chave específica. Se quiser campos padronizados, defina-os na sua própria aplicação e documente-os junto com os seus renderers registados.
:::

Consulte a [página de sintaxe dos Renderers](/syntax/renderers/) para a gramática e semântica. Esta página cobre a API de registo para adicionar os seus próprios renderers ao validador e ao editor.

## A interface Renderer

Um `Renderer` declara:

- **`name`** — o identificador que os utilizadores escrevem após o pipe do alias
- **`arg_schema`** — um tuple de entradas `ArgSpec(type, required)` que controlam a validação do número e tipo de argumentos
- **`metadata`** — um dict/map/objeto opcional livre para uso aplicacional; o flyql nunca inspeciona o seu conteúdo
- **`diagnose(args, parsed_column)`** — um hook opcional por renderer que retorna `Diagnostic[]` para validação personalizada

## Exemplo Python

```python
from flyql.flyql_type import Type
from flyql.renderers import ArgSpec, Renderer, RendererRegistry
from flyql.columns import parse, diagnose
from flyql.core.validator import Diagnostic
from flyql.core.range import Range


class HrefRenderer(Renderer):
    arg_schema = (ArgSpec(type=Type.String, required=True),)

    @property
    def name(self) -> str:
        return "href"

    @property
    def metadata(self):
        return {"html_safe": True}

    def diagnose(self, arguments, parsed_column):
        if arguments and "{{value}}" not in arguments[0]:
            return [Diagnostic(
                range=Range(0, 1),
                message="href template should contain {{value}}",
                severity="warning",
                code="custom_href_no_placeholder",
            )]
        return []


registry = RendererRegistry()
registry.register(HrefRenderer())

cols = parse('url as link|href("/users/{{value}}")',
             capabilities={"transformers": True, "renderers": True})

# ...depois execute o validador com ambos os registos:
diags = diagnose(cols, schema, renderer_registry=registry)
```

## Hook de cadeia ao nível do registo

Para regras ao nível da cadeia (por ex. "proibir `href` depois de `truncate`") utilize `set_diagnose`:

```python
def chain_rule(parsed_column, chain):
    names = [r["name"] for r in chain]
    if "truncate" in names and "href" in names:
        if names.index("truncate") < names.index("href"):
            return [Diagnostic(
                range=Range(0, 1),
                message="href cannot follow truncate",
                severity="error",
                code="chain_forbidden",
            )]
    return []

registry.set_diagnose(chain_rule)
```

O hook de cadeia executa após os hooks `diagnose` por renderer e após as verificações integradas de nome desconhecido / contagem de argumentos / tipo de argumentos.

## Exemplo Go

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    "github.com/iamtelescope/flyql/golang/columns"
    "github.com/iamtelescope/flyql/golang/flyqltype"
    "github.com/iamtelescope/flyql/golang/transformers"
)

type HrefRenderer struct{ columns.BaseRenderer }

func (HrefRenderer) Name() string { return "href" }
func (HrefRenderer) ArgSchema() []transformers.ArgSpec {
    return []transformers.ArgSpec{{Type: flyqltype.String, Required: true}}
}

reg := columns.DefaultRendererRegistry()
_ = reg.Register(HrefRenderer{})

opts := columns.DiagnoseOptions{RendererRegistry: reg}
diags := columns.DiagnoseWithOptions(parsedColumns, schema, opts)
```

## Exemplo JavaScript

```javascript
import { Renderer, RendererRegistry, ArgSpec } from 'flyql/renderers'
import { Type } from 'flyql'
import { parse, diagnose } from 'flyql/columns'

class HrefRenderer extends Renderer {
    get name() { return 'href' }
    get argSchema() { return [new ArgSpec(Type.String, true)] }
    diagnose(args, col) {
        if (args && args[0] && !args[0].includes('{{value}}')) {
            return [{
                range: { start: 0, end: 1 },
                message: 'href template should contain {{value}}',
                severity: 'warning',
                code: 'custom_href_no_placeholder',
            }]
        }
        return []
    }
}

const registry = new RendererRegistry()
registry.register(new HrefRenderer())

const cols = parse('url as link|href("/users/{{value}}")',
    { transformers: true, renderers: true })

const diags = diagnose(cols, schema, null, registry)
```

## Integração no editor flyql-vue

O componente Vue `FlyqlColumns` aceita uma prop `rendererRegistry`. Quando fornecida, o editor ativa a análise de renderers pós-alias, emite sugestões de renderers e apresenta diagnósticos de renderers:

```vue
<template>
  <FlyqlColumns
    v-model="query"
    :columns="schema"
    :registry="transformerRegistry"
    :renderer-registry="rendererRegistry"
  />
</template>
```

Quando `rendererRegistry` é omitido ou `null`, a análise de renderers é completamente desativada — os utilizadores que escrevem `|` pós-alias veem uma rejeição ao nível do parser em vez de diagnósticos `unknown_renderer` a cada tecla. Consulte [Componente Columns](/editor/columns-component/) para a referência completa das props.
