## Принадлежность списку

Используйте `in`, чтобы проверить, входит ли значение в список, и `not in`, чтобы проверить обратное:

```
status in [200, 201, 204]
env not in ['prod', 'staging']
```

## Правила синтаксиса

- Значения заключаются в квадратные скобки `[ ]` и разделяются запятыми
- Все значения в списке должны быть одного типа (все строки или все числа)
- Строковые значения должны быть в кавычках: `['a', 'b', 'c']`
- Числовые значения пишутся без кавычек: `[1, 2, 3]`
- Пробелы вокруг запятых и скобок допустимы

### Списки строк

```
method in ['GET', 'POST', 'PUT']
env not in ['dev', 'test', 'local']
```

Для строковых значений можно использовать как одинарные, так и двойные кавычки.

### Списки чисел

```
status in [200, 201, 204]
port not in [80, 443]
```

### Пустые списки

Пустые списки допустимы:

```
status in []        # всегда false — ничто не совпадает с пустым списком
status not in []    # всегда true — всё «не входит» в пустой список
```

## Согласованность типов

Списки **должны** содержать значения одного типа — только строки или только числа. Смешивание типов в одном списке вызывает ошибку парсинга:

```
# Допустимо — все числа
status in [200, 201, 204]

# Допустимо — все строки
env in ['prod', 'staging']

# Недопустимо — смешанные типы (ошибка парсинга)
values in [200, 'ok']
```

Парсер отслеживает тип первого значения в списке и отклоняет все последующие значения другого типа. Это проверяется на этапе парсинга, до того как запрос попадёт в генератор или обработчик совпадений.

Когда SQL-генератор обрабатывает список, он дополнительно проверяет совместимость типов значений списка с типом целевой колонки. Например, передача строковых значений для числовой колонки вызовет ошибку валидации при генерации SQL.

Поле `valuesType` в разобранном выражении указывает, содержит ли список значения типа `"string"` или `"number"`. Для пустых списков `valuesType` равно `null`.

## Комбинация с другими операторами

Операторы работы со списками можно комбинировать с булевыми операторами, как и любые другие условия:

```
status in [200, 201] and method in ['GET', 'POST']
level = error or status not in [200, 201, 204]
(status in [500, 502, 503] or level = critical) and env=prod
```

## Примеры

```
# HTTP-коды успешных ответов
status in [200, 201, 204]

# Исключить тестовые окружения
env not in ['dev', 'test', 'staging']

# Несколько условий со списками
method in ['GET', 'POST'] and status not in [401, 403, 404]
```
