FlyQL はカラムスキーマで**暦日**（`Type.Date`）と**時点**（`Type.DateTime`）を区別し、その区別を使って ISO-8601 文字列リテラルに対するマッチャーのコアーションを制御します。一方 SQL ジェネレータは文字列リテラルをそのまま通します — 解析するのは DB です。

## 暦日 vs. 時点

`Type.Date` — 暦日のみ。比較は `YYYYMMDD` まで切り詰められます。DB 型 `date` / `date32` を吸収します。

`Type.DateTime` — タイムライン上の点、ミリ秒解像度。比較は UTC ミリ秒で行われます。`datetime`、`datetime64`、`timestamp`、`timestamptz`、`year` を吸収します。

セマンティック型の宣言はマッチャーにとって重要です: `'2026-04-06T15:30:00Z'` と比較される `Date` カラムは、比較前にレコード値を Y/M/D 成分まで切り詰めます。一方 `DateTime` カラムはミリ秒まで比較します。

`Type.Date` と宣言されたカラムが datetime 形状の値（ネイティブの `datetime` / `time.Time` / `Date` インスタンス、または時刻成分を含む ISO 文字列）を受け取った場合、FlyQL はカラムごとに一度だけ `UserWarning`（Python）/ ログ行（Go）/ `console.warn`（JS）を出力し、`Type.DateTime` への移行を示唆します。

## 文字列リテラルとしての日付

date/datetime カラムを ISO-8601 形式の文字列リテラルと比較します。ジェネレータはリテラルをそのまま通し、DB が実行時に解析します。

<FlyqlCode code={`due_date < '2023-12-31'`} />
<FlyqlCode code={`created_at >= '2023-12-31T00:00:00Z'`} />
<FlyqlCode code={`event_time >= '2023-01-01' and event_time <= '2023-12-31'`} />

マッチャーでは、スキーマで宣言された `Date` / `DateTime` カラムが ISO-8601 リテラルとレコード値の両方の自動解析を駆動します。よくあるケースでは明示的な変換関数は不要です。

### 受け付ける iso8601 形式

マッチャーの `iso8601` ファミリーは寛容です:

- `YYYY-MM-DD` — 純粋な暦日（Date カラム用、または DateTime カラムではカラム tz の深夜）。
- `YYYY-MM-DDTHH:MM:SS` — ナイーブ datetime。
- `YYYY-MM-DD HH:MM:SS` — スペース区切りのナイーブ datetime（解析前に正規化）。
- `YYYY-MM-DDTHH:MM:SS.ffffff` — 任意のサブ秒（ms 解像度のみ — 下記「精度」参照）。
- 上記いずれも `Z` または `±HH:MM` オフセット接尾辞を付けられます。

### ナイーブリテラル向けのカラムメタデータ

`DateTime` カラム宣言はオプションのメタデータを持てます:

- `tz` — ナイーブ ISO 文字列を解釈するための IANA タイムゾーン名。Evaluator の `default_timezone` に、最終的に `"UTC"` にフォールバックします。
- `unit` — `"ms"`（デフォルト）、`"s"`、または `"ns"`。数値レコード値に適用され、非数値は無視します。

```json
{
  "ts":        { "type": "datetime", "tz": "Europe/Moscow", "unit": "s" },
  "event_day": { "type": "date" }
}
```

:::note[文字列はクォートが必須]
日付リテラルは文字列なので、**必ず**クォートで囲んでください。クォートが無い場合、FlyQL はその値をカラム参照として解釈します。<FlyqlCode code={`due_date < 2023-12-31`} inline /> は `2023` という名前のカラムに対する算術として parse され、日付リテラルにはなりません。[値と型 — String](/ja/syntax/values/#string) を参照してください。
:::

## 時間関数

「直近 1 時間」「今週の始めから」「今日から」のような相対時間クエリには、組み込みの時間関数 <FlyqlCode code={`ago(...)`} inline />、<FlyqlCode code={`now()`} inline />、<FlyqlCode code={`today()`} inline />、<FlyqlCode code={`startOf(...)`} inline /> を使います。

<FlyqlCode code={`created_at > ago(1h)`} />
<FlyqlCode code={`updated_at >= ago(7d)`} />
<FlyqlCode code={`expires_at < now()`} />
<FlyqlCode code={`event_time > startOf('day')`} />
<FlyqlCode code={`scheduled_for >= today('Europe/Berlin')`} />

完全な文法（単位、順序ルール、タイムゾーン引数）は [値と型 — 時間関数](/ja/syntax/values/#temporal-functions) を参照してください。

## サマータイム（DST）

ナイーブ datetime リテラル（オフセットなし、`Z` なし）を DST を観測するタイムゾーンで解釈する場合、FlyQL は仕様で定義された一意の解決を選びます:

- **後退（あいまいな壁時計）** — 早い方の発生が選ばれます（Python セマンティクスの `fold=0`）。例: `America/New_York` での `'2026-11-01 01:30:00'` は 05:30 UTC（EDT）に解決され、06:30 UTC（EST）にはなりません。
- **前進（存在しない壁時計）** — 値は拒否されます。マッチャーはそのレコードに対して `false` を返します（スキップ）。例: `America/New_York` での `'2026-03-08 02:30:00'` は存在しません。

前進ギャップに落ちるレコードは、黙って前方正規化されるのではなく、解析不可として扱われます。

## サポートされるレコード値型（Python）

- `datetime.datetime` — tz-aware またはナイーブ（ナイーブはカラム `tz` または Evaluator デフォルトを使用）。
- `datetime.date` — 暦日のみ。
- `pandas.Timestamp` — `datetime` のサブクラスなので動作します。
- ISO-8601 文字列 — 上記の受け付ける形式を参照。
- `int` / `float` — デフォルトで epoch ミリ秒。epoch 秒はカラム `unit="s"`、ナノ秒は `"ns"` を設定します。

現時点で未サポート:

- `numpy.datetime64` — マッチャーに渡す前に変換してください: `pd.Timestamp(value).to_pydatetime()` または `value.astype("datetime64[ms]").astype(int)`。

## 精度

FlyQL のマッチャーは時間値を**ミリ秒粒度**で比較します。ソースデータのサブミリ秒精度は比較前に切り捨てられます:

- Python — `int(dt.timestamp() * 1000)` がマイクロ秒を捨てます。
- Go — `UnixMilli()` がナノ秒を捨てます。
- JavaScript — `Date` はネイティブに ms です。

ナノ秒精度が必要な場合は、レコードを bigint ナノ秒に前処理し、`unit="ns"` を宣言してください。JS 経路は `Number` を使い `2^53` を超えると精度を失います — ナノ秒 BigInt 経路は後続仕様に予約されています。

## スキーマなしの数値

カラムスキーマなしの数値レコード値は **epoch ミリ秒**として扱われます。自動検出ヒューリスティックはありません — レコード `{"ts": 1712434800}`（慣例的には epoch 秒）は `1970-01-20` として解釈され、`2024` にはなりません。epoch 秒データを扱うユーザーは `unit="s"` を宣言する必要があります:

```json
{ "ts": { "type": "datetime", "unit": "s" } }
```

これは意図的に厳格です: 自動検出ヒューリスティック（例: `value < 10^10 → 秒`）は境界や正当な古い日付値で失敗するため、静かな推測より明示的な宣言を優先します。

## どちらを使うか

| 目的 | 使うもの | 例 |
|------|---------|------|
| 固定の暦日付やタイムスタンプとの比較 | 文字列リテラル | <FlyqlCode code={`due_date < '2023-12-31'`} inline /> |
| スライドする時間ウィンドウとの比較 | <FlyqlCode code={`ago()`} inline /> | <FlyqlCode code={`created_at > ago(1h)`} inline /> |
| 現在時刻との比較 | <FlyqlCode code={`now()`} inline /> | <FlyqlCode code={`expires_at < now()`} inline /> |
| 今日との比較（タイムゾーン対応） | <FlyqlCode code={`today()`} inline /> | <FlyqlCode code={`scheduled_for = today('UTC')`} inline /> |
| 日・週・月の始まりとの比較 | <FlyqlCode code={`startOf()`} inline /> | <FlyqlCode code={`event_time > startOf('week')`} inline /> |

文字列リテラルは不変で、同じクエリ文字列は毎日同じ SQL を返します。時間関数は実行時に評価されるため、同じ FlyQL クエリでも実行ごとに異なる時間ウィンドウを生成します。
