## Принадлежность списку

Используйте <FlyqlToken type="operator">in</FlyqlToken>, чтобы проверить, входит ли значение в список, и <FlyqlToken type="operator">not in</FlyqlToken>, чтобы проверить обратное:

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['prod', 'staging']`} />

## Правила синтаксиса

- Значения заключаются в квадратные скобки <FlyqlToken type="operator">[ ]</FlyqlToken> и разделяются запятыми
- Значения могут быть разных типов: <FlyqlCode code={`[200, 'ok', true, null]`} inline />
- Строковые значения должны быть в кавычках: <FlyqlCode code={`['a', 'b', 'c']`} inline />
- Числовые значения пишутся без кавычек: <FlyqlCode code={`[1, 2, 3]`} inline />
- Пробелы вокруг запятых и скобок допустимы

### Списки строк

<FlyqlCode code={`method in ['GET', 'POST', 'PUT']`} />
<FlyqlCode code={`env not in ['dev', 'test', 'local']`} />

Для строковых значений можно использовать как одинарные, так и двойные кавычки.

### Списки чисел

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`port not in [80, 443]`} />

### Пустые списки

Пустые списки допустимы:

<FlyqlCode code={`status in []`} />

Пустой список <FlyqlToken type="operator">in</FlyqlToken> всегда false — ничто не совпадает с пустым списком.

<FlyqlCode code={`status not in []`} />

Пустой список <FlyqlToken type="operator">not in</FlyqlToken> всегда true — всё «не входит» в пустой список.

## Смешанные типы

Списки поддерживают гетерогенные значения — можно смешивать строки, числа, булевы значения и null:

<FlyqlCode code={`field in [200, 'ok', true, null]`} />

Каждый элемент имеет свой тип в массиве `valuesTypes` разобранного выражения (например, `["int", "string", "bool", "null"]`). Для пустых списков `valuesTypes` равен `null`.

Когда SQL-генератор обрабатывает список, он проверяет совместимость типов значений списка с типом целевой колонки. Например, передача строковых значений в числовую колонку вызовет ошибку валидации при генерации SQL.

## Согласованность типов в IN-списке

Парсер **всегда** принимает гетерогенные списки — проверка типов живёт в SQL-генераторах, а не в парсере. Каждый генератор (ClickHouse, PostgreSQL, StarRocks) применяет одну и ту же двухуровневую логику пропуска прежде, чем возбудить ошибку.

### Уровень 1 — проверка на стороне вызова

Перед вызовом валидатора генератор проверяет три условия:

1. У колонки **объявлен тип** (`column.flyqlType()` не пустой и не `Unknown`).
2. Ключ **не сегментированный** (одиночный сегмент вроде `category`, а не `meta.category`).
3. Список **однородный** — у всех элементов один и тот же тип значения (`len(set(values_types)) == 1` в Python; `new Set(valuesTypes).size === 1` в JavaScript; аналогично в Go).

Если **хотя бы одно** из этих условий не выполняется, валидатор пропускается полностью, и SQL генерируется как есть. Гетерогенные списки проходят. Списки против колонок без типа проходят. Списки против сегментированных ключей (JSON-пути, индексы массивов) проходят.

### Уровень 2 — проверка элементов

Когда проверка на стороне вызова пройдена, валидатор перебирает каждый элемент и возбуждает ошибку на первом несовместимом значении согласно карте совместимости:

| Тип колонки | Допустимые типы элементов |
|-------------|---------------------------|
| `String`    | `string`                  |
| `Int`       | `int`, `float`            |
| `Float`     | `int`, `float`            |
| `Bool`      | `bool`, `int`             |
| `Date`      | `string`                  |

Остальные типы колонок (`Array`, `Map`, `JSON`, `JSONString`, `Struct`, `Duration`, `Unknown`) отсутствуют в карте и молча пропускают проверку элементов.

### Текст ошибки

При неудаче валидации все три реализации возбуждают **идентичное** сообщение об ошибке:

<FlyqlCode code={`type mismatch in IN list: {column_type} column cannot contain {value_type} values`} />

Конкретно:

- Python: `FlyqlError("type mismatch in IN list: int column cannot contain string values")`
- Go: `errors.New("type mismatch in IN list: int column cannot contain string values")`
- JavaScript: `new Error("type mismatch in IN list: int column cannot contain string values")`

Подстрока `type mismatch in IN list:` стабильна для всех трёх языков и трёх SQL-диалектов, поэтому по ней безопасно искать grep'ом.

Полный сквозной пример с тремя случаями (однородное соответствие, гетерогенный пропуск, однородный неверный тип с ошибкой) см. в разделе [Рецепты — согласованность типов в IN-списке](/ru/syntax/recipes/#согласованность-типов-в-in-списке).

## Параметры в списках

Плейсхолдеры параметров могут появляться внутри IN-списков, свободно смешиваясь с литеральными значениями:

<FlyqlCode code={`status in [$x, $y, 200]`} />
<FlyqlCode code={`env not in [$current, 'dev']`} />

Объекты `Parameter` остаются в массиве `values` до тех пор, пока `bindParams()` их не разрешит. Полный справочник см. в разделе [Параметры](/ru/syntax/parameters/).

## Комбинация с другими операторами

Операторы работы со списками можно комбинировать с булевыми операторами, как и любые другие условия:

<FlyqlCode code={`status in [200, 201] and method in ['GET', 'POST']`} />
<FlyqlCode code={`level = 'error' or status not in [200, 201, 204]`} />
<FlyqlCode code={`(status in [500, 502, 503] or level = 'critical') and env='prod'`} />

## Примеры

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['dev', 'test', 'staging']`} />
<FlyqlCode code={`method in ['GET', 'POST'] and status not in [401, 403, 404]`} />
