## Dot Notation

Use dot-separated paths to access nested fields in JSON, Map, or structured data:

<FlyqlCode code={`request.url = "/api/users"`} />
<FlyqlCode code={`response.headers.content_type = "application/json"`} />
<FlyqlCode code={`metadata.labels.env = prod`} />

Each dot-separated segment traverses one level into the nested structure. The SQL generators translate these paths into the correct syntax for each database dialect (JSON path access, Map lookups, etc.).

## Quoted Key Segments

When a key segment itself contains a dot, wrap it in quotes so the dot is not treated as a separator:

<FlyqlCode code={`"foo.bar".baz = value`} />

Here:
- `"foo.bar"` is a single key segment (the dot is literal)
- <FlyqlToken type="key">baz</FlyqlToken> is a child of `"foo.bar"`

Without quotes, <FlyqlToken type="key">foo.bar.baz</FlyqlToken> would be three segments: <FlyqlToken type="key">foo</FlyqlToken> → <FlyqlToken type="key">bar</FlyqlToken> → <FlyqlToken type="key">baz</FlyqlToken>.

### Quoting Rules

Both single and double quotes work for key segments:

<FlyqlCode code={`"foo.bar".baz = value`} />
<FlyqlCode code={`'foo.bar'.baz = value`} />

Only the segments containing dots need to be quoted:

<FlyqlCode code={`metadata."dotted.key".value = 123`} />

## Path Depth

There is no limit on nesting depth:

<FlyqlCode code={`a.b.c.d.e = value`} />

Each segment is traversed in order when generating SQL or evaluating in-memory.

## Combined with Operators

Nested keys work with all operators:

<FlyqlCode code={`request.status >= 400`} />
<FlyqlCode code={`response.body ~ "error.*"`} />
<FlyqlCode code={`metadata.tags in ['prod', 'staging']`} />
<FlyqlCode code={`request.headers.content_type != "text/html"`} />
<FlyqlCode code={`not request.authenticated`} />

## Examples
