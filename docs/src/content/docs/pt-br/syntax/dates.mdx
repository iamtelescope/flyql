FlyQL distingue **dias de calendário** (`Type.Date`) de **instantes no tempo** (`Type.DateTime`) nos esquemas de colunas, e usa essa distinção para orientar a coerção do matcher sobre literais string ISO-8601. Os geradores SQL, por outro lado, passam os literais string exatamente como estão — quem faz o parse é o banco de dados.

## Dia de calendário vs. instante

`Type.Date` — apenas dia de calendário. As comparações são truncadas para `YYYYMMDD`. Absorve os tipos brutos do banco `date` / `date32`.

`Type.DateTime` — ponto na linha temporal com resolução de milissegundos. As comparações acontecem em milissegundos UTC. Absorve `datetime`, `datetime64`, `timestamp`, `timestamptz`, `year`.

Declarar o tipo semântico importa para o matcher: uma coluna `Date` comparada contra `'2026-04-06T15:30:00Z'` trunca o valor do registro para seu componente Y/M/D antes da comparação, enquanto uma coluna `DateTime` compara até o milissegundo.

Se uma coluna declarada como `Type.Date` recebe um valor com formato de datetime (instância nativa `datetime` / `time.Time` / `Date`, ou string ISO com componente de horário), FlyQL emite uma vez por coluna um `UserWarning` (Python) / linha de log (Go) / `console.warn` (JS) sugerindo migração para `Type.DateTime`.

## Data como literal string

Compare uma coluna date/datetime com um literal string ISO-8601. Os geradores passam o literal exatamente como está; o banco de dados o interpreta em tempo de execução.

<FlyqlCode code={`due_date < '2023-12-31'`} />
<FlyqlCode code={`created_at >= '2023-12-31T00:00:00Z'`} />
<FlyqlCode code={`event_time >= '2023-01-01' and event_time <= '2023-12-31'`} />

No matcher, as colunas `Date` / `DateTime` declaradas no esquema orientam o parse automático dos literais ISO-8601 E dos valores do registro. Para o caso comum não é necessária nenhuma função de conversão explícita.

### Formas iso8601 aceitas

A família `iso8601` do matcher é permissiva:

- `YYYY-MM-DD` — data de calendário pura (para colunas Date, ou meia-noite na tz da coluna para colunas DateTime).
- `YYYY-MM-DDTHH:MM:SS` — datetime naive.
- `YYYY-MM-DD HH:MM:SS` — datetime naive com separador de espaço (normalizado antes do parse).
- `YYYY-MM-DDTHH:MM:SS.ffffff` — subsegundo opcional (apenas resolução ms — veja "Precisão" abaixo).
- Qualquer uma das formas anteriores com sufixo de offset `Z` ou `±HH:MM`.

### Metadados de coluna para literais naive

Uma declaração de coluna `DateTime` pode conter metadados opcionais:

- `tz` — nome de fuso horário IANA usado para interpretar strings ISO naive. Volta para `default_timezone` do Evaluator, depois para `"UTC"`.
- `unit` — `"ms"` (padrão), `"s"` ou `"ns"`. Aplicado a valores numéricos do registro; valores não numéricos o ignoram.

```json
{
  "ts":        { "type": "datetime", "tz": "Europe/Moscow", "unit": "s" },
  "event_day": { "type": "date" }
}
```

:::note[Strings devem estar entre aspas]
Literais de data são strings e **devem** estar entre aspas. Sem aspas, FlyQL interpreta o valor como referência de coluna. <FlyqlCode code={`due_date < 2023-12-31`} inline /> seria interpretado como aritmética sobre uma coluna chamada `2023`, não como literal de data. Veja [Valores e Tipos — String](/pt-br/syntax/values/#string).
:::

## Funções temporais

Para consultas de tempo relativo — "na última hora", "desde o início desta semana", "desde hoje" — FlyQL fornece funções temporais embutidas: <FlyqlCode code={`ago(...)`} inline />, <FlyqlCode code={`now()`} inline />, <FlyqlCode code={`today()`} inline /> e <FlyqlCode code={`startOf(...)`} inline />.

<FlyqlCode code={`created_at > ago(1h)`} />
<FlyqlCode code={`updated_at >= ago(7d)`} />
<FlyqlCode code={`expires_at < now()`} />
<FlyqlCode code={`event_time > startOf('day')`} />
<FlyqlCode code={`scheduled_for >= today('Europe/Berlin')`} />

A gramática completa (unidades, regras de ordem, argumentos de fuso horário) está coberta em [Valores e Tipos — Funções Temporais](/pt-br/syntax/values/#temporal-functions).

## Horário de verão (DST)

Quando um literal datetime naive (sem offset, sem `Z`) é interpretado em uma tz que observa DST, FlyQL escolhe a resolução inequívoca definida pela especificação:

- **Retrocesso (relógio de parede ambíguo)** — a ocorrência anterior é escolhida (`fold=0` na semântica Python). Exemplo: `'2026-11-01 01:30:00'` em `America/New_York` resolve para 05:30 UTC (EDT), não para 06:30 UTC (EST).
- **Avanço (relógio de parede inexistente)** — o valor é rejeitado. O matcher retorna `false` para esse registro (é ignorado). Exemplo: `'2026-03-08 02:30:00'` em `America/New_York` não existe.

Registros que caem na lacuna do avanço são tratados como não interpretáveis, em vez de serem silenciosamente normalizados para a frente.

## Tipos de valor de registro suportados (Python)

- `datetime.datetime` — tz-aware ou naive (naive usa a `tz` da coluna ou o padrão do Evaluator).
- `datetime.date` — apenas dia de calendário.
- `pandas.Timestamp` — funciona porque é subclasse de `datetime`.
- Strings ISO-8601 — veja as formas aceitas acima.
- `int` / `float` — milissegundos epoch por padrão; defina `unit="s"` na coluna para segundos epoch ou `"ns"` para nanossegundos.

Não suportados atualmente:

- `numpy.datetime64` — converta antes de passar para o matcher: `pd.Timestamp(value).to_pydatetime()` ou `value.astype("datetime64[ms]").astype(int)`.

## Precisão

O matcher do FlyQL compara valores temporais com **granularidade de milissegundos**. A precisão sub-milissegundo nos dados de origem é truncada antes da comparação:

- Python — `int(dt.timestamp() * 1000)` descarta microssegundos.
- Go — `UnixMilli()` descarta nanossegundos.
- JavaScript — `Date` é nativamente em ms.

Usuários que precisam de precisão em nanossegundos devem pré-processar os registros para nanossegundos bigint e declarar `unit="ns"`. O caminho JS usa `Number`, que perde precisão acima de `2^53` — o caminho BigInt de nanossegundos está reservado para uma especificação posterior.

## Numéricos sem esquema

Valores numéricos de registro sem esquema de coluna são tratados como **milissegundos epoch**. Não há heurística de autodetecção — um registro `{"ts": 1712434800}` (segundos epoch, por convenção) é interpretado como `1970-01-20` e não como `2024`. Usuários com dados em segundos epoch DEVEM declarar `unit="s"`:

```json
{ "ts": { "type": "datetime", "unit": "s" } }
```

É rigoroso por design: uma heurística de autodetecção (p. ex. `value < 10^10 → segundos`) falha na fronteira e em valores de datas antigas legítimos, então preferimos declarações explícitas a palpites silenciosos.

## Quando usar cada uma

| Intenção | Use | Exemplo |
|----------|-----|---------|
| Comparar com data ou timestamp fixo | Literal string | <FlyqlCode code={`due_date < '2023-12-31'`} inline /> |
| Comparar com janela temporal deslizante | <FlyqlCode code={`ago()`} inline /> | <FlyqlCode code={`created_at > ago(1h)`} inline /> |
| Comparar com o momento atual | <FlyqlCode code={`now()`} inline /> | <FlyqlCode code={`expires_at < now()`} inline /> |
| Comparar com hoje (ciente do fuso horário) | <FlyqlCode code={`today()`} inline /> | <FlyqlCode code={`scheduled_for = today('UTC')`} inline /> |
| Comparar com início do dia/semana/mês | <FlyqlCode code={`startOf()`} inline /> | <FlyqlCode code={`event_time > startOf('week')`} inline /> |

Literais string permanecem constantes — a mesma string de consulta retorna o mesmo SQL dia após dia. Funções temporais são avaliadas em tempo de execução, então a mesma consulta FlyQL produz uma janela temporal diferente em cada execução.
