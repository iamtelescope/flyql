import { Tabs, TabItem } from '@astrojs/starlight/components'

`flyql.Type`은 flyql에서 컬럼 및 값 타입을 위한 **유일한 정규 어휘**입니다. 모든 transformer, validator, SQL generator가 이 enum을 기반으로 dispatch합니다. 각 언어는 동일한 12개의 값을 동일한 소문자 문자열 ID로 노출하므로, JSON 테스트 픽스처와 직렬화된 컬럼은 언어에 의존하지 않습니다.

## 포함 기준

다음 중 **하나 이상**에 해당하면 해당 값은 enum에 포함됩니다.

1. validator 동작이 달라지는 경우.
2. generator dispatch가 방언 간에 다른 경우.
3. 사용자가 명시적으로 확인한 필요성을 가진 미래 지향적 타입인 경우 (현재: `Duration`).

추측에 의한 추가는 명확히 지양합니다. 새로운 타입이 필요한 경우, 제안하기 전에 이 기준에 비추어 평가하세요.

## 12개의 값

| 값 | 목적 | Validator 동작 | Generator dispatch | 흡수되는 원시 DB 타입 |
|---|---|---|---|---|
| **`String`** | 텍스트 값 | `like`, `ilike`, `regex`, `not regex` 허용. IN 리스트 문자열 타이핑 | 문자열 비교 SQL | varchar, char, text, FixedString, blob, UUID, IPv4/v6 |
| **`Int`** | 정수 | 숫자 순서 연산자. regex 거부. IN 리스트 숫자 타이핑 | 숫자 비교 SQL | int8/16/32/64/128/256, uint*, bigint, smallint, tinyint |
| **`Float`** | 부동소수점 수 | 숫자 순서 연산자. regex 거부 | 숫자 비교 SQL | float, double, decimal, numeric, real |
| **`Bool`** | 불리언 값 | 동등 및 truthy만 가능. 순서 연산 거부 | 불리언 SQL | bool, boolean |
| **`Date`** | 시점을 나타내는 시간적 값 | 시간 함수 호출(`ago()`, `now()`) 허용. 날짜로 변환되는 문자열 리터럴 허용 | 선택적 타임존 포함 시간 비교 SQL | date, date32, datetime, datetime64, timestamp, year |
| **`Duration`** | 간격/기간 값 | 기간 리터럴(`30m1s`, `1h`, `7d`) 허용. 문자열 전용 연산 거부 | *(아직 특별한 dispatch 없음 — 아래 참고)* | ClickHouse `Interval*`, PostgreSQL `interval` |
| **`Array`** | 값의 순서 있는 컬렉션 | `has` 연산자 허용. 세그먼트 키 접근은 **0 기반** 숫자 인덱스 사용 | `col[idx]` 요소 접근 | array, ClickHouse `Array(...)`, PG arrays |
| **`Map`** | 동적 키를 가진 키→값 컬렉션 | 세그먼트 키 접근은 문자열 키 사용 | `col['key']` 조회 | ClickHouse `Map(...)`, PostgreSQL `hstore` |
| **`Struct`** | 이름 있는(또는 위치 기반) 필드를 가진 고정 형태 레코드 | 세그먼트 키 접근은 필드 이름 사용 | 방언별 필드 접근 (ClickHouse에서 `tupleElement(col,'name')`, StarRocks에서 `` `col`.`field` ``) | ClickHouse `Tuple`, StarRocks `STRUCT` |
| **`JSON`** | 반정형 JSON 문서 | 세그먼트 키 접근은 JSON 경로 사용 | JSON 경로 SQL (`JSON_VALUE`, `JSONExtractString`) | ClickHouse `JSON`, PostgreSQL `json`/`jsonb`, StarRocks `JSON` |
| **`JSONString`** | 내용이 유효한 JSON인 텍스트 컬럼 | `JSON`과 동일 (연산자, 세그먼트 경로 접근) | 쿼리 시점 파싱 JSON SQL (PG에서 `(col::jsonb)`, CH에서 `JSONExtract*`, SR에서 `parse_json(col)->…`) | 합성 원시 타입 토큰 `"jsonstring"`으로 명시적 선언 |
| **`Unknown`** | flyql이 추론할 수 없는 타입의 문서화된 폴백 | 연산자는 기본값으로 폴스루. 경로 접근은 "unsupported column type" 에러. transformer 수용 불가 | 특별한 dispatch 없음. 원시 컬럼 참조만 가능 | ClickHouse `Nothing`/`Object`/`Variant`/`Dynamic`/geometry, StarRocks `bitmap`/`hll`, 인식되지 않는 모든 것 |

:::note[Duration은 미래 지향적 타입입니다]
`Duration`은 현재 존재하여 방언 generator가 원시 `Interval*` 및 `interval` 타입을 즉시 바인딩할 수 있습니다. 그러나 현재 `Duration` 타입 컬럼과 `Unknown` 타입 컬럼에 대해 다르게 분기하는 generator는 없습니다. duration 컬럼에 대한 시간적 검증은 후속 스펙에서 계획되어 있습니다.
:::

## `Type.JSONString` — JSON 문서를 포함하는 텍스트 컬럼

`Type.JSONString`은 정규 컬럼 타입이며, 직교하는 capability 플래그가 아닙니다. 컬럼을 `type: "jsonstring"` (플레인 오브젝트 스키마) 또는 정규 `Column` / `NewColumn` 생성자에 `Type.JSONString` / `flyqltype.JSONString`을 전달하여 선언하세요. 컬럼은 기반 DB에서 text/varchar로 저장되지만 내용은 유효한 JSON이며, generator는 방언별 파싱 함수로 접근을 래핑합니다.

- **ClickHouse**: 세그먼트 키 접근은 텍스트 컬럼에 대해 `JSONExtract*`를 사용합니다.
- **PostgreSQL**: 세그먼트 키 접근은 JSON 경로 연산자를 적용하기 전에 `(col::jsonb)` 캐스트를 사용합니다.
- **StarRocks**: 세그먼트 키 접근은 `parse_json(col)->'path'`를 사용합니다.

세그먼트 경로 접근, `has`, truthy/falsy, 동등/부등 모두 동작합니다. validator는 타입별 연산자 제한을 적용하지 않습니다. 현재 `String` 컬럼에 대해 작성할 수 있는 것은 이 리팩터링 후 `JSONString` 컬럼에 대해서도 작성할 수 있습니다. 차이는 순전히 generator 출력 SQL에 있습니다.

### 이전 `jsonstring` 불리언에서의 마이그레이션

이전에는 JSON을 담고 있는 텍스트 컬럼을 두 필드 `type: "String"` **그리고** `jsonstring: true`로 선언했습니다. 불리언 필드는 `flyql.Column`, 모든 방언 `Column`/`ColumnDef`, 모든 플레인 오브젝트 로더에서 **제거**되었습니다. 레거시 스키마는 마이그레이션이 필요합니다.

```diff
- { "body": { "jsonstring": true, "type": "String" } }
+ { "body": { "type": "jsonstring" } }
```

플레인 오브젝트 스키마에 `jsonstring` 키가 존재하면 (`"jsonstring": false` 잔여물 포함) `FromPlainObject` / `from_plain_object` / `fromPlainObject`에서 대상 마이그레이션 에러가 발생합니다. 이식 가능한 마이그레이션 스니펫:

- **jq (≥1.6):** `jq 'walk(if type == "object" then del(.jsonstring) else . end)' config.json > config.new.json`
- **Python:** `python3 -c "import json,sys; d=json.load(open('config.json')); [c.pop('jsonstring',None) for c in (d.values() if isinstance(d,dict) else d)]; json.dump(d, open('config.new.json','w'), indent=2)"`

단순한 `sed`는 피하세요. `jsonstring`이 오브젝트의 마지막 키인 경우 고아 쉼표가 남고, GNU sed와 BSD sed 간에 이식성이 없습니다.

## `Type.Any` — transformer 입력 와일드카드

`Type.Any` (문자열 값 `"any"`)는 transformer의 입력 타입으로만 사용되는 sentinel입니다. transformer가 모든 상위 컬럼/transformer 출력 타입을 받아들임을 선언합니다. 위에 나열된 컬럼 적격 값에 의도적으로 포함되지 **않습니다**.

- `parseFlyQLType("any")` / `ParseType("any")`는 에러를 발생시킵니다. 엄격한 파서는 `"any"`를 컬럼 타입 토큰으로 거부합니다.
- `TransformerRegistry.register`는 `output_type`이나 `ArgSpec.type`에 `Any`를 선언한 transformer를 거부합니다. 모든 transformer는 여전히 구체적인 출력 타입을 선언해야 합니다. `Any`를 출력에 사용하면 하류 체인 타이핑이 깨집니다.

AST validator, columns validator, 또는 런타임 체인 헬퍼가 `input_type == Any`인 transformer를 만나면 입력 타입 동등성 검사는 건너뜁니다. 하류 엄격성은 그대로 유지됩니다 — `col|accepts_any_returning_array|upper`는 `array`가 `upper`의 유효한 입력이 아니므로 `upper`에서 여전히 에러가 발생합니다.

선언 및 사용에 대해서는 [사용자 정의 transformer — Any 입력 transformer](/advanced/custom-transformers/#any-입력-transformer)를 참조하세요.

## 엄격 모드 파싱

`parseFlyQLType` 헬퍼 (Go에서 `flyql.ParseType`, Python에서 `flyql.parse_flyql_type`, JS에서 `parseFlyQLType`)는 **엄격**합니다. 11개의 유효한 소문자 토큰 중 하나가 아닌 문자열에 대해 에러를 발생시킵니다.

엄격 모드는 필수입니다. 관대한 파싱은 하류 소비자를 무음으로 손상시키기 때문입니다. 레거시 `normalized_type` JSON 키는 감지되어 이 페이지를 가리키는 대상 마이그레이션 에러를 생성합니다.

## 방언 컬럼에서 `flyql.ColumnSchema`로의 브릿지

각 방언 패키지는 `ToFlyQLSchema` (Go), `to_flyql_schema` (Python), `toFlyQLSchema` (JS) 헬퍼를 제공합니다. 이것은 방언 컬럼을 validator에 전달하는 정규 방법입니다.

<Tabs syncKey="lang">
<TabItem label="Go">

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    clickhouse "github.com/iamtelescope/flyql/golang/generators/clickhouse"
)

cols := []*clickhouse.Column{
    clickhouse.NewColumn(clickhouse.ColumnDef{Name: "host", Type: "String"}),
    clickhouse.NewColumn(clickhouse.ColumnDef{Name: "count", Type: "Int64"}),
    clickhouse.NewColumn(clickhouse.ColumnDef{Name: "payload", Type: "jsonstring"}),
}
schema := clickhouse.ToFlyQLSchema(cols)
diags := flyql.Diagnose(ast, schema, nil)
```

</TabItem>
<TabItem label="Python">

```python
from flyql import diagnose
from flyql.generators import clickhouse

cols = [
    clickhouse.Column("host", _type="String"),
    clickhouse.Column("count", _type="Int64"),
    clickhouse.Column("payload", _type="jsonstring"),
]
schema = clickhouse.to_flyql_schema(cols)
diags = diagnose(ast, schema)
```

</TabItem>

## 이전 어휘에서의 마이그레이션

통합 이전에 flyql에는 3개의 병렬 "타입" 어휘가 있었습니다.

1. `transformers.TransformerType` — `string | int | float | bool | array`
2. 방언 `NormalizedType*` 상수 — `string | int | float | bool | date | array | map | tuple | json | geometry | interval | special` (방언마다 다름)
3. `flyql.Column.NormalizedType` — (2)와 일치할 것으로 기대되는 자유 형식 문자열

세 가지 모두 `flyql.Type`으로 통합되었습니다. 마이그레이션 시 이름 변경:

- `tuple` (ClickHouse) → `struct`
- `interval*` (CH/PG) → `duration`
- `geometry`, `special` → `unknown`
- `hstore` (PostgreSQL) → `map`
- `jsonb` (PostgreSQL) → `json`

이전에 `flyql.FromPlainObject({"col": {"normalized_type": "string"}})`를 호출했다면, 새로운 형태는 `{"col": {"type": "string"}}`입니다. 레거시 키는 이 페이지를 가리키는 대상 마이그레이션 에러를 생성합니다.

## Matcher는 의도적으로 스키마가 없습니다

인메모리 matcher는 `flyql.Type`을 소비**하지 않으며** 스키마 매개변수를 받지 않습니다. matcher에서의 동적 JSON 감지(`isProbablyJSONString` 휴리스틱)는 의도적입니다. 런타임에 matcher는 값을 직접 검사합니다. 매칭 전에 스키마 기반 사전 검증이 필요하면 `ColumnSchema`에 대해 먼저 `diagnose()`를 실행하세요.
