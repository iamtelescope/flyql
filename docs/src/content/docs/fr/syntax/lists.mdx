## Appartenance a une liste

Utilisez <FlyqlToken type="operator">in</FlyqlToken> pour verifier si une valeur se trouve dans une liste, et <FlyqlToken type="operator">not in</FlyqlToken> pour verifier qu'elle ne s'y trouve pas :

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['prod', 'staging']`} />

## Regles de syntaxe

- Les valeurs sont entre crochets <FlyqlToken type="operator">[ ]</FlyqlToken> et separees par des virgules
- Les valeurs peuvent etre de types mixtes : <FlyqlCode code={`[200, 'ok', true, null]`} inline />
- Les valeurs de type chaine doivent etre entre guillemets : <FlyqlCode code={`['a', 'b', 'c']`} inline />
- Les valeurs numeriques ne sont pas entre guillemets : <FlyqlCode code={`[1, 2, 3]`} inline />
- Les espaces autour des virgules et des crochets sont autorises

### Listes de chaines

<FlyqlCode code={`method in ['GET', 'POST', 'PUT']`} />
<FlyqlCode code={`env not in ['dev', 'test', 'local']`} />

Les guillemets simples et doubles fonctionnent pour les valeurs de type chaine.

### Listes numeriques

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`port not in [80, 443]`} />

### Listes vides

Les listes vides sont autorisees :

<FlyqlCode code={`status in []`} />

Une liste <FlyqlToken type="operator">in</FlyqlToken> vide est toujours fausse — rien ne correspond a une liste vide.

<FlyqlCode code={`status not in []`} />

Une liste <FlyqlToken type="operator">not in</FlyqlToken> vide est toujours vraie — tout est "pas dans" une liste vide.

## Types mixtes

Les listes prennent en charge des valeurs heterogenes — vous pouvez melanger chaines, nombres, booleens et null :

<FlyqlCode code={`field in [200, 'ok', true, null]`} />

Chaque element possede son propre type dans le tableau `valuesTypes` de l'expression analysee (par ex. `["int", "string", "bool", "null"]`). Pour les listes vides, `valuesTypes` est `null`.

Lorsqu'un generateur SQL traite une liste, il valide que les types de valeurs de la liste sont compatibles avec le type de la colonne cible. Par exemple, passer des valeurs de type chaine a une colonne numerique generera une erreur de validation lors de la generation SQL.

## Cohérence des types dans les listes IN

Le parser accepte **toujours** les listes hétérogènes — la validation de type vit dans les générateurs SQL, pas dans le parser. Chaque générateur (ClickHouse, PostgreSQL, StarRocks) applique la même logique de saut à deux couches avant de lever une erreur.

### Couche 1 — vérification au site d'appel

Avant d'invoquer le validateur, le générateur vérifie trois conditions :

1. La colonne a un **type déclaré** (`column.flyqlType()` n'est pas vide et n'est pas `Unknown`).
2. La clé **n'est pas segmentée** (un segment unique comme `category`, pas `meta.category`).
3. La liste est **homogène** — tous les éléments ont le même type de valeur (`len(set(values_types)) == 1` en Python ; `new Set(valuesTypes).size === 1` en JavaScript ; équivalent en Go).

Si **l'une** de ces conditions est fausse, le validateur est entièrement ignoré et le SQL est généré tel quel. Les listes hétérogènes passent. Les listes contre les colonnes sans type passent. Les listes contre les clés segmentées (chemins JSON, indices de tableau) passent.

### Couche 2 — vérification des éléments

Lorsque la vérification au site d'appel est satisfaite, le validateur itère sur chaque élément et lève une erreur sur la première valeur incompatible, selon cette carte de compatibilité :

| Type de colonne | Types d'élément autorisés |
|-----------------|---------------------------|
| `String`        | `string`                  |
| `Int`           | `int`, `float`            |
| `Float`         | `int`, `float`            |
| `Bool`          | `bool`, `int`             |
| `Date`          | `string`                  |

Les autres types de colonne (`Array`, `Map`, `JSON`, `JSONString`, `Struct`, `Duration`, `Unknown`) ne figurent pas dans la carte et contournent silencieusement la vérification des éléments.

### Le texte de l'erreur

Quand la validation échoue, les trois implémentations lèvent le message d'erreur **identique** :

<FlyqlCode code={`type mismatch in IN list: {column_type} column cannot contain {value_type} values`} />

Concrètement :

- Python : `FlyqlError("type mismatch in IN list: int column cannot contain string values")`
- Go : `errors.New("type mismatch in IN list: int column cannot contain string values")`
- JavaScript : `new Error("type mismatch in IN list: int column cannot contain string values")`

La sous-chaîne `type mismatch in IN list:` est stable dans les trois langages et les trois dialectes SQL, donc il est sûr d'y faire grep.

Pour une démonstration end-to-end exécutable des trois cas (correspondance homogène, contournement hétérogène, erreur de type incorrect homogène), voir [Recettes — Cohérence des types dans les listes IN](/fr/syntax/recipes/#cohérence-des-types-dans-les-listes-in).

## Parametres dans les listes

Les espaces reserves de parametres peuvent apparaitre dans les listes IN, melanges librement avec des valeurs litterales :

<FlyqlCode code={`status in [$x, $y, 200]`} />
<FlyqlCode code={`env not in [$current, 'dev']`} />

Les objets `Parameter` se trouvent dans le tableau `values` jusqu'a ce que `bindParams()` les resolve. Voir [Parametres](/fr/syntax/parameters/) pour la reference complete.

## Combinaison avec d'autres operateurs

Les operateurs de liste se combinent avec les operateurs booleens comme toute autre condition :

<FlyqlCode code={`status in [200, 201] and method in ['GET', 'POST']`} />
<FlyqlCode code={`level = 'error' or status not in [200, 201, 204]`} />
<FlyqlCode code={`(status in [500, 502, 503] or level = 'critical') and env='prod'`} />

## Exemples

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['dev', 'test', 'staging']`} />
<FlyqlCode code={`method in ['GET', 'POST'] and status not in [401, 403, 404]`} />
