## 리스트 포함 여부

값이 리스트에 포함되는지 확인하려면 `in`, 포함되지 않는지 확인하려면 `not in`을 사용합니다.

```
status in [200, 201, 204]
env not in ['prod', 'staging']
```

## 구문 규칙

- 값은 대괄호 `[ ]`로 감싸고 쉼표로 구분합니다
- 리스트의 모든 값은 같은 타입이어야 합니다 (모두 문자열 또는 모두 숫자)
- 문자열 값은 따옴표로 감싸야 합니다: `['a', 'b', 'c']`
- 숫자 값은 따옴표 없이 작성합니다: `[1, 2, 3]`
- 쉼표와 대괄호 주변 공백은 허용됩니다

### 문자열 리스트

```
method in ['GET', 'POST', 'PUT']
env not in ['dev', 'test', 'local']
```

문자열 값에는 작은따옴표와 큰따옴표를 모두 사용할 수 있습니다.

### 숫자 리스트

```
status in [200, 201, 204]
port not in [80, 443]
```

### 빈 리스트

빈 리스트도 허용됩니다.

```
status in []        # always false — nothing matches an empty list
status not in []    # always true — everything is "not in" an empty list
```

## 타입 일관성

리스트에는 **동일한 타입의 값만** 들어 있어야 합니다. 하나의 리스트 안에 타입이 섞이면 parse error가 발생합니다.

```
# Valid — all numbers
status in [200, 201, 204]

# Valid — all strings
env in ['prod', 'staging']

# Invalid — mixed types (parse error)
values in [200, 'ok']
```

Parser는 리스트의 첫 번째 값 타입을 추적하고, 이후 다른 타입의 값이 나오면 거부합니다. 이 검사는 쿼리가 generator나 matcher에 도달하기 전에 parse 시점에 수행됩니다.

SQL generator가 리스트를 처리할 때는, 리스트 값 타입이 대상 컬럼 타입과 호환되는지도 추가로 검증합니다. 예를 들어 숫자 컬럼에 문자열 값을 넘기면 SQL 생성 중 validation error가 발생합니다.

Parse된 expression의 `valuesType` 필드는 리스트가 `"string"` 값인지 `"number"` 값인지 나타냅니다. 빈 리스트의 경우 `valuesType`은 `null`입니다.

## 다른 연산자와의 조합

리스트 연산자는 다른 조건과 마찬가지로 불리언 연산자와 결합할 수 있습니다.

```
status in [200, 201] and method in ['GET', 'POST']
level = error or status not in [200, 201, 204]
(status in [500, 502, 503] or level = critical) and env=prod
```

## 예제

```
# HTTP 성공 코드
status in [200, 201, 204]

# 테스트 환경 제외
env not in ['dev', 'test', 'staging']

# 여러 리스트 조건
method in ['GET', 'POST'] and status not in [401, 403, 404]
```
