## Regex Matching

Use <FlyqlToken type="operator">{'~'}</FlyqlToken> to match a value against a regular expression, and <FlyqlToken type="operator">{'!~'}</FlyqlToken> to negate the match:

<FlyqlCode code={`message ~ "error.*timeout"`} />
<FlyqlCode code={`path !~ "^/health"`} />

| Operator | Meaning |
|----------|---------|
| <FlyqlToken type="operator">{'~'}</FlyqlToken> | Value matches regex pattern |
| <FlyqlToken type="operator">{'!~'}</FlyqlToken> | Value does not match regex pattern |

### Regex Pattern Syntax

The regex syntax depends on the target backend:

- **ClickHouse** — uses `match()` function (RE2 syntax)
- **PostgreSQL** — uses <FlyqlToken type="operator">{'~'}</FlyqlToken> and <FlyqlToken type="operator">{'!~'}</FlyqlToken> operators (POSIX regex)
- **StarRocks** — uses `regexp` function
- **In-memory matcher** — uses the host language's regex engine (Go `regexp`, Python `re2`, JS `RegExp`)

Quote the pattern when it contains spaces or special characters:

<FlyqlCode code={`message ~ "connection (refused|reset)"`} />
<FlyqlCode code={`host ~ "^prod-[0-9]+"`} />

### Regex Flags

You can modify regex behavior using inline flags at the start of your pattern with the `(?...)` syntax. FlyQL supports three flags that work across all backends:

#### `(?i)` — Case-Insensitive

Makes the pattern match regardless of letter case:

<FlyqlCode code={`message ~ "(?i)error"`} />
<FlyqlCode code={`level ~ "(?i)^(warn|error|fatal)$"`} />

`(?i)error` matches "Error", "ERROR", "error", etc.

#### `(?m)` — Multiline

Changes `^` and `$` to match the start and end of each **line** instead of the entire string:

<FlyqlCode code={`log ~ "(?m)^ERROR"`} />

Without `(?m)`, `^ERROR` only matches if the entire string starts with "ERROR". With `(?m)`, it also matches "INFO first\nERROR second" because "ERROR" appears at the start of a line.

#### `(?s)` — Dotall

Makes `.` match **any** character including newlines. By default, `.` does not match `\n`:

<FlyqlCode code={`log ~ "(?s)start.*end"`} />

Without `(?s)`, `start.*end` does not match "start\nend" because `.` won't cross the newline. With `(?s)`, it matches.

#### Combining Flags

Flags can be combined in a single group:

<FlyqlCode code={`log ~ "(?im)^ERROR"`} />
<FlyqlCode code={`log ~ "(?ims)^START.*END$"`} />

#### Backend Support

All three flags work across every FlyQL backend:

| Backend | `(?i)` | `(?m)` | `(?s)` |
|---------|--------|--------|--------|
| **ClickHouse** | Yes | Yes | Yes |
| **PostgreSQL** | Yes | Yes | Yes |
| **StarRocks** | Yes | Yes | Yes |
| **Go matcher** | Yes | Yes | Yes |
| **Python matcher** | Yes | Yes | Yes |
| **JavaScript matcher** | Yes | Yes | Yes |

Combine case-insensitive matching with negation to exclude patterns regardless of case:

<FlyqlCode code={`message ~ "(?i)error" and message !~ "(?i)warning"`} />

## LIKE Matching

Use <FlyqlToken type="operator">like</FlyqlToken> and <FlyqlToken type="operator">ilike</FlyqlToken> for SQL-style pattern matching with wildcards:

<FlyqlCode code={`host like 'prod%'`} />
<FlyqlCode code={`service like 'api-%'`} />
<FlyqlCode code={`message ilike '%Error%'`} />

| Operator | Meaning |
|----------|---------|
| <FlyqlToken type="operator">like</FlyqlToken> | Value matches pattern (case-sensitive) |
| <FlyqlToken type="operator">not like</FlyqlToken> | Value does not match pattern |
| <FlyqlToken type="operator">ilike</FlyqlToken> | Value matches pattern (case-insensitive) |
| <FlyqlToken type="operator">not ilike</FlyqlToken> | Value does not match pattern (case-insensitive) |

### Wildcards

Two wildcard characters are available in <FlyqlToken type="operator">like</FlyqlToken> / <FlyqlToken type="operator">ilike</FlyqlToken> patterns:

| Wildcard | Meaning | Example | Matches |
|----------|---------|---------|---------|
| `%` | Any sequence of characters (including empty) | <FlyqlCode code={`host like 'prod%'`} inline /> | `prod`, `prod-1`, `production` |
| `_` | Exactly one character | <FlyqlCode code={`code like 'E__'`} inline /> | `E01`, `Exx`, but not `E1` or `Error` |

Wildcards can appear anywhere in the pattern — start, middle, end, or combined:

<FlyqlCode code={`message like '%timeout%'`} />
<FlyqlCode code={`host like 'web-_-prod'`} />
<FlyqlCode code={`path like '/api/%/status'`} />

### Escaping Wildcards

To match a literal `%` or `_`, escape it with a backslash:

<FlyqlCode code={`value like '100\\%'`} />
<FlyqlCode code={`col like 'foo\\_bar'`} />

`100\%` matches the literal string "100%", not "100" followed by anything.

### Case-Insensitive Matching

Use <FlyqlToken type="operator">ilike</FlyqlToken> for case-insensitive pattern matching:

<FlyqlCode code={`message ilike '%error%'`} />
<FlyqlCode code={`host not ilike 'TEST%'`} />

### LIKE vs. Regex

| Use case | Syntax | Example |
|----------|--------|---------|
| Prefix match | <FlyqlCode code={`key like 'prefix%'`} inline /> | <FlyqlCode code={`host like 'prod%'`} inline /> |
| Contains | <FlyqlCode code={`key like '%word%'`} inline /> | <FlyqlCode code={`msg like '%error%'`} inline /> |
| Complex pattern | <FlyqlCode code={`key ~ "pattern"`} inline /> | <FlyqlCode code={`host ~ "^prod-[0-9]+"`} inline /> |

Use <FlyqlToken type="operator">like</FlyqlToken> for simple wildcard matching. Use regex (<FlyqlToken type="operator">{'~'}</FlyqlToken>) when you need character classes, alternation, or other advanced patterns.

### Backend Support

All SQL backends (ClickHouse, PostgreSQL, StarRocks) emit native `LIKE` / `ILIKE` / `NOT LIKE` / `NOT ILIKE` SQL operators. The in-memory matcher converts the pattern to a regex internally.

:::note
StarRocks `ILIKE` requires version 2.5 or later.
:::

## Examples

<FlyqlCode code={`message ~ "error|fail"`} />
<FlyqlCode code={`path ~ "^/api/v2"`} />
<FlyqlCode code={`path !~ "^/health"`} />
<FlyqlCode code={`host like 'prod%'`} />
<FlyqlCode code={`host like 'prod%' and message ~ "timeout.*" and path !~ "^/health"`} />
