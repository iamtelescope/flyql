FlyQL कॉलम स्कीमा में **कैलेंडर दिनों** (`Type.Date`) को **समय के क्षणों** (`Type.DateTime`) से अलग करता है, और इस भेद का उपयोग ISO-8601 स्ट्रिंग लिटरल पर मैचर कोएर्शन संचालित करने के लिए करता है। इसके विपरीत, SQL जनरेटर स्ट्रिंग लिटरल को जैसा-का-तैसा पास करते हैं — डेटाबेस उन्हें पार्स करता है।

## कैलेंडर दिन बनाम क्षण

`Type.Date` — केवल कैलेंडर दिन। तुलनाएँ `YYYYMMDD` तक ट्रंकेट होती हैं। कच्चे DB प्रकार `date` / `date32` को अवशोषित करता है।

`Type.DateTime` — टाइमलाइन पर बिंदु, मिलीसेकंड रिज़ॉल्यूशन पर। तुलनाएँ UTC मिलीसेकंड में होती हैं। `datetime`, `datetime64`, `timestamp`, `timestamptz`, `year` को अवशोषित करता है।

सिमेंटिक प्रकार घोषित करना मैचर के लिए महत्वपूर्ण है: `'2026-04-06T15:30:00Z'` की तुलना में एक `Date` कॉलम रिकॉर्ड मान को तुलना से पहले उसके Y/M/D घटक तक ट्रंकेट करता है, जबकि एक `DateTime` कॉलम मिलीसेकंड तक तुलना करता है।

यदि `Type.Date` घोषित कॉलम को datetime-आकार का मान मिलता है (मूल `datetime` / `time.Time` / `Date` इंस्टेंस, या समय घटक वाला ISO स्ट्रिंग), तो FlyQL `Type.DateTime` में माइग्रेशन का सुझाव देते हुए प्रति कॉलम एक बार `UserWarning` (Python) / लॉग लाइन (Go) / `console.warn` (JS) उत्सर्जित करता है।

## स्ट्रिंग लिटरल के रूप में दिनांक

एक date/datetime कॉलम की तुलना ISO-8601 स्ट्रिंग लिटरल से करें। जनरेटर लिटरल को जैसा-का-तैसा पास करते हैं; डेटाबेस निष्पादन समय पर इसे पार्स करता है।

<FlyqlCode code={`due_date < '2023-12-31'`} />
<FlyqlCode code={`created_at >= '2023-12-31T00:00:00Z'`} />
<FlyqlCode code={`event_time >= '2023-01-01' and event_time <= '2023-12-31'`} />

मैचर में, स्कीमा-घोषित `Date` / `DateTime` कॉलम ISO-8601 लिटरल और रिकॉर्ड मान दोनों की स्वचालित पार्सिंग संचालित करते हैं। सामान्य मामले के लिए किसी स्पष्ट रूपांतरण फ़ंक्शन की आवश्यकता नहीं है।

### स्वीकार्य iso8601 रूप

मैचर का `iso8601` परिवार उदार है:

- `YYYY-MM-DD` — शुद्ध कैलेंडर तिथि (Date कॉलम के लिए, या DateTime कॉलम के लिए कॉलम-tz में मध्यरात्रि)।
- `YYYY-MM-DDTHH:MM:SS` — नेव datetime।
- `YYYY-MM-DD HH:MM:SS` — स्पेस सेपरेटर वाला नेव datetime (पार्सिंग से पहले सामान्यीकृत)।
- `YYYY-MM-DDTHH:MM:SS.ffffff` — वैकल्पिक उप-सेकंड (केवल ms रिज़ॉल्यूशन — नीचे "परिशुद्धता" देखें)।
- उपरोक्त में से कोई भी `Z` या `±HH:MM` ऑफसेट प्रत्यय के साथ।

### नेव लिटरल के लिए कॉलम मेटाडेटा

एक `DateTime` कॉलम घोषणा वैकल्पिक मेटाडेटा ले सकती है:

- `tz` — नेव ISO स्ट्रिंग्स की व्याख्या के लिए IANA समय-क्षेत्र नाम। Evaluator के `default_timezone` पर, फिर `"UTC"` पर फॉलबैक करता है।
- `unit` — `"ms"` (डिफ़ॉल्ट), `"s"`, या `"ns"`। संख्यात्मक रिकॉर्ड मानों पर लागू; गैर-संख्यात्मक मान इसे अनदेखा करते हैं।

```json
{
  "ts":        { "type": "datetime", "tz": "Europe/Moscow", "unit": "s" },
  "event_day": { "type": "date" }
}
```

:::note[स्ट्रिंग्स को कोट किया जाना चाहिए]
दिनांक लिटरल स्ट्रिंग्स हैं और उन्हें **अनिवार्य रूप से** कोट किया जाना चाहिए। बिना कोट के, FlyQL मान को कॉलम संदर्भ के रूप में व्याख्या करता है। <FlyqlCode code={`due_date < 2023-12-31`} inline /> को `2023` नामक कॉलम पर अंकगणित के रूप में पार्स किया जाएगा, दिनांक लिटरल के रूप में नहीं। देखें [मान और प्रकार — String](/hi/syntax/values/#string)।
:::

## टेम्पोरल फ़ंक्शन

सापेक्ष समय क्वेरीज़ के लिए — "पिछले एक घंटे में", "इस सप्ताह की शुरुआत से", "आज से" — FlyQL अंतर्निहित टेम्पोरल फ़ंक्शन प्रदान करता है: <FlyqlCode code={`ago(...)`} inline />, <FlyqlCode code={`now()`} inline />, <FlyqlCode code={`today()`} inline /> और <FlyqlCode code={`startOf(...)`} inline />।

<FlyqlCode code={`created_at > ago(1h)`} />
<FlyqlCode code={`updated_at >= ago(7d)`} />
<FlyqlCode code={`expires_at < now()`} />
<FlyqlCode code={`event_time > startOf('day')`} />
<FlyqlCode code={`scheduled_for >= today('Europe/Berlin')`} />

पूरी व्याकरण (इकाइयाँ, क्रम नियम, समय-क्षेत्र तर्क) [मान और प्रकार — टेम्पोरल फ़ंक्शन](/hi/syntax/values/#temporal-functions) में शामिल है।

## डेलाइट सेविंग टाइम (DST)

जब एक नेव datetime लिटरल (कोई ऑफसेट नहीं, कोई `Z` नहीं) DST-पालक tz में व्याख्या किया जाता है, FlyQL स्पेक द्वारा परिभाषित असंदिग्ध समाधान चुनता है:

- **फॉल-बैक (अस्पष्ट वॉल क्लॉक)** — पहले वाला घटना चुना जाता है (Python सिमेंटिक्स में `fold=0`)। उदाहरण: `America/New_York` में `'2026-11-01 01:30:00'` 05:30 UTC (EDT) पर हल होता है, 06:30 UTC (EST) पर नहीं।
- **स्प्रिंग-फॉरवर्ड (अस्तित्वहीन वॉल क्लॉक)** — मान अस्वीकार कर दिया जाता है। मैचर उस रिकॉर्ड के लिए `false` लौटाता है (छोड़ दिया जाता है)। उदाहरण: `America/New_York` में `'2026-03-08 02:30:00'` मौजूद नहीं है।

स्प्रिंग-फॉरवर्ड गैप में गिरने वाले रिकॉर्ड को चुपचाप आगे सामान्यीकृत करने के बजाय अ-पार्स योग्य के रूप में माना जाता है।

## समर्थित रिकॉर्ड-मान प्रकार (Python)

- `datetime.datetime` — tz-aware या नेव (नेव कॉलम `tz` या Evaluator डिफ़ॉल्ट का उपयोग करता है)।
- `datetime.date` — केवल कैलेंडर दिन।
- `pandas.Timestamp` — काम करता है क्योंकि यह `datetime` की उपवर्ग है।
- ISO-8601 स्ट्रिंग्स — ऊपर स्वीकार्य रूप देखें।
- `int` / `float` — डिफ़ॉल्ट रूप से epoch मिलीसेकंड; epoch सेकंड के लिए कॉलम `unit="s"` या नैनोसेकंड के लिए `"ns"` सेट करें।

आज समर्थित नहीं:

- `numpy.datetime64` — मैचर को पास करने से पहले परिवर्तित करें: `pd.Timestamp(value).to_pydatetime()` या `value.astype("datetime64[ms]").astype(int)`।

## परिशुद्धता

FlyQL का मैचर टेम्पोरल मानों की तुलना **मिलीसेकंड ग्रैन्युलैरिटी** पर करता है। स्रोत डेटा में उप-मिलीसेकंड परिशुद्धता तुलना से पहले ट्रंकेट हो जाती है:

- Python — `int(dt.timestamp() * 1000)` माइक्रोसेकंड्स को छोड़ देता है।
- Go — `UnixMilli()` नैनोसेकंड्स को छोड़ देता है।
- JavaScript — `Date` मूल रूप से ms है।

जिन उपयोगकर्ताओं को नैनोसेकंड परिशुद्धता चाहिए, उन्हें रिकॉर्ड को bigint नैनोसेकंड में प्री-प्रोसेस करना होगा और `unit="ns"` घोषित करना होगा। JS पथ `Number` का उपयोग करता है जो `2^53` से ऊपर परिशुद्धता खोता है — नैनोसेकंड BigInt पथ बाद के स्पेक के लिए आरक्षित है।

## स्कीमा-रहित संख्यात्मक

कॉलम स्कीमा के बिना संख्यात्मक रिकॉर्ड मानों को **epoch मिलीसेकंड** के रूप में माना जाता है। कोई ऑटो-डिटेक्शन हेयुरिस्टिक नहीं — एक रिकॉर्ड `{"ts": 1712434800}` (परंपरा के अनुसार epoch सेकंड) को `1970-01-20` के रूप में व्याख्या किया जाता है, `2024` नहीं। epoch-seconds डेटा वाले उपयोगकर्ताओं को `unit="s"` घोषित करना ज़रूरी है:

```json
{ "ts": { "type": "datetime", "unit": "s" } }
```

यह डिज़ाइन से सख्त है: एक ऑटो-डिटेक्शन हेयुरिस्टिक (जैसे `value < 10^10 → seconds`) सीमा पर और वैध पुरानी तिथि मानों पर विफल होती है, इसलिए हम चुप्पी अनुमानों पर स्पष्ट घोषणाओं को प्राथमिकता देते हैं।

## कब क्या उपयोग करें

| उद्देश्य | उपयोग करें | उदाहरण |
|---------|-----------|--------|
| एक निश्चित कैलेंडर तिथि या टाइमस्टैम्प से तुलना | स्ट्रिंग लिटरल | <FlyqlCode code={`due_date < '2023-12-31'`} inline /> |
| एक स्लाइडिंग समय विंडो से तुलना | <FlyqlCode code={`ago()`} inline /> | <FlyqlCode code={`created_at > ago(1h)`} inline /> |
| वर्तमान क्षण से तुलना | <FlyqlCode code={`now()`} inline /> | <FlyqlCode code={`expires_at < now()`} inline /> |
| आज से तुलना (समय-क्षेत्र-जागरूक) | <FlyqlCode code={`today()`} inline /> | <FlyqlCode code={`scheduled_for = today('UTC')`} inline /> |
| दिन/सप्ताह/महीने की शुरुआत से तुलना | <FlyqlCode code={`startOf()`} inline /> | <FlyqlCode code={`event_time > startOf('week')`} inline /> |

स्ट्रिंग लिटरल स्थिर रहते हैं — एक ही क्वेरी स्ट्रिंग दिन-प्रतिदिन समान SQL देती है। टेम्पोरल फ़ंक्शन निष्पादन समय पर मूल्यांकित होते हैं, इसलिए वही FlyQL क्वेरी हर बार चलने पर अलग समय विंडो उत्पन्न करती है।
