## リスト所属

<FlyqlToken type="operator">in</FlyqlToken> は値がリストに含まれるかを、<FlyqlToken type="operator">not in</FlyqlToken> は含まれないかを判定します。

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['prod', 'staging']`} />

## 構文ルール

- 値は角括弧 <FlyqlToken type="operator">[ ]</FlyqlToken> で囲み、カンマ区切りで記述します
- リスト内の値はすべて同じ型である必要があります（文字列のみ、または数値のみ）
- 文字列はクォートが必要です: <FlyqlCode code={`['a', 'b', 'c']`} inline />
- 数値はクォートしません: <FlyqlCode code={`[1, 2, 3]`} inline />
- カンマや角括弧の前後のスペースは任意です

### 文字列リスト

<FlyqlCode code={`method in ['GET', 'POST', 'PUT']`} />
<FlyqlCode code={`env not in ['dev', 'test', 'local']`} />

文字列値にはシングルクォートとダブルクォートのどちらも使えます。

### 数値リスト

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`port not in [80, 443]`} />

### 空リスト

空リストも使用できます。

<FlyqlCode code={`status in []`} />

空の <FlyqlToken type="operator">in</FlyqlToken> リストは常に false です。空リストに一致するものはありません。

<FlyqlCode code={`status not in []`} />

空の <FlyqlToken type="operator">not in</FlyqlToken> リストは常に true です。すべてが空リストに「not in」です。

## 混在型

リストは異種値をサポートします。文字列・数値・真偽値・null を混在させることができます。

<FlyqlCode code={`field in [200, 'ok', true, null]`} />

parse 済み expression の `valuesTypes` 配列に各要素の型が記録されます（例: `["int", "string", "bool", "null"]`）。空リストの場合、`valuesTypes` は `null` です。

SQL generator はリストを処理する際、リスト値の型が対象カラムの型と互換性があるかを検証します。たとえば数値カラムに文字列値を渡すと、SQL 生成時に validation error になります。

## IN リストの型整合性

parser は **常に** 異種型リストを受け付けます — 型検証は parser ではなく SQL generator に存在します。各 generator（ClickHouse、PostgreSQL、StarRocks）はエラーをスローする前に同じ二段階のスキップロジックを適用します。

### 第 1 層 — 呼び出し側ゲート

validator を呼び出す前に、generator は次の 3 つの条件をチェックします:

1. カラムに**宣言された型**がある（`column.flyqlType()` が空でも `Unknown` でもない）。
2. key が**分割されていない**（`category` のような単一セグメント。`meta.category` ではない）。
3. リストが**同質**である — 全要素が同じ値型を持つ（Python では `len(set(values_types)) == 1`、JavaScript では `new Set(valuesTypes).size === 1`、Go でも同様）。

これらのうち**いずれか**が偽であれば、validator は完全にスキップされ、SQL はそのまま生成されます。異種型リスト、型のないカラムに対するリスト、分割 key（JSON パス、配列インデックス）に対するリストはすべて通過します。

### 第 2 層 — 要素チェック

呼び出し側ゲートを通過した場合、validator は各要素を反復して、次の互換性マップに従って最初の非互換値でエラーをスローします:

| カラム型 | 許容される要素型 |
|----------|------------------|
| `String` | `string`         |
| `Int`    | `int`、`float`   |
| `Float`  | `int`、`float`   |
| `Bool`   | `bool`、`int`    |
| `Date`   | `string`         |

その他のカラム型（`Array`、`Map`、`JSON`、`JSONString`、`Struct`、`Duration`、`Unknown`）はマップに無く、要素チェックを暗黙的にバイパスします。

### エラー文字列

検証に失敗した場合、3 つの実装すべてが**同一の**エラーメッセージをスローします:

<FlyqlCode code={`type mismatch in IN list: {column_type} column cannot contain {value_type} values`} />

具体的には:

- Python: `FlyqlError("type mismatch in IN list: int column cannot contain string values")`
- Go: `errors.New("type mismatch in IN list: int column cannot contain string values")`
- JavaScript: `new Error("type mismatch in IN list: int column cannot contain string values")`

サブストリング `type mismatch in IN list:` は 3 言語・3 SQL 方言で安定しているため、grep で安全に検索できます。

3 ケース（同質マッチ、異種型バイパス、同質型違いエラー）の実行可能なエンドツーエンドのデモは、[レシピ — IN リストの型整合性](/ja/syntax/recipes/#in-リストの型整合性) を参照してください。

## リスト内のパラメータ

パラメータプレースホルダは IN リスト内にも使え、リテラル値と自由に混在させることができます。

<FlyqlCode code={`status in [$x, $y, 200]`} />
<FlyqlCode code={`env not in [$current, 'dev']`} />

`Parameter` オブジェクトは、`bindParams()` で解決されるまで `values` 配列の中に残ります。完全なリファレンスは [パラメータ](/ja/syntax/parameters/) を参照してください。

## 他の演算子との組み合わせ

リスト演算子は、他の条件と同様にブール演算子と組み合わせて使えます。

<FlyqlCode code={`status in [200, 201] and method in ['GET', 'POST']`} />
<FlyqlCode code={`level = 'error' or status not in [200, 201, 204]`} />
<FlyqlCode code={`(status in [500, 502, 503] or level = 'critical') and env='prod'`} />

## 例

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['dev', 'test', 'staging']`} />
<FlyqlCode code={`method in ['GET', 'POST'] and status not in [401, 403, 404]`} />
