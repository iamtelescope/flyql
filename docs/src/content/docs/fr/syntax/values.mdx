FlyQL prend en charge 10 types de valeurs : **Integer**, **BigInt**, **Float**, **String**, **Boolean**, **Null**, **Array**, **Column**, **Function** et **Parameter**. Chaque valeur dans l'AST possede un champ `value_type` explicite.

## Types de valeurs

### Integer

Nombres entiers dans la plage int64 :

<FlyqlCode code={`count = 42`} />
<FlyqlCode code={`status != 200`} />
<FlyqlCode code={`age >= 18`} />

AST : `value_type: "integer"`

### BigInt

Entiers depassant la plage int64 (Go/Python) ou `MAX_SAFE_INTEGER` (JavaScript) :

<FlyqlCode code={`id = 18446744073709551615`} />

AST : `value_type: "bigint"`

### Float

Nombres avec un point decimal :

<FlyqlCode code={`price > 9.99`} />
<FlyqlCode code={`ratio = 3.14`} />
<FlyqlCode code={`temperature<=-0.5`} />

AST : `value_type: "float"`

### String

Les valeurs textuelles **doivent etre entre guillemets** simples ou doubles. Les valeurs sans guillemets sont interpretees comme des references de colonne, pas comme des chaines.

<FlyqlCode code={`level = 'error'`} />
<FlyqlCode code={`user = "John Doe"`} />
<FlyqlCode code={`message = 'hello world'`} />
<FlyqlCode code={`count = '42'`} />

AST : `value_type: "string"`

Echappez les guillemets avec un antislash :

<FlyqlCode code={`user = 'John\\'s'`} />
<FlyqlCode code={`message = "said \\"hello\\""`} />

:::tip
Les guillemets simples et doubles sont interchangeables : choisissez le delimiteur qui evite d'avoir a echapper. Utilisez les guillemets doubles quand la valeur contient des apostrophes, et les guillemets simples quand elle contient des guillemets doubles :

<FlyqlCode code={`user = "John's"`} />
<FlyqlCode code={`message = 'said "hello"'`} />
:::

### Boolean

Les mots-cles <FlyqlToken type="boolean">true</FlyqlToken> et <FlyqlToken type="boolean">false</FlyqlToken> (en minuscules, sensible a la casse) :

<FlyqlCode code={`active = true`} />
<FlyqlCode code={`debug = false`} />
<FlyqlCode code={`active != true`} />

AST : `value_type: "boolean"`, la valeur est un booleen natif.

Les variantes en majuscules (`True`, `FALSE`) sont traitees comme des references de colonne, pas comme des booleens.

**Generation SQL :**
| Dialecte | FlyQL | SQL genere |
|----------|-------|------------|
| ClickHouse | <FlyqlCode code={`field = true`} inline /> | `field = true` |
| ClickHouse | <FlyqlCode code={`field = false`} inline /> | `field = false` |
| PostgreSQL | <FlyqlCode code={`field = true`} inline /> | `field = TRUE` |
| PostgreSQL | <FlyqlCode code={`field = false`} inline /> | `field = FALSE` |
| StarRocks | <FlyqlCode code={`field = true`} inline /> | `field = true` |
| StarRocks | <FlyqlCode code={`field = false`} inline /> | `field = false` |

### Null

Le mot-cle <FlyqlToken type="null">null</FlyqlToken> (en minuscules, sensible a la casse) :

<FlyqlCode code={`field = null`} />
<FlyqlCode code={`field != null`} />

AST : `value_type: "null"`, la valeur est un null natif.

Les variantes en majuscules (`Null`, `NULL`) sont traitees comme des references de colonne, pas comme null.

### Reference de colonne

Toute valeur sans guillemets qui n'est pas un nombre, un booleen, null ou une fonction temporelle est traitee comme une **reference de colonne** — un pointeur vers une autre colonne pour des comparaisons colonne-a-colonne :

<FlyqlCode code={`price > min_price`} />
<FlyqlCode code={`updated_at > created_at`} />
<FlyqlCode code={`field = otherfield`} />

AST : `value_type: "column"`

Si vous souhaitez comparer a une valeur de type chaine, vous **devez** utiliser des guillemets :

<FlyqlCode code={`level = 'error'`} />

Sans guillemets, <FlyqlCode code={`level = error`} inline /> traite `error` comme une reference a une colonne nommee `error`.

**Restrictions :** Seuls les operateurs <FlyqlToken type="operator">=</FlyqlToken> et <FlyqlToken type="operator">!=</FlyqlToken> sont valides avec null. L'utilisation de <FlyqlToken type="operator">{'>'}</FlyqlToken>, <FlyqlToken type="operator">{'<'}</FlyqlToken>, <FlyqlToken type="operator">{'>='}</FlyqlToken>, <FlyqlToken type="operator">{'<='}</FlyqlToken>, <FlyqlToken type="operator">{'~'}</FlyqlToken> ou <FlyqlToken type="operator">{'!~'}</FlyqlToken> avec null produit une erreur d'analyse.

**Generation SQL :** <FlyqlCode code={`field = null`} inline /> produit `field IS NULL`, <FlyqlCode code={`field != null`} inline /> produit `field IS NOT NULL` (tous les dialectes).

### Array

Les tableaux sont utilises avec les operateurs <FlyqlToken type="operator">in</FlyqlToken> et <FlyqlToken type="operator">not in</FlyqlToken>. Les elements peuvent etre de types mixtes (heterogenes) :

<FlyqlCode code={`status in [200, 201, 202]`} />
<FlyqlCode code={`name in ['alice', 'bob']`} />
<FlyqlCode code={`field in [1, "hello", true, null]`} />

Chaque element possede son propre type dans le tableau `values_types`.

## Expressions truthy/falsy

Une cle autonome sans operateur verifie si le champ a une valeur vraie (truthy) :

<FlyqlCode code={`active`} />
<FlyqlCode code={`message and status`} />

Une valeur est consideree **falsy** si elle est :
- `null` / `None` / absente
- Chaine vide `""`
- Zero `0`
- Booleen <FlyqlToken type="boolean">false</FlyqlToken>

Tout le reste est **truthy**.

Utilisez <FlyqlToken type="operator">not</FlyqlToken> pour verifier les valeurs falsy :

<FlyqlCode code={`not archived`} />
<FlyqlCode code={`active and not debug`} />

Les verifications truthy/falsy sont distinctes des comparaisons avec null : <FlyqlCode code={`not field`} inline /> correspond lorsque le champ est null, vide ou zero. <FlyqlCode code={`field = null`} inline /> correspond uniquement lorsque le champ est specifiquement null.

## Espaces

FlyQL tolere les espaces. Les espaces autour des operateurs sont facultatifs :

<FlyqlCode code={`status = 200`} />
<FlyqlCode code={`status =200`} />

Tous les exemples ci-dessus sont equivalents.

## Constantes LiteralKind

Les valeurs d'expression analysees portent une etiquette `LiteralKind`. Les constantes sont exportees depuis le package racine dans chaque implementation.

**Python :**
```python
from flyql import LiteralKind
# LiteralKind.INTEGER, LiteralKind.BIGINT, LiteralKind.FLOAT,
# LiteralKind.STRING, LiteralKind.BOOLEAN, LiteralKind.NULL, LiteralKind.ARRAY
```

**Go :**
```go
import "github.com/iamtelescope/flyql/golang/literal"
// literal.Integer, literal.BigInt, literal.Float,
// literal.String, literal.Boolean, literal.Null, literal.Array
```

**JavaScript :**
```javascript
import { LiteralKind } from 'flyql'
// LiteralKind.INTEGER, LiteralKind.BIGINT, LiteralKind.FLOAT,
// LiteralKind.STRING, LiteralKind.BOOLEAN, LiteralKind.NULL, LiteralKind.ARRAY
```

## Fonctions temporelles

FlyQL prend en charge les appels de fonctions temporelles comme valeurs du cote droit des expressions de comparaison. Elles permettent des filtres relatifs au temps sans coder en dur les horodatages.

AST : `value_type: "function"`, `value` est un objet `FunctionCall`.

### `ago(duration)`

Renvoie un horodatage relatif a l'heure actuelle. Prend en charge les durees composees.

<FlyqlCode code={`timestamp > ago(1h)`} />
<FlyqlCode code={`created_at >= ago(7d)`} />
<FlyqlCode code={`updated_at > ago(1h30m)`} />

**Unites (minuscules) :** `s` (secondes), `m` (minutes), `h` (heures), `d` (jours), `w` (semaines).

**Grammaire.** Une duree est une ou plusieurs paires `<entier><unite>`. Les unites doivent apparaitre dans un **ordre strictement decroissant** et **au plus une fois chacune** : `w > d > h > m > s`. Cela suit la convention Prometheus / Grafana / Loki.

Valide :

<FlyqlCode code={`ago(30s)`} />
<FlyqlCode code={`ago(2h15m)`} />
<FlyqlCode code={`ago(1w2d3h4m5s)`} />
<FlyqlCode code={`ago(1w30s)`} />

Rejete lors de l'analyse :

- `ago(30m1h)` — unites dans le mauvais ordre
- `ago(1h1h)` — unite repetee
- `ago(3h1w)` — unite plus grande apres plus petite
- `ago(1H)` — les lettres d'unite doivent etre en minuscules

:::note[Pourquoi pas de mois / annee ?]
Les unites de mois et d'annee ne sont **pas** prises en charge intentionnellement : leur longueur varie (28 a 31 jours, 365/366 jours), de sorte que le meme litteral signifierait un nombre different de secondes selon le contexte. Les fenetres temporelles relatives au calendrier sont exprimees via [`startOf('month')`](#startofunit).
:::

### `now()`

Renvoie l'horodatage actuel.

<FlyqlCode code={`expires_at < now()`} />

### `today()`

Renvoie la date du jour. Accepte un argument optionnel de fuseau horaire IANA.

<FlyqlCode code={`date = today()`} />
<FlyqlCode code={`date = today('Europe/Berlin')`} />

### `startOf(unit)`

Renvoie le debut du jour, de la semaine (lundi) ou du mois en cours. Accepte un fuseau horaire optionnel.

<FlyqlCode code={`created_at > startOf('day')`} />
<FlyqlCode code={`created_at > startOf('week')`} />
<FlyqlCode code={`created_at > startOf('month', 'Asia/Tokyo')`} />

**Unites valides :** `'day'`, `'week'`, `'month'`

### Restrictions sur les operateurs

Les fonctions temporelles ne peuvent etre utilisees qu'avec les operateurs de comparaison : `=`, `!=`, `>`, `>=`, `<`, `<=`.

L'utilisation de `~` ou `!~` avec une fonction temporelle produit une erreur d'analyse. Les operateurs comme `in`, `has`, `like` et `ilike` utilisent des chemins d'analyse separes et ne sont pas compatibles avec les valeurs de fonction.

### Reference de colonne sans parentheses

Les noms de fonctions sans parentheses sont traites comme des references de colonne, pas comme des appels de fonction :

<FlyqlCode code={`field = ago`} />

Cela produit `value_type: "column"`, pas un appel de fonction.

## Parameter

Les espaces reserves de parametres (`$name` pour les nommes, `$1` pour les positionnels) remplacent des valeurs a resoudre a l'execution par `bindParams()`. Ils peuvent apparaitre partout ou une valeur litterale est valide — y compris dans les listes IN, les arguments de fonctions temporelles et les arguments de transformateurs.

<FlyqlCode code={`status = $code`} />
<FlyqlCode code={`level in [$primary, 'warn']`} />
<FlyqlCode code={`created > ago($duration)`} />

AST : `value_type: "parameter"`, la valeur est un objet `Parameter` avec les champs `name` et `positional`.

Voir [Parametres](/fr/syntax/parameters/) pour la reference complete.
