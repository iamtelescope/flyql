`tokenize()` é uma função pura que transforma uma expressão FlyQL numa lista plana de tokens tipados. É a base comum sobre a qual qualquer realçador de sintaxe, pretty-printer ou formatador FlyQL pode ser construído. A função está disponível nas três linguagens com semântica byte a byte idêntica — a mesma entrada produz os mesmos tokens em JavaScript, Python e Go.

Use-a quando precisar de controlo ao nível do token sobre o render (saída ANSI em consola, HTML personalizado, decorações de editor, linters baseados em tokens). Se apenas precisar de uma string HTML, prefira o utilitário de mais alto nível [`flyql/highlight`](/pt/getting-started/javascript/#importações-de-subpath-disponíveis) (apenas JavaScript).

## Forma do token

Cada token é um registo simples com quatro campos:

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `text` | `string` | A porção da entrada original que este token cobre. |
| `type` | `string` | Um dos valores `CharType` (ver [abaixo](#tipos-de-token)). |
| `start` | `int` | Offset baseado em zero do primeiro carácter de `text`. |
| `end` | `int` | Offset baseado em zero um além do último carácter — `end == start + len(text)`. |

Os tokens satisfazem dois invariantes em que pode confiar:

- **Ida e volta:** concatenar o `text` de cada token por ordem reproduz exatamente a entrada original.
- **Offsets sem intervalos:** `tokens[0].start == 0`, `tokens[i].start == tokens[i-1].end` para todo `i > 0`, e `tokens[-1].end == len(input)` (para entradas não vazias).

Estas duas propriedades juntas significam que um formatador pode percorrer a lista de tokens, emitir decorações à volta de cada `text`, e nunca falhar um carácter nem sobrepor uma fronteira.

## API

`tokenize` aceita uma string de entrada e um modo opcional. Apenas o modo `"query"` está implementado nas três linguagens; o modo `"columns"` é exclusivo do JavaScript.

### JavaScript

```javascript
import { tokenize } from 'flyql/tokenize'

const tokens = tokenize("status=200 and region='us-east'")
// [
//   { text: 'status',    type: 'flyqlKey',      start:  0, end:  6 },
//   { text: '=',         type: 'flyqlOperator', start:  6, end:  7 },
//   { text: '200',       type: 'number',        start:  7, end: 10 },
//   { text: ' ',         type: 'space',         start: 10, end: 11 },
//   { text: 'and',       type: 'flyqlOperator', start: 11, end: 14 },
//   { text: ' ',         type: 'space',         start: 14, end: 15 },
//   { text: 'region',    type: 'flyqlKey',      start: 15, end: 21 },
//   { text: '=',         type: 'flyqlOperator', start: 21, end: 22 },
//   { text: "'us-east'", type: 'string',        start: 22, end: 31 },
// ]

// Modo columns (apenas JavaScript)
const columnTokens = tokenize('id as user_id', { mode: 'columns' })
```

### Python

```python
from flyql import tokenize

tokens = tokenize("status=200 and region='us-east'")
for t in tokens:
    print(t.text, t.type.value, t.start, t.end)

# O Token Python é uma dataclass frozen com os campos: text, type, start, end
```

O modo columns não é suportado em Python e lança `ValueError`:

```python
tokenize("id", mode="columns")
# ValueError: columns mode is only available in the JavaScript package
```

### Go

```go
import "github.com/iamtelescope/flyql/golang"

tokens, err := flyql.Tokenize("status=200 and region='us-east'", "query")
if err != nil {
    // não-nil para modos não suportados; "query" nunca gera erro
}
for _, t := range tokens {
    fmt.Println(t.Text, t.Type, t.Start, t.End)
}
```

O modo columns devolve um erro não-nil: `columns mode is only available in the JavaScript package`.

## Tipos de token

O tokenizador emite constantes `CharType` do parser principal. No modo query, os tipos interessantes são:

| Tipo | Exemplo | Quando é produzido |
|------|---------|--------------------|
| `flyqlKey` | `status`, `meta.region` | Nomes de campo à esquerda de um operador |
| `flyqlOperator` | `=`, `!=`, `>=`, `and`, `or`, `not` | Operadores de comparação e booleanos |
| `number` | `200`, `-42`, `3.14`, `1e5` | Literais numéricos (ver [regra numérica canónica](#regra-numérica-canónica)) |
| `string` | `'ok'`, `"hello"` | Literais de string entre aspas |
| `flyqlBoolean` | `true`, `false` | Literais booleanos |
| `flyqlNull` | `null` | Literal null |
| `flyqlColumn` | `alice`, `Infinity` | Valor sem aspas que não é um literal — tratado como referência de coluna |
| `flyqlPipe` | `\|` | Pipe separador de transformers |
| `flyqlTransformer` | `upper`, `chars` | Nomes de transformers |
| `flyqlArgument` | `25` (dentro de `chars(25)`) | Argumento de transformer |
| `flyqlParameter` | `$name`, `$1` | Marcadores de parâmetros |
| `flyqlWildcard` | `*` | Wildcard dentro de valores |
| `space` | ` ` | Espaço em branco entre tokens |
| `flyqlError` | `y` em `x!y` | Cauda da entrada que o parser não conseguiu consumir |

O modo columns usa o seu próprio conjunto: `column`, `alias`, `aliasOperator`, `operator`, `transformer`, `argument`, `space`, `error`.

Note que `flyqlValue` **nunca aparece** na saída — cada token de valor bruto do parser é promovido pela regra abaixo antes de `tokenize` retornar.

## A regra de upgrade de VALUE

O parser marca inicialmente cada carácter de valor como `flyqlValue`. Antes de retornar, `tokenize` promove cada token de valor para um tipo mais específico:

1. `"true"` ou `"false"` → `flyqlBoolean`
2. `"null"` → `flyqlNull`
3. corresponde à [regra numérica canónica](#regra-numérica-canónica) → `number`
4. começa por `'` ou `"` → `string`
5. caso contrário → `flyqlColumn`

Esta lógica só se aplica no modo query. O modo columns não tem conceito de valor.

## Regra numérica canónica

Um token de valor é classificado como `number` apenas quando corresponde exatamente a esta expressão regular:

```
^-?\d+(\.\d+)?([eE][+-]?\d+)?$
```

É **intencionalmente mais estreita** que `parseFloat` do JavaScript, `float()` do Python ou `strconv.ParseFloat` do Go. As seguintes entradas **não** são números e caem para `flyqlColumn`:

- `Infinity`, `-Infinity`
- `NaN`
- literais hexadecimais como `0x1F`
- números rodeados de espaços como `  42  `

As três linguagens partilham a mesma expressão regular, pelo que `val=Infinity` é tokenizado de forma idêntica em JavaScript, Python e Go.

## Token ERROR final

Se o parser parar antes de consumir toda a entrada, `tokenize` acrescenta um token final que cobre a cauda não parsada:

```javascript
tokenize('x!y')
// [
//   { text: 'x', type: 'flyqlKey',      start: 0, end: 1 },
//   { text: '!', type: 'flyqlOperator', start: 1, end: 2 },
//   { text: 'y', type: 'flyqlError',    start: 2, end: 3 },
// ]
```

Isto preserva o invariante de ida e volta mesmo para entradas malformadas. Os formatadores podem renderizar o token `flyqlError` final (ou o token `error` no modo columns) com um estilo distinto — o utilitário `highlight()` do JavaScript usa a classe CSS `flyql-error`.

Entrada vazia devolve lista vazia, nunca um token error final.

## Limitações

- **O modo columns é exclusivo do JavaScript.** Apenas o parser de colunas JavaScript regista saída tipada por carácter; os parsers de colunas em Python e Go não o fazem, por isso `tokenize()` nessas linguagens não se pode estender ao modo columns.
- **Garantias apenas para ASCII.** A paridade de offsets entre linguagens só é garantida para entradas ASCII. Entradas não-ASCII podem produzir valores `start`/`end` diferentes entre linguagens devido a diferenças byte/codepoint/unidade UTF-16.
- **Sem números de linha ou coluna.** Os tokens só expõem offsets de carácter. Se precisar de pares `(linha, coluna)` para um editor, calcule-os a partir de `start` sobre o texto original.
