:::note
Los renderers nunca se ejecutan dentro de flyql — tu aplicación es responsable
de interpretar `parsed_column.renderers` y renderizar el valor.
:::

:::note
`metadata` es de formato libre y definido por el desarrollador. flyql core y
el editor nunca leen ninguna clave específica de él. Si deseas campos
estandarizados, defínelos en tu propia aplicación y documéntalos junto a
tus renderers registrados.
:::

Consulta la [página de sintaxis de Renderers](/syntax/renderers/) para
gramática y semántica. Esta página cubre la API de registro para añadir
tus propios renderers al validador y al editor.

## La interfaz Renderer

Un `Renderer` declara:

- **`name`** — el identificador que los usuarios escriben después de la pipe del alias
- **`arg_schema`** — una tupla de entradas `ArgSpec(type, required)` que
  controlan la validación de cantidad y tipo de argumentos
- **`metadata`** — un dict/map/object opcional de formato libre para uso de la
  aplicación; flyql nunca inspecciona su contenido
- **`diagnose(args, parsed_column)`** — un hook opcional por renderer que
  devuelve `Diagnostic[]` para validación personalizada

## Ejemplo en Python

```python
from flyql.flyql_type import Type
from flyql.renderers import ArgSpec, Renderer, RendererRegistry
from flyql.columns import parse, diagnose
from flyql.core.validator import Diagnostic
from flyql.core.range import Range


class HrefRenderer(Renderer):
    arg_schema = (ArgSpec(type=Type.String, required=True),)

    @property
    def name(self) -> str:
        return "href"

    @property
    def metadata(self):
        return {"html_safe": True}

    def diagnose(self, arguments, parsed_column):
        if arguments and "{{value}}" not in arguments[0]:
            return [Diagnostic(
                range=Range(0, 1),
                message="href template should contain {{value}}",
                severity="warning",
                code="custom_href_no_placeholder",
            )]
        return []


registry = RendererRegistry()
registry.register(HrefRenderer())

cols = parse('url as link|href("/users/{{value}}")',
             capabilities={"transformers": True, "renderers": True})

# ...then run the validator with both registries:
diags = diagnose(cols, schema, renderer_registry=registry)
```

## Hook de cadena a nivel de registro

Para reglas a nivel de cadena (por ejemplo, "prohibir `href` después de
`truncate`") usa `set_diagnose`:

```python
def chain_rule(parsed_column, chain):
    names = [r["name"] for r in chain]
    if "truncate" in names and "href" in names:
        if names.index("truncate") < names.index("href"):
            return [Diagnostic(
                range=Range(0, 1),
                message="href cannot follow truncate",
                severity="error",
                code="chain_forbidden",
            )]
    return []

registry.set_diagnose(chain_rule)
```

El hook de cadena se ejecuta después de los hooks `diagnose` por renderer y
después de las verificaciones integradas de nombre desconocido / cantidad de
argumentos / tipo de argumento.

## Ejemplo en Go

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    "github.com/iamtelescope/flyql/golang/columns"
    "github.com/iamtelescope/flyql/golang/flyqltype"
    "github.com/iamtelescope/flyql/golang/transformers"
)

type HrefRenderer struct{ columns.BaseRenderer }

func (HrefRenderer) Name() string { return "href" }
func (HrefRenderer) ArgSchema() []transformers.ArgSpec {
    return []transformers.ArgSpec{{Type: flyqltype.String, Required: true}}
}

reg := columns.DefaultRendererRegistry()
_ = reg.Register(HrefRenderer{})

opts := columns.DiagnoseOptions{RendererRegistry: reg}
diags := columns.DiagnoseWithOptions(parsedColumns, schema, opts)
```

## Ejemplo en JavaScript

```javascript
import { Renderer, RendererRegistry, ArgSpec } from 'flyql/renderers'
import { Type } from 'flyql'
import { parse, diagnose } from 'flyql/columns'

class HrefRenderer extends Renderer {
    get name() { return 'href' }
    get argSchema() { return [new ArgSpec(Type.String, true)] }
    diagnose(args, col) {
        if (args && args[0] && !args[0].includes('{{value}}')) {
            return [{
                range: { start: 0, end: 1 },
                message: 'href template should contain {{value}}',
                severity: 'warning',
                code: 'custom_href_no_placeholder',
            }]
        }
        return []
    }
}

const registry = new RendererRegistry()
registry.register(new HrefRenderer())

const cols = parse('url as link|href("/users/{{value}}")',
    { transformers: true, renderers: true })

const diags = diagnose(cols, schema, null, registry)
```

## Integración con el editor flyql-vue

El componente Vue `FlyqlColumns` acepta una prop `rendererRegistry`. Cuando
se proporciona, el editor activa el parseo de renderers post-alias, emite
sugerencias de renderers y muestra diagnósticos de renderers:

```vue
<template>
  <FlyqlColumns
    v-model="query"
    :columns="schema"
    :registry="transformerRegistry"
    :renderer-registry="rendererRegistry"
  />
</template>
```

Cuando `rendererRegistry` se omite o es `null`, el parseo de renderers se
desactiva completamente — los usuarios que escriban `|` post-alias verán un
rechazo a nivel de parser en lugar de diagnósticos `unknown_renderer` en cada
pulsación de tecla. Consulta [Columns Component](/editor/columns-component/)
para la referencia completa de props.
