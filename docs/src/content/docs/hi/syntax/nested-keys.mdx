## डॉट नोटेशन

JSON, Map, या संरचित डेटा में नेस्टेड फ़ील्ड एक्सेस करने के लिए डॉट-विभाजित पथ का उपयोग करें:

<FlyqlCode code={`request.url = "/api/users"`} />
<FlyqlCode code={`response.headers.content_type = "application/json"`} />
<FlyqlCode code={`metadata.labels.env = 'prod'`} />

प्रत्येक डॉट-विभाजित खंड नेस्टेड संरचना में एक स्तर गहरा जाता है। SQL जनरेटर इन पथों को प्रत्येक डेटाबेस डायलेक्ट के लिए सही सिंटैक्स में अनुवादित करते हैं (JSON पथ एक्सेस, Map लुकअप, आदि)।

## उद्धृत कुंजी खंड

जब कोई कुंजी खंड स्वयं एक डॉट रखता हो, तो उसे उद्धरण चिह्नों में लपेटें ताकि डॉट को विभाजक न माना जाए:

<FlyqlCode code={`"foo.bar".baz = value`} />

यहाँ:
- `"foo.bar"` एक ही कुंजी खंड है (डॉट शाब्दिक है)
- <FlyqlToken type="key">baz</FlyqlToken> `"foo.bar"` का चाइल्ड है

उद्धरण चिह्नों के बिना, <FlyqlToken type="key">foo.bar.baz</FlyqlToken> तीन खंड होंगे: <FlyqlToken type="key">foo</FlyqlToken> → <FlyqlToken type="key">bar</FlyqlToken> → <FlyqlToken type="key">baz</FlyqlToken>।

### उद्धरण नियम

कुंजी खंडों के लिए सिंगल और डबल दोनों उद्धरण चिह्न काम करते हैं:

<FlyqlCode code={`"foo.bar".baz = value`} />
<FlyqlCode code={`'foo.bar'.baz = value`} />

केवल डॉट वाले खंडों को उद्धृत करना आवश्यक है:

<FlyqlCode code={`metadata."dotted.key".value = 123`} />

## पथ गहराई

नेस्टिंग गहराई पर कोई सीमा नहीं है:

<FlyqlCode code={`a.b.c.d.e = value`} />

SQL जनरेट करते या इन-मेमोरी मूल्यांकन करते समय प्रत्येक खंड क्रम से ट्रैवर्स किया जाता है।

## ऑपरेटर के साथ संयोजन

नेस्टेड कुंजी सभी ऑपरेटर के साथ काम करती हैं:

<FlyqlCode code={`request.status >= 400`} />
<FlyqlCode code={`response.body ~ "error.*"`} />
<FlyqlCode code={`metadata.tags in ['prod', 'staging']`} />
<FlyqlCode code={`request.headers.content_type != "text/html"`} />
<FlyqlCode code={`not request.authenticated`} />

## ऐरे एलिमेंट एक्सेस

ऐरे टाइप के कॉलम के लिए, एलिमेंट तक पहुँचने के लिए शून्य-आधारित संख्यात्मक पाथ सेगमेंट का उपयोग करें:

<FlyqlCode code={`tags.0 = 'important'`} />
<FlyqlCode code={`tags.1 = 'second'`} />

FlyQL में ऐरे इंडेक्सिंग **सभी SQL डायलेक्ट्स और इन-मेमोरी मैचर में 0-आधारित** है। जनरेटर flyql इंडेक्स को प्रत्येक टारगेट की मूल SQL परंपरा में अनुवादित करते हैं — ClickHouse, StarRocks और PostgreSQL के ऐरे SQL स्तर पर 1-आधारित होते हैं, इसलिए flyql का `tags.0` हर डायलेक्ट में `tags[1]` के रूप में उत्सर्जित होता है (डायलेक्ट-विशिष्ट आइडेंटिफ़ायर कोटिंग और फ़ंक्शन रूप के साथ)। इसका अर्थ है कि एक ही flyql अभिव्यक्ति प्रत्येक डायलेक्ट के लिए अलग SQL स्ट्रिंग्स उत्पन्न करती है लेकिन समान अंतर्निहित डेटा पर **समान परिणाम पर मूल्यांकित होती है**।
