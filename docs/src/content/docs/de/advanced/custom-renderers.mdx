:::note
Renderers werden niemals innerhalb von flyql ausgeführt — deine App ist
dafür verantwortlich, `parsed_column.renderers` zu interpretieren und den
Wert zu rendern.
:::

:::note
`metadata` ist frei definierbar und entwicklerdefiniert. flyql core und der
Editor lesen niemals einen bestimmten Schlüssel daraus. Wenn du standardisierte
Felder möchtest, definiere sie in deiner eigenen App und dokumentiere sie
zusammen mit deinen registrierten Renderers.
:::

Siehe die [Renderers-Syntaxseite](/syntax/renderers/) für Grammatik und
Semantik. Diese Seite behandelt die Registrierungs-API zum Hinzufügen eigener
Renderers zum Validator und Editor.

## Das Renderer-Interface

Ein `Renderer` deklariert:

- **`name`** — der Bezeichner, den Benutzer nach der Alias-Pipe eingeben
- **`arg_schema`** — ein Tupel von `ArgSpec(type, required)`-Einträgen, die
  die Validierung von Argumentanzahl und -typ steuern
- **`metadata`** — ein optionales frei definierbares Dict/Map/Object für die
  Anwendung; flyql inspiziert dessen Inhalt nie
- **`diagnose(args, parsed_column)`** — ein optionaler pro-Renderer-Hook,
  der `Diagnostic[]` für benutzerdefinierte Validierung zurückgibt

## Python-Beispiel

```python
from flyql.flyql_type import Type
from flyql.renderers import ArgSpec, Renderer, RendererRegistry
from flyql.columns import parse, diagnose
from flyql.core.validator import Diagnostic
from flyql.core.range import Range


class HrefRenderer(Renderer):
    arg_schema = (ArgSpec(type=Type.String, required=True),)

    @property
    def name(self) -> str:
        return "href"

    @property
    def metadata(self):
        return {"html_safe": True}

    def diagnose(self, arguments, parsed_column):
        if arguments and "{{value}}" not in arguments[0]:
            return [Diagnostic(
                range=Range(0, 1),
                message="href template should contain {{value}}",
                severity="warning",
                code="custom_href_no_placeholder",
            )]
        return []


registry = RendererRegistry()
registry.register(HrefRenderer())

cols = parse('url as link|href("/users/{{value}}")',
             capabilities={"transformers": True, "renderers": True})

# ...then run the validator with both registries:
diags = diagnose(cols, schema, renderer_registry=registry)
```

## Registry-Level Chain-Hook

Für Regeln auf Kettenebene (z. B. "`href` nach `truncate` verbieten")
verwende `set_diagnose`:

```python
def chain_rule(parsed_column, chain):
    names = [r["name"] for r in chain]
    if "truncate" in names and "href" in names:
        if names.index("truncate") < names.index("href"):
            return [Diagnostic(
                range=Range(0, 1),
                message="href cannot follow truncate",
                severity="error",
                code="chain_forbidden",
            )]
    return []

registry.set_diagnose(chain_rule)
```

Der Chain-Hook wird nach den pro-Renderer-`diagnose`-Hooks und nach den
eingebauten Prüfungen für unbekannten Namen / Argumentanzahl / Argumenttyp
ausgeführt.

## Go-Beispiel

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    "github.com/iamtelescope/flyql/golang/columns"
    "github.com/iamtelescope/flyql/golang/flyqltype"
    "github.com/iamtelescope/flyql/golang/transformers"
)

type HrefRenderer struct{ columns.BaseRenderer }

func (HrefRenderer) Name() string { return "href" }
func (HrefRenderer) ArgSchema() []transformers.ArgSpec {
    return []transformers.ArgSpec{{Type: flyqltype.String, Required: true}}
}

reg := columns.DefaultRendererRegistry()
_ = reg.Register(HrefRenderer{})

opts := columns.DiagnoseOptions{RendererRegistry: reg}
diags := columns.DiagnoseWithOptions(parsedColumns, schema, opts)
```

## JavaScript-Beispiel

```javascript
import { Renderer, RendererRegistry, ArgSpec } from 'flyql/renderers'
import { Type } from 'flyql'
import { parse, diagnose } from 'flyql/columns'

class HrefRenderer extends Renderer {
    get name() { return 'href' }
    get argSchema() { return [new ArgSpec(Type.String, true)] }
    diagnose(args, col) {
        if (args && args[0] && !args[0].includes('{{value}}')) {
            return [{
                range: { start: 0, end: 1 },
                message: 'href template should contain {{value}}',
                severity: 'warning',
                code: 'custom_href_no_placeholder',
            }]
        }
        return []
    }
}

const registry = new RendererRegistry()
registry.register(new HrefRenderer())

const cols = parse('url as link|href("/users/{{value}}")',
    { transformers: true, renderers: true })

const diags = diagnose(cols, schema, null, registry)
```

## Einbindung in den flyql-vue-Editor

Die Vue-Komponente `FlyqlColumns` akzeptiert eine `rendererRegistry`-Prop.
Wenn diese bereitgestellt wird, aktiviert der Editor das Post-Alias-Renderer-Parsing,
gibt Renderer-Vorschläge aus und zeigt Renderer-Diagnosen an:

```vue
<template>
  <FlyqlColumns
    v-model="query"
    :columns="schema"
    :registry="transformerRegistry"
    :renderer-registry="rendererRegistry"
  />
</template>
```

Wenn `rendererRegistry` weggelassen wird oder `null` ist, wird das
Renderer-Parsing vollständig deaktiviert — Benutzer, die Post-Alias-`|`
eingeben, sehen eine Parser-Level-Ablehnung statt `unknown_renderer`-Diagnosen
bei jedem Tastendruck. Siehe [Columns Component](/editor/columns-component/)
für die vollständige Prop-Referenz.
