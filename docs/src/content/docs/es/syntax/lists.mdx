## Pertenencia a listas

Usa <FlyqlToken type="operator">in</FlyqlToken> para comprobar si un valor está en una lista y <FlyqlToken type="operator">not in</FlyqlToken> para comprobar que no lo está:

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['prod', 'staging']`} />

## Reglas de sintaxis

- Los valores van entre corchetes <FlyqlToken type="operator">[ ]</FlyqlToken> y se separan con comas
- Todos los valores de una lista deben ser del mismo tipo: todos strings o todos números
- Los strings deben ir entre comillas: <FlyqlCode code={`['a', 'b', 'c']`} inline />
- Los valores numéricos no llevan comillas: <FlyqlCode code={`[1, 2, 3]`} inline />
- Se permiten espacios alrededor de comas y corchetes

### Listas de strings

<FlyqlCode code={`method in ['GET', 'POST', 'PUT']`} />
<FlyqlCode code={`env not in ['dev', 'test', 'local']`} />

Se pueden usar comillas simples o dobles para los strings.

### Listas numéricas

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`port not in [80, 443]`} />

### Listas vacías

Se permiten listas vacías:

<FlyqlCode code={`status in []`} />

Una lista <FlyqlToken type="operator">in</FlyqlToken> vacía es siempre false — nada coincide con una lista vacía.

<FlyqlCode code={`status not in []`} />

Una lista <FlyqlToken type="operator">not in</FlyqlToken> vacía es siempre true — todo está "not in" en una lista vacía.

## Consistencia de tipos

Las listas **deben** contener valores de un tipo uniforme: todos strings o todos números. Mezclar tipos dentro de una misma lista produce un error de parseo:

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env in ['prod', 'staging']`} />

Lo siguiente es inválido (tipos mezclados — error de parseo):

<FlyqlCode code={`values in [200, 'ok']`} />

El parser rastrea el tipo del primer valor de la lista y rechaza cualquier valor posterior de un tipo distinto. Esto se valida en tiempo de parseo, antes de que la consulta llegue a cualquier generador o matcher.

Cuando un generador SQL procesa una lista, además valida que los tipos de los valores sean compatibles con el tipo de la columna de destino. Por ejemplo, pasar strings a una columna numérica lanzará un error de validación durante la generación de SQL.

El campo `valuesType` de la expresión parseada indica si la lista contiene valores `"string"` o `"number"`. En listas vacías, `valuesType` es `null`.

## Combinación con otros operadores

Los operadores de lista se combinan con los operadores booleanos como cualquier otra condición:

<FlyqlCode code={`status in [200, 201] and method in ['GET', 'POST']`} />
<FlyqlCode code={`level = error or status not in [200, 201, 204]`} />
<FlyqlCode code={`(status in [500, 502, 503] or level = critical) and env=prod`} />

## Ejemplos

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['dev', 'test', 'staging']`} />
<FlyqlCode code={`method in ['GET', 'POST'] and status not in [401, 403, 404]`} />
