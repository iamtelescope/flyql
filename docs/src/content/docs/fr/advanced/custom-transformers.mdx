Le systeme de transformateurs de FlyQL est extensible. Vous pouvez enregistrer des transformateurs personnalises qui fonctionnent de maniere identique aux transformateurs integres (`upper`, `lower`, `len`) a travers les parseurs, les generateurs SQL, le matcher en memoire et l'autocompletion de l'editeur.

## Interface des transformateurs

Chaque transformateur implemente 6 membres :

| Membre | Type | Description |
|--------|------|-------------|
| `name` | string | Identifiant unique en minuscules (par ex., `"firstoctet"`) |
| `inputType` | Type | Type d'entree attendu (`string`, `int`, `float`, `bool`, `array`), ou `any` pour une entree joker qui accepte n'importe quel type amont — voir [Transformateurs avec entree `Any`](#transformateurs-avec-entree-any) |
| `outputType` | Type | Type de sortie apres transformation |
| `argSchema` | ArgSpec[] | Types d'arguments declares et arite (par defaut : vide = pas d'arguments) |
| `sql(dialect, columnRef, args)` | string | Genere du SQL specifique au dialecte enveloppant la reference de colonne |
| `apply(value, args)` | any | Transforme une valeur a l'execution en memoire |

La methode `sql()` recoit le nom du dialecte SQL (`"clickhouse"`, `"postgresql"`, `"starrocks"`), une chaine de reference de colonne et la liste d'arguments parses. Elle retourne une expression SQL enveloppant cette reference. La methode `apply()` recoit une valeur a l'execution et la liste d'arguments, et retourne la valeur transformee.

### ArgSpec

Chaque entree dans `argSchema` declare un argument attendu :

| Champ | Type | Par defaut | Description |
|-------|------|------------|-------------|
| `type` | Type | — | Type d'argument attendu (`string`, `int`, `float`, `bool`) |
| `required` | boolean | `true` | Si l'argument doit etre fourni |

L'arite est derivee de la longueur du schema : les arguments requis doivent etre presents, les arguments optionnels en fin de liste peuvent etre omis. Les transformateurs sans arguments (comme `upper`, `lower`, `len`) heritent du schema vide par defaut.

## API du registre

| Methode | Description |
|---------|-------------|
| `defaultRegistry()` | Retourne un nouveau registre pre-charge avec les transformateurs integres |
| `registry.register(transformer)` | Ajoute un transformateur personnalise (leve une erreur si le nom est deja enregistre) |
| `registry.get(name)` | Recherche un transformateur par nom (retourne null/None/nil si non trouve) |
| `registry.names()` | Liste tous les noms de transformateurs enregistres |

Le principe : obtenir un registre par defaut, enregistrer vos transformateurs personnalises, passer le registre aux generateurs et au matcher.

## Python

Definissez un transformateur personnalise en heritant de `Transformer` :

```python
from flyql.transformers import Transformer, Type, default_registry

class FirstOctetTransformer(Transformer):
    @property
    def name(self): return "firstoctet"
    @property
    def input_type(self): return Type.STRING
    @property
    def output_type(self): return Type.INT
    def sql(self, dialect, column_ref, args=None):
        if dialect == "clickhouse":
            return f"toUInt8(splitByChar('.', {column_ref})[1])"
        return f"CAST(SPLIT_PART({column_ref}, '.', 1) AS INTEGER)"
    def apply(self, value, args=None):
        return int(str(value).split('.')[0])
```

L'attribut de classe `arg_schema` a pour valeur par defaut `()` (pas d'arguments). Pour declarer des arguments, surchargez-le avec un tuple de `ArgSpec` :

```python
from typing import ClassVar, Tuple
from flyql.transformers.base import ArgSpec

class SubstringTransformer(Transformer):
    arg_schema: ClassVar[Tuple[ArgSpec, ...]] = (
        ArgSpec(type=Type.INT),                    # start (required)
        ArgSpec(type=Type.INT, required=False),    # length (optional)
    )
    # ... name, input_type, output_type, sql, apply ...
```

Enregistrement et utilisation :

```python
from flyql import parse
from flyql.generators.postgresql.generator import to_sql_where, Column

registry = default_registry()
registry.register(FirstOctetTransformer())

result = parse("src_ip|firstoctet > 192")
columns = {"src_ip": Column("src_ip", "text")}
sql = to_sql_where(result.root, columns, registry=registry)
# CAST(SPLIT_PART(src_ip, '.', 1) AS INTEGER) > 192
```

Pour le matching en memoire :

```python
from flyql.matcher.evaluator import Evaluator
from flyql.matcher.record import Record

evaluator = Evaluator(registry=registry)
record = Record({"src_ip": "193.0.0.1"})
evaluator.evaluate(result.root, record)  # True (193 > 192)
```

## JavaScript

Definissez un transformateur personnalise en etendant `Transformer` :

```javascript
import { Transformer, Type, ArgSpec, defaultRegistry } from 'flyql/transformers'

class FirstOctetTransformer extends Transformer {
    get name() { return 'firstoctet' }
    get inputType() { return Type.STRING }
    get outputType() { return Type.INT }
    sql(dialect, columnRef, args = []) {
        if (dialect === 'clickhouse') return `toUInt8(splitByChar('.', ${columnRef})[1])`
        return `CAST(SPLIT_PART(${columnRef}, '.', 1) AS INTEGER)`
    }
    apply(value, args = []) { return parseInt(String(value).split('.')[0], 10) }
}
```

Le getter `argSchema` a pour valeur par defaut `[]` (pas d'arguments). Pour declarer des arguments, surchargez-le :

```javascript
class SubstringTransformer extends Transformer {
    get argSchema() {
        return [
            new ArgSpec(Type.INT, true),     // start (required)
            new ArgSpec(Type.INT, false),    // length (optional)
        ]
    }
    // ... name, inputType, outputType, sql, apply ...
}
```

Enregistrement et utilisation :

```javascript
import { parse } from 'flyql'
import { generateWhere, newColumn } from 'flyql/generators/postgresql'

const registry = defaultRegistry()
registry.register(new FirstOctetTransformer())

const result = parse("src_ip|firstoctet > 192")
const columns = { src_ip: newColumn({ name: 'src_ip', type: 'text' }) }
const sql = generateWhere(result.root, columns, registry)
// CAST(SPLIT_PART(src_ip, '.', 1) AS INTEGER) > 192
```

Pour le matching en memoire :

```javascript
import { match } from 'flyql/matcher'

match("src_ip|firstoctet > 192", { src_ip: "193.0.0.1" }, registry) // true
```

Pour l'autocompletion de l'editeur — les transformateurs personnalises apparaissent aux cotes des transformateurs integres :

```javascript
import { EditorEngine } from 'flyql-vue'

const engine = new EditorEngine(columns, { registry })
// After typing "src_ip|", suggestions include "firstoctet", "upper", "lower", "len"
```

## Go

Definissez un transformateur personnalise en implementant l'interface `Transformer` :

```go
package main

import (
    "fmt"
    "strconv"
    "strings"
    "github.com/iamtelescope/flyql/golang/transformers"
)

type FirstOctet struct{}

func (f FirstOctet) Name() string               { return "firstoctet" }
func (f FirstOctet) InputType() transformers.Type  { return transformers.TransformerTypeString }
func (f FirstOctet) OutputType() transformers.Type { return transformers.TransformerTypeInt }
func (f FirstOctet) ArgSchema() []transformers.ArgSpec        { return []transformers.ArgSpec{} }
func (f FirstOctet) SQL(dialect, colRef string, args []any) string {
    if dialect == "clickhouse" {
        return fmt.Sprintf("toUInt8(splitByChar('.', %s)[1])", colRef)
    }
    return fmt.Sprintf("CAST(SPLIT_PART(%s, '.', 1) AS INTEGER)", colRef)
}
func (f FirstOctet) Apply(value interface{}, args []any) interface{} {
    parts := strings.SplitN(fmt.Sprint(value), ".", 2)
    n, _ := strconv.Atoi(parts[0])
    return n
}
```

Pour declarer des arguments, retournez des entrees `ArgSpec` depuis `ArgSchema()` :

```go
func (s Substring) ArgSchema() []transformers.ArgSpec {
    return []transformers.ArgSpec{
        {Type: transformers.TransformerTypeInt, Required: true},   // start (required)
        {Type: transformers.TransformerTypeInt, Required: false},  // length (optional)
    }
}
```

Enregistrement et utilisation :

```go
import (
    flyql "github.com/iamtelescope/flyql/golang"
    postgresqlgen "github.com/iamtelescope/flyql/golang/generators/postgresql"
)

registry := transformers.DefaultRegistry()
registry.Register(FirstOctet{})

result, _ := flyql.Parse("src_ip|firstoctet > 192")
columns := map[string]*postgresqlgen.Column{
    "src_ip": postgresqlgen.NewColumn("src_ip", "text", nil),
}
sql, _ := postgresqlgen.ToSQLWhere(result.Root, columns, registry)
// CAST(SPLIT_PART(src_ip, '.', 1) AS INTEGER) > 192
```

Pour le matching en memoire :

```go
import "github.com/iamtelescope/flyql/golang/matcher"

matched, _ := matcher.Match("src_ip|firstoctet > 192", map[string]any{"src_ip": "193.0.0.1"}, registry)
// matched == true
```

## Transformateurs avec entree `Any`

Un transformateur peut declarer `inputType = Type.Any` (valeur de chaine `"any"`) pour accepter n'importe quel type amont. Cas d'usage motivant : un transformateur `type` qui retourne le nom du type de la colonne.

Regles :

- `Any` n'est valide que sur `inputType`. Declarer `Any` comme `outputType` ou sur n'importe quel `ArgSpec.type` est rejete au moment de `register()`. Chaque transformateur doit declarer un type de sortie concret.
- Lorsqu'un transformateur a entree `Any` est utilise dans une chaine, la verification d'egalite du type d'entree est ignoree. La rigueur en aval est preservee — par exemple `col|accepts_any_returning_array|upper` echoue toujours sur `upper` car `array` n'est pas une entree valide pour `upper`.
- `parseFlyQLType("any")` leve une erreur : `any` n'est pas un type de colonne ou de valeur valide.

```python
from flyql.flyql_type import Type
from flyql.transformers import Transformer

class TypeTransformer(Transformer):
    @property
    def name(self): return "type"
    @property
    def input_type(self): return Type.Any      # accepte n'importe quel type amont
    @property
    def output_type(self): return Type.String  # doit etre concret
    def sql(self, dialect, column_ref, args=None):
        if dialect == "clickhouse":
            return f"toTypeName({column_ref})"
        return f"pg_typeof({column_ref})::text"
    def apply(self, value, args=None):
        return type(value).__name__
```

    return fmt.Sprintf("pg_typeof(%s)::text", columnRef)

    if value == nil {

    return reflect.TypeOf(value).String()

## Compatibilite des types

Les transformateurs personnalises participent a la meme validation de chaine de types que les transformateurs integres. Si vous chainez `field|firstoctet|upper`, le systeme detecte que `firstoctet` produit un `int` mais `upper` requiert un `string`.

Types disponibles : voir [`flyql.Type`](/advanced/column-types/). La seule exception est `Any`, qui est un joker d'entree pour transformateur et n'est pas un type de colonne ou de valeur valide.

## Validation avec `diagnose()`

La fonction `diagnose()` parcourt un AST analysé en le confrontant à un schéma de colonnes et un registre de transformateurs, et retourne des enregistrements `Diagnostic` positionnés sans lever d'exception. Elle vérifie :

- **Colonne inconnue** — clé de base absente de la liste des colonnes
- **Transformateur inconnu** — nom du transformateur absent du registre
- **Arité incorrecte** — trop peu ou trop d'arguments par rapport à `argSchema`
- **Type d'argument incorrect** — le type de la valeur de l'argument ne correspond pas au type déclaré dans `ArgSpec` (l'élargissement int vers float est autorisé)
- **Conflit de type de chaîne** — le type de sortie d'un transformateur ne correspond pas au type d'entrée du suivant, ou le type de la colonne ne correspond pas au type d'entrée du premier transformateur

Chaque diagnostic contient un `range` (position source), `message`, `severity` (`"error"`) et `code` (`"unknown_column"`, `"unknown_transformer"`, `"arg_count"`, `"arg_type"`, `"chain_type"`).

```python
# Python
from flyql import parse, Column, ColumnSchema, Type, diagnose

ast = parse("hstt|upper=X").root
columns = ColumnSchema.from_columns([Column(name="host", column_type=Type.String)])
diags = diagnose(ast, columns)  # columns is a ColumnSchema
# [Diagnostic(range=Range(0, 4), message="column 'hstt' is not defined", severity='error', code='unknown_column')]
```

```javascript
// JavaScript
import { parse, Column, ColumnSchema, Type, diagnose } from 'flyql'

const ast = parse('hstt|upper=X').root
const columns = ColumnSchema.fromColumns([new Column('host', false, Type.String)])
const diags = diagnose(ast, columns)  // columns is a ColumnSchema
// [Diagnostic { range: Range(0, 4), message: "column 'hstt' is not defined", severity: 'error', code: 'unknown_column' }]
```

```go
// Go
import flyql "github.com/iamtelescope/flyql/golang"

result, _ := flyql.Parse("hstt|upper=X")
schema := flyql.FromColumns([]flyql.Column{flyql.NewColumn("host", flyql.TypeString)})
diags := flyql.Diagnose(result.Root, schema, nil) // nil = default registry
// []Diagnostic{{Range: {0, 4}, Message: "column 'hstt' is not defined", Severity: "error", Code: "unknown_column"}}
```

Passez un registre personnalisé comme troisième argument si vous avez des transformateurs personnalisés enregistrés. Passez `nil`/`None`/`null` pour utiliser le registre par défaut.

## Sécurité

Les méthodes `sql()` des transformateurs personnalisés ne reçoivent que des références de colonnes (depuis le schéma), jamais de valeurs fournies par l'utilisateur. Les valeurs utilisateur sont échappées/paramétrées par le générateur. Cela préserve la garantie anti-injection de FlyQL — le SQL personnalisé enveloppe les noms de colonnes, pas les entrées utilisateur.
