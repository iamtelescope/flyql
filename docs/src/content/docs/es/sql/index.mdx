Los generadores de FlyQL producen un fragmento de cláusula WHERE como una cadena SQL plana con todos los valores embebidos y escapados. La misma consulta FlyQL genera SQL distinto según el dialecto de destino.

## Dialectos compatibles

- [ClickHouse](/es/sql/clickhouse/): sin comillas en nombres de columnas, `match()` para regex y funciones `JSONExtract` para strings JSON
- [PostgreSQL](/es/sql/postgresql/): identificadores entre comillas dobles, `~` para regex y operadores JSONB
- [StarRocks](/es/sql/starrocks/): identificadores entre backticks, `regexp()` para regex y `parse_json()` para strings JSON

## Diferencias principales

| Característica | ClickHouse | PostgreSQL | StarRocks |
|----------------|-----------|------------|-----------|
| Comillas en columnas | Ninguna | `"name"` | `` `name` `` |
| Regex | `match(col, 'pat')` | `"col" ~ 'pat'` | `` regexp(`col`, 'pat') `` |
| Regex negada | `NOT match(col, 'pat')` | `"col" !~ 'pat'` | `` NOT regexp(`col`, 'pat') `` |
| Comodín | `col LIKE 'val%'` | `"col" LIKE 'val%'` | `` `col` LIKE 'val%' `` |
| Operadores booleanos | `AND` / `OR` | `AND` / `OR` | `AND` / `OR` |
| `IN []` vacío | `0` | `FALSE` | `0` |
| `NOT IN []` vacío | `1` | `TRUE` | `1` |
| <FlyqlToken type="operator">has</FlyqlToken> sobre string | `position(col, 'v') > 0` | `position('v' in "col") > 0` | `` INSTR(`col`, 'v') > 0 `` |
| <FlyqlToken type="operator">has</FlyqlToken> sobre array | `has(col, 'v')` | `'v' = ANY("col")` | `` array_contains(`col`, 'v') `` |
| <FlyqlToken type="operator">has</FlyqlToken> sobre map | `mapContains(col, 'k')` | `"col" ? 'k'` | `` array_contains(map_keys(`col`), 'k') `` |
| <FlyqlToken type="operator">has</FlyqlToken> sobre JSON | `JSONHas(toJSONString(col), 'k')` | `"col" ? 'k'` | `json_exists(...)` |

## Acceso a JSON: la mayor diferencia

Cada dialecto maneja las columnas JSON anidadas de forma muy distinta.

**Entrada FlyQL:** <FlyqlCode code={`data.user.name = 'john'`} inline />

**ClickHouse** con columna JSON nativa:
```sql
data.`user`.`name` = 'john'
```

**ClickHouse** con JSON almacenado como string:
```sql
multiIf(JSONType(data, 'user', 'name') = 'String',
  equals(JSONExtractString(data, 'user', 'name'), 'john'), 0)
```

**PostgreSQL** con JSONB:
```sql
(jsonb_typeof("data"->'user'->'name') = 'string'
  AND "data"->'user'->>'name' = 'john')
```

**StarRocks** con JSON nativo:
```sql
`data`->'\"user\"'->'\"name\"' = 'john'
```

Consulta la página de cada dialecto para ver ejemplos completos.

Los índices de matriz de FlyQL son uniformemente base 0. Los tres dialectos usan matrices base 1 a nivel de SQL, por lo que los generadores suman 1 al índice de flyql antes de emitir. La misma expresión FlyQL produce distintas cadenas SQL por dialecto (debido al entrecomillado de identificadores y a la sintaxis de las funciones), pero se evalúa al mismo resultado sobre los mismos datos subyacentes.

## Formato de salida

Los generadores devuelven un fragmento de cláusula WHERE, sin la palabra clave `WHERE` y sin placeholders parametrizados. Todos los valores se escapan y se insertan directamente:

```
message = 'hello' and count > 10
```

No así:

```
WHERE message = ? AND count > ?
```
