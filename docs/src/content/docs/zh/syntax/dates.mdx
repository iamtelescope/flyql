FlyQL 在列模式中区分**日历日**（`Type.Date`）与**时间瞬时**（`Type.DateTime`），并据此驱动 matcher 对 ISO-8601 字符串字面量的强制转换。而 SQL 生成器则原样传递字符串字面量 — 由数据库负责解析。

## 日历日 vs. 瞬时

`Type.Date` — 仅日历日。比较会被截断到 `YYYYMMDD`。吸收原生 DB 类型 `date` / `date32`。

`Type.DateTime` — 时间轴上的一点，毫秒分辨率。比较在 UTC 毫秒下进行。吸收 `datetime`、`datetime64`、`timestamp`、`timestamptz`、`year`。

声明语义类型对 matcher 很重要：`Date` 列与 `'2026-04-06T15:30:00Z'` 比较时，会在比较前把记录值截断到 Y/M/D 分量;而 `DateTime` 列则比较到毫秒级别。

如果一个声明为 `Type.Date` 的列收到 datetime 形状的值（原生 `datetime` / `time.Time` / `Date` 实例,或带时间分量的 ISO 字符串），FlyQL 会针对该列一次性发出 `UserWarning`（Python）/ 日志行（Go）/ `console.warn`（JS），建议迁移到 `Type.DateTime`。

## 字符串字面量形式的日期

把 date/datetime 列与 ISO-8601 格式的字符串字面量做比较。生成器原样传递字面量；数据库在执行时解析它。

<FlyqlCode code={`due_date < '2023-12-31'`} />
<FlyqlCode code={`created_at >= '2023-12-31T00:00:00Z'`} />
<FlyqlCode code={`event_time >= '2023-01-01' and event_time <= '2023-12-31'`} />

在 matcher 中,由模式声明的 `Date` / `DateTime` 列同时驱动 ISO-8601 字面量和记录值的自动解析。对于常见场景,无需显式的转换函数。

### 接受的 iso8601 形式

matcher 的 `iso8601` 家族很宽松：

- `YYYY-MM-DD` — 纯日历日期（用于 Date 列，或对于 DateTime 列表示列-tz 中的午夜）。
- `YYYY-MM-DDTHH:MM:SS` — naive datetime。
- `YYYY-MM-DD HH:MM:SS` — 使用空格分隔符的 naive datetime（解析前归一化）。
- `YYYY-MM-DDTHH:MM:SS.ffffff` — 可选亚秒（仅 ms 分辨率 — 见下文"精度"）。
- 以上任意形式附带 `Z` 或 `±HH:MM` 偏移后缀。

### naive 字面量的列元数据

`DateTime` 列声明可以携带可选的元数据：

- `tz` — 用于解释 naive ISO 字符串的 IANA 时区名称。回退到 Evaluator 的 `default_timezone`，再回退到 `"UTC"`。
- `unit` — `"ms"`（默认）、`"s"` 或 `"ns"`。应用于数值记录值；非数值会忽略它。

```json
{
  "ts":        { "type": "datetime", "tz": "Europe/Moscow", "unit": "s" },
  "event_day": { "type": "date" }
}
```

:::note[字符串必须加引号]
日期字面量本质上是字符串，**必须**加引号。如果不加引号，FlyQL 会把它当作列引用。<FlyqlCode code={`due_date < 2023-12-31`} inline /> 会被解析为对名为 `2023` 的列做算术运算，而不是日期字面量。参见[值与类型 — String](/zh/syntax/values/#string)。
:::

## 时间函数

对于相对时间查询 — "过去一小时"、"本周开始至今"、"自今天起" — FlyQL 提供内置的时间函数：<FlyqlCode code={`ago(...)`} inline />、<FlyqlCode code={`now()`} inline />、<FlyqlCode code={`today()`} inline /> 与 <FlyqlCode code={`startOf(...)`} inline />。

<FlyqlCode code={`created_at > ago(1h)`} />
<FlyqlCode code={`updated_at >= ago(7d)`} />
<FlyqlCode code={`expires_at < now()`} />
<FlyqlCode code={`event_time > startOf('day')`} />
<FlyqlCode code={`scheduled_for >= today('Europe/Berlin')`} />

完整的语法（单位、顺序规则、时区参数）请参见[值与类型 — 时间函数](/zh/syntax/values/#temporal-functions)。

## 夏令时（DST）

当一个 naive datetime 字面量（无偏移、无 `Z`）在遵守 DST 的 tz 中解释时，FlyQL 选择规范定义的明确解释：

- **回退（钟面歧义）** — 选择较早的出现（Python 语义中的 `fold=0`）。示例：`America/New_York` 中的 `'2026-11-01 01:30:00'` 解析为 05:30 UTC（EDT），而不是 06:30 UTC（EST）。
- **跃进（不存在的钟面）** — 拒绝该值。matcher 对该记录返回 `false`（跳过）。示例：`America/New_York` 中的 `'2026-03-08 02:30:00'` 并不存在。

落入跃进间隙的记录被视为不可解析，而不是静默向前归一化。

## 支持的记录值类型（Python）

- `datetime.datetime` — 带时区或 naive（naive 使用列 `tz` 或 Evaluator 默认值）。
- `datetime.date` — 仅日历日。
- `pandas.Timestamp` — 由于它是 `datetime` 的子类,所以可用。
- ISO-8601 字符串 — 见上面接受的形式。
- `int` / `float` — 默认 epoch 毫秒；设置列 `unit="s"` 表示 epoch 秒，`"ns"` 表示纳秒。

当前不支持：

- `numpy.datetime64` — 在传给 matcher 前转换：`pd.Timestamp(value).to_pydatetime()` 或 `value.astype("datetime64[ms]").astype(int)`。

## 精度

FlyQL 的 matcher 以**毫秒粒度**比较时间值。源数据中的亚毫秒精度会在比较前被截断：

- Python — `int(dt.timestamp() * 1000)` 丢弃微秒。
- Go — `UnixMilli()` 丢弃纳秒。
- JavaScript — `Date` 原生就是 ms。

需要纳秒精度的用户必须把记录预处理为 bigint 纳秒并声明 `unit="ns"`。JS 路径使用 `Number`，在 `2^53` 以上会损失精度 — 纳秒 BigInt 路径保留给后续规范。

## 无模式数值

没有列模式的数值记录值按 **epoch 毫秒**处理。没有自动检测启发式 — 记录 `{"ts": 1712434800}`（按惯例为 epoch 秒）会被解释为 `1970-01-20` 而不是 `2024`。使用 epoch 秒数据的用户必须声明 `unit="s"`：

```json
{ "ts": { "type": "datetime", "unit": "s" } }
```

这是设计上的严格：自动检测启发式（例如 `value < 10^10 → 秒`）在边界和合法的旧日期值上失败，因此我们偏好显式声明而非静默猜测。

## 何时选择哪种方式

| 意图 | 使用 | 示例 |
|------|------|------|
| 与固定的日历日期或时间戳比较 | 字符串字面量 | <FlyqlCode code={`due_date < '2023-12-31'`} inline /> |
| 与滑动时间窗口比较 | <FlyqlCode code={`ago()`} inline /> | <FlyqlCode code={`created_at > ago(1h)`} inline /> |
| 与当前时刻比较 | <FlyqlCode code={`now()`} inline /> | <FlyqlCode code={`expires_at < now()`} inline /> |
| 与今天比较（带时区） | <FlyqlCode code={`today()`} inline /> | <FlyqlCode code={`scheduled_for = today('UTC')`} inline /> |
| 与某天/某周/某月的开始比较 | <FlyqlCode code={`startOf()`} inline /> | <FlyqlCode code={`event_time > startOf('week')`} inline /> |

字符串字面量保持不变 — 同一条查询语句日复一日返回同样的 SQL。时间函数在执行时计算，因此同一条 FlyQL 查询每次运行都会得到不同的时间窗口。
