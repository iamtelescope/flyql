FlyQL은 컬럼 스키마에서 **달력 날짜**(`Type.Date`)와 **시점**(`Type.DateTime`)을 구분하고, 이 구분을 사용해 ISO-8601 문자열 리터럴에 대한 매처 코어션을 제어합니다. 반면 SQL 생성기는 문자열 리터럴을 그대로 전달합니다 — 파싱은 데이터베이스가 수행합니다.

## 달력 날짜 vs. 시점

`Type.Date` — 달력 날짜만. 비교는 `YYYYMMDD`로 잘립니다. 원시 DB 타입 `date` / `date32`를 흡수합니다.

`Type.DateTime` — 타임라인상의 지점, 밀리초 해상도. 비교는 UTC 밀리초로 수행됩니다. `datetime`, `datetime64`, `timestamp`, `timestamptz`, `year`를 흡수합니다.

시맨틱 타입 선언은 매처에게 중요합니다: `'2026-04-06T15:30:00Z'`와 비교되는 `Date` 컬럼은 비교 전에 레코드 값을 Y/M/D 구성 요소로 자르는 반면, `DateTime` 컬럼은 밀리초까지 비교합니다.

`Type.Date`로 선언된 컬럼이 datetime 형태의 값(네이티브 `datetime` / `time.Time` / `Date` 인스턴스, 또는 시간 구성 요소가 있는 ISO 문자열)을 받으면, FlyQL은 컬럼당 한 번 `UserWarning`(Python) / 로그 라인(Go) / `console.warn`(JS)을 내보내고 `Type.DateTime`으로의 마이그레이션을 제안합니다.

## 문자열 리터럴 형태의 날짜

date/datetime 컬럼을 ISO-8601 형식의 문자열 리터럴과 비교합니다. 생성기는 리터럴을 그대로 전달하고, 데이터베이스가 실행 시점에 파싱합니다.

<FlyqlCode code={`due_date < '2023-12-31'`} />
<FlyqlCode code={`created_at >= '2023-12-31T00:00:00Z'`} />
<FlyqlCode code={`event_time >= '2023-01-01' and event_time <= '2023-12-31'`} />

매처에서는 스키마로 선언된 `Date` / `DateTime` 컬럼이 ISO-8601 리터럴 및 레코드 값 모두의 자동 파싱을 구동합니다. 일반적인 경우에는 명시적인 변환 함수가 필요하지 않습니다.

### 허용되는 iso8601 형식

매처의 `iso8601` 계열은 관대합니다:

- `YYYY-MM-DD` — 순수 달력 날짜(Date 컬럼 또는 DateTime 컬럼의 경우 컬럼-tz의 자정).
- `YYYY-MM-DDTHH:MM:SS` — 네이브 datetime.
- `YYYY-MM-DD HH:MM:SS` — 공백 구분자가 있는 네이브 datetime(파싱 전에 정규화됨).
- `YYYY-MM-DDTHH:MM:SS.ffffff` — 선택적 서브초(ms 해상도만 — 아래 "정밀도" 참조).
- 위 중 어느 것이든 `Z` 또는 `±HH:MM` 오프셋 접미사와 함께 사용 가능.

### 네이브 리터럴을 위한 컬럼 메타데이터

`DateTime` 컬럼 선언은 선택적 메타데이터를 가질 수 있습니다:

- `tz` — 네이브 ISO 문자열을 해석하는 데 사용되는 IANA 시간대 이름. Evaluator의 `default_timezone`으로, 다음에는 `"UTC"`로 폴백합니다.
- `unit` — `"ms"`(기본값), `"s"`, 또는 `"ns"`. 숫자 레코드 값에 적용되며 비숫자 값은 무시합니다.

```json
{
  "ts":        { "type": "datetime", "tz": "Europe/Moscow", "unit": "s" },
  "event_day": { "type": "date" }
}
```

:::note[문자열은 따옴표가 필수]
날짜 리터럴은 문자열이며 **반드시** 따옴표로 감싸야 합니다. 따옴표가 없으면 FlyQL은 값을 컬럼 참조로 해석합니다. <FlyqlCode code={`due_date < 2023-12-31`} inline />는 `2023`이라는 이름의 컬럼에 대한 산술로 파싱되며, 날짜 리터럴이 되지 않습니다. [값과 타입 — String](/ko/syntax/values/#string)을 참조하세요.
:::

## 시간 함수

상대 시간 쿼리 — "지난 한 시간 내", "이번 주 시작부터", "오늘부터" — 의 경우 FlyQL은 내장 시간 함수를 제공합니다: <FlyqlCode code={`ago(...)`} inline />, <FlyqlCode code={`now()`} inline />, <FlyqlCode code={`today()`} inline />, <FlyqlCode code={`startOf(...)`} inline />.

<FlyqlCode code={`created_at > ago(1h)`} />
<FlyqlCode code={`updated_at >= ago(7d)`} />
<FlyqlCode code={`expires_at < now()`} />
<FlyqlCode code={`event_time > startOf('day')`} />
<FlyqlCode code={`scheduled_for >= today('Europe/Berlin')`} />

전체 문법(단위, 순서 규칙, 시간대 인수)은 [값과 타입 — 시간 함수](/ko/syntax/values/#temporal-functions)에서 다룹니다.

## 일광 절약 시간(DST)

네이브 datetime 리터럴(오프셋 없음, `Z` 없음)이 DST를 따르는 tz에서 해석될 때, FlyQL은 스펙에서 정의한 명확한 해결을 선택합니다:

- **폴백(모호한 벽시계)** — 더 이른 발생이 선택됩니다(Python 시맨틱의 `fold=0`). 예: `America/New_York`의 `'2026-11-01 01:30:00'`은 06:30 UTC(EST)가 아니라 05:30 UTC(EDT)로 해석됩니다.
- **스프링포워드(존재하지 않는 벽시계)** — 값이 거부됩니다. 매처는 해당 레코드에 대해 `false`를 반환합니다(건너뜀). 예: `America/New_York`의 `'2026-03-08 02:30:00'`은 존재하지 않습니다.

스프링포워드 갭에 해당하는 레코드는 조용히 앞으로 정규화되는 대신 파싱 불가능한 것으로 취급됩니다.

## 지원되는 레코드 값 타입(Python)

- `datetime.datetime` — tz-aware 또는 네이브(네이브는 컬럼 `tz` 또는 Evaluator 기본값 사용).
- `datetime.date` — 달력 날짜만.
- `pandas.Timestamp` — `datetime`의 서브클래스이므로 작동합니다.
- ISO-8601 문자열 — 위 허용 형식 참조.
- `int` / `float` — 기본적으로 epoch 밀리초; epoch 초는 컬럼 `unit="s"`, 나노초는 `"ns"` 설정.

현재 미지원:

- `numpy.datetime64` — 매처에 전달하기 전에 변환: `pd.Timestamp(value).to_pydatetime()` 또는 `value.astype("datetime64[ms]").astype(int)`.

## 정밀도

FlyQL의 매처는 시간 값을 **밀리초 단위**로 비교합니다. 소스 데이터의 서브밀리초 정밀도는 비교 전에 잘립니다:

- Python — `int(dt.timestamp() * 1000)`가 마이크로초를 버립니다.
- Go — `UnixMilli()`가 나노초를 버립니다.
- JavaScript — `Date`는 네이티브로 ms입니다.

나노초 정밀도가 필요한 사용자는 레코드를 bigint 나노초로 전처리하고 `unit="ns"`를 선언해야 합니다. JS 경로는 `Number`를 사용하며 `2^53` 초과 시 정밀도를 잃습니다 — 나노초 BigInt 경로는 후속 스펙을 위해 예약되어 있습니다.

## 스키마 없는 숫자

컬럼 스키마가 없는 숫자 레코드 값은 **epoch 밀리초**로 처리됩니다. 자동 감지 휴리스틱은 없습니다 — 레코드 `{"ts": 1712434800}`(관례상 epoch 초)은 `2024`가 아니라 `1970-01-20`으로 해석됩니다. epoch-초 데이터를 사용하는 사용자는 `unit="s"`를 선언해야 합니다:

```json
{ "ts": { "type": "datetime", "unit": "s" } }
```

이는 의도적으로 엄격합니다: 자동 감지 휴리스틱(예: `value < 10^10 → 초`)은 경계와 정당한 이전 날짜 값에서 실패하므로, 조용한 추측보다 명시적 선언을 선호합니다.

## 언제 무엇을 사용할까

| 의도 | 사용 | 예시 |
|------|------|------|
| 고정된 달력 날짜 또는 타임스탬프와 비교 | 문자열 리터럴 | <FlyqlCode code={`due_date < '2023-12-31'`} inline /> |
| 슬라이딩 시간 윈도우와 비교 | <FlyqlCode code={`ago()`} inline /> | <FlyqlCode code={`created_at > ago(1h)`} inline /> |
| 현재 순간과 비교 | <FlyqlCode code={`now()`} inline /> | <FlyqlCode code={`expires_at < now()`} inline /> |
| 오늘과 비교(시간대 인식) | <FlyqlCode code={`today()`} inline /> | <FlyqlCode code={`scheduled_for = today('UTC')`} inline /> |
| 일/주/월의 시작과 비교 | <FlyqlCode code={`startOf()`} inline /> | <FlyqlCode code={`event_time > startOf('week')`} inline /> |

문자열 리터럴은 일정합니다 — 같은 쿼리 문자열은 매일 같은 SQL을 반환합니다. 시간 함수는 실행 시 평가되므로, 같은 FlyQL 쿼리가 실행될 때마다 다른 시간 윈도우를 생성합니다.
