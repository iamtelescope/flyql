## Regex Matching

Use <FlyqlToken type="operator">{'~'}</FlyqlToken> to match a value against a regular expression, and <FlyqlToken type="operator">{'!~'}</FlyqlToken> to negate the match:

<FlyqlCode code={`message ~ "error.*timeout"`} />
<FlyqlCode code={`path !~ "^/health"`} />

| Operator | Meaning |
|----------|---------|
| <FlyqlToken type="operator">{'~'}</FlyqlToken> | Value matches regex pattern |
| <FlyqlToken type="operator">{'!~'}</FlyqlToken> | Value does not match regex pattern |

### Regex Pattern Syntax

The regex syntax depends on the target backend:

- **ClickHouse** — uses `match()` function (RE2 syntax)
- **PostgreSQL** — uses <FlyqlToken type="operator">{'~'}</FlyqlToken> and <FlyqlToken type="operator">{'!~'}</FlyqlToken> operators (POSIX regex)
- **StarRocks** — uses `regexp` function
- **In-memory matcher** — uses the host language's regex engine (Go `regexp`, Python `re2`, JS `RegExp`)

Quote the pattern when it contains spaces or special characters:

<FlyqlCode code={`message ~ "connection (refused|reset)"`} />
<FlyqlCode code={`host ~ "^prod-[0-9]+"`} />

### Regex Flags

You can modify regex behavior using inline flags at the start of your pattern with the `(?...)` syntax. FlyQL supports three flags that work across all backends:

#### `(?i)` — Case-Insensitive

Makes the pattern match regardless of letter case:

<FlyqlCode code={`message ~ "(?i)error"`} />
<FlyqlCode code={`level ~ "(?i)^(warn|error|fatal)$"`} />

`(?i)error` matches "Error", "ERROR", "error", etc.

#### `(?m)` — Multiline

Changes `^` and `$` to match the start and end of each **line** instead of the entire string:

<FlyqlCode code={`log ~ "(?m)^ERROR"`} />

Without `(?m)`, `^ERROR` only matches if the entire string starts with "ERROR". With `(?m)`, it also matches "INFO first\nERROR second" because "ERROR" appears at the start of a line.

#### `(?s)` — Dotall

Makes `.` match **any** character including newlines. By default, `.` does not match `\n`:

<FlyqlCode code={`log ~ "(?s)start.*end"`} />

Without `(?s)`, `start.*end` does not match "start\nend" because `.` won't cross the newline. With `(?s)`, it matches.

#### Combining Flags

Flags can be combined in a single group:

<FlyqlCode code={`log ~ "(?im)^ERROR"`} />
<FlyqlCode code={`log ~ "(?ims)^START.*END$"`} />

#### Backend Support

All three flags work across every FlyQL backend:

| Backend | `(?i)` | `(?m)` | `(?s)` |
|---------|--------|--------|--------|
| **ClickHouse** | Yes | Yes | Yes |
| **PostgreSQL** | Yes | Yes | Yes |
| **StarRocks** | Yes | Yes | Yes |
| **Go matcher** | Yes | Yes | Yes |
| **Python matcher** | Yes | Yes | Yes |
| **JavaScript matcher** | Yes | Yes | Yes |

Combine case-insensitive matching with negation to exclude patterns regardless of case:

<FlyqlCode code={`message ~ "(?i)error" and message !~ "(?i)warning"`} />

## Wildcard Matching

Use <FlyqlToken type="operator">{'*'}</FlyqlToken> at the end of an unquoted value for glob-style prefix matching:

<FlyqlCode code={`host = prod*`} />
<FlyqlCode code={`service = api-*`} />

This matches any value that starts with the given prefix. Wildcards are only supported at the end of a value — `*prod` or `pr*od` are not supported.

### Wildcard vs. Regex

| Pattern | Syntax | Example |
|---------|--------|---------|
| Wildcard (prefix) | <FlyqlCode code={`key = value*`} inline /> | <FlyqlCode code={`host = prod*`} inline /> |
| Regex (full) | <FlyqlCode code={`key ~ "pattern"`} inline /> | <FlyqlCode code={`host ~ "^prod-[0-9]+"`} inline /> |

Use wildcards for simple prefix matching. Use regex when you need more complex patterns.

## Examples

<FlyqlCode code={`message ~ "error|fail"`} />
<FlyqlCode code={`path ~ "^/api/v2"`} />
<FlyqlCode code={`path !~ "^/health"`} />
<FlyqlCode code={`host = prod*`} />
<FlyqlCode code={`host = prod* and message ~ "timeout.*" and path !~ "^/health"`} />
