FlyQL admite 10 tipos de valor: **Integer**, **BigInt**, **Float**, **String**, **Boolean**, **Null**, **Array**, **Column**, **Function** y **Parameter**. Cada valor del AST incluye un campo `value_type` explícito.

## Tipos de valor

### Integer

Números enteros dentro del rango de int64:

<FlyqlCode code={`count = 42`} />
<FlyqlCode code={`status != 200`} />
<FlyqlCode code={`age >= 18`} />

AST: `value_type: "integer"`

### BigInt

Enteros que superan el rango de int64 (Go/Python) o `MAX_SAFE_INTEGER` (JavaScript):

<FlyqlCode code={`id = 18446744073709551615`} />

AST: `value_type: "bigint"`

### Float

Números con punto decimal:

<FlyqlCode code={`price > 9.99`} />
<FlyqlCode code={`ratio = 3.14`} />
<FlyqlCode code={`temperature<=-0.5`} />

AST: `value_type: "float"`

### String

Valores de texto, con o sin comillas:

<FlyqlCode code={`level = 'error'`} />
<FlyqlCode code={`user = "John Doe"`} />
<FlyqlCode code={`message = 'hello world'`} />
<FlyqlCode code={`count = '42'`} />

AST: `value_type: "string"`

Escapa las comillas con barra invertida:

```
user = 'John
message="said \"hello\""
```

### Boolean

Las palabras clave <FlyqlToken type="boolean">true</FlyqlToken> y <FlyqlToken type="boolean">false</FlyqlToken> en minúsculas y sensibles a mayúsculas:

<FlyqlCode code={`active = true`} />
<FlyqlCode code={`debug = false`} />
<FlyqlCode code={`active != true`} />

AST: `value_type: "boolean"`, el valor es un boolean nativo.

Las variantes con mayúsculas (`True`, `FALSE`) se tratan como strings.

**Generación SQL:**
| Dialecto | FlyQL | SQL generado |
|---------|-------|--------------|
| ClickHouse | <FlyqlCode code={`field = true`} inline /> | `field = true` |
| ClickHouse | <FlyqlCode code={`field = false`} inline /> | `field = false` |
| PostgreSQL | <FlyqlCode code={`field = true`} inline /> | `field = TRUE` |
| PostgreSQL | <FlyqlCode code={`field = false`} inline /> | `field = FALSE` |
| StarRocks | <FlyqlCode code={`field = true`} inline /> | `field = true` |
| StarRocks | <FlyqlCode code={`field = false`} inline /> | `field = false` |

### Null

La palabra clave <FlyqlToken type="null">null</FlyqlToken> en minúsculas y sensible a mayúsculas:

<FlyqlCode code={`field = null`} />
<FlyqlCode code={`field != null`} />

AST: `value_type: "null"`, el valor es un null nativo.

Las variantes con mayúsculas (`Null`, `NULL`) se tratan como strings.

**Restricciones:** solo los operadores <FlyqlToken type="operator">=</FlyqlToken> y <FlyqlToken type="operator">!=</FlyqlToken> son válidos con null. Usar <FlyqlToken type="operator">{'>'}</FlyqlToken>, <FlyqlToken type="operator">{'<'}</FlyqlToken>, <FlyqlToken type="operator">{'>='}</FlyqlToken>, <FlyqlToken type="operator">{'<='}</FlyqlToken>, <FlyqlToken type="operator">{'~'}</FlyqlToken> o <FlyqlToken type="operator">{'!~'}</FlyqlToken> con null produce un error de parseo.

**Generación SQL:** <FlyqlCode code={`field = null`} inline /> genera `field IS NULL` y <FlyqlCode code={`field != null`} inline /> genera `field IS NOT NULL` en todos los dialectos.

### Array

Los arrays se usan con <FlyqlToken type="operator">in</FlyqlToken> y <FlyqlToken type="operator">not in</FlyqlToken>. Sus elementos pueden mezclar tipos:

<FlyqlCode code={`status in [200, 201, 202]`} />
<FlyqlCode code={`name in ['alice', 'bob']`} />
<FlyqlCode code={`field in [1, "hello", true, null]`} />

Cada elemento lleva su propio tipo en el array `values_types`.

## Expresiones truthy/falsy

Una clave independiente sin operador comprueba si el campo tiene un valor truthy:

<FlyqlCode code={`active`} />
<FlyqlCode code={`message and status`} />

Un valor se considera **falsy** si es:
- `null` / `None` / inexistente
- string vacío `""`
- cero `0`
- boolean <FlyqlToken type="boolean">false</FlyqlToken>

Todo lo demás es **truthy**.

Usa <FlyqlToken type="operator">not</FlyqlToken> para comprobar valores falsy:

<FlyqlCode code={`not archived`} />
<FlyqlCode code={`active and not debug`} />

Las comprobaciones truthy/falsy son distintas de las comparaciones con null: <FlyqlCode code={`not field`} inline /> coincide cuando el campo es null, vacío o cero. <FlyqlCode code={`field = null`} inline /> coincide solo cuando el campo es específicamente null.

## Espacios en blanco

FlyQL tolera espacios en blanco. Los espacios alrededor de los operadores son opcionales:

<FlyqlCode code={`status = 200`} />
<FlyqlCode code={`status =200`} />

Todas las variantes anteriores son equivalentes.

## Módulo de constantes de tipos

Todas las constantes de tipo están disponibles en el módulo `flyql.types`:

**Python:**
```python
from flyql.types import ValueType
# ValueType.INTEGER, ValueType.BIGINT, ValueType.FLOAT,
# ValueType.STRING, ValueType.BOOLEAN, ValueType.NULL, ValueType.ARRAY
```

**Go:**
```go
import "github.com/iamtelescope/flyql/golang/types"
// types.Integer, types.BigInt, types.Float,
// types.String, types.Boolean, types.Null, types.Array
```

**JavaScript:**
```javascript
import { ValueType } from 'flyql/types'
// ValueType.INTEGER, ValueType.BIGINT, ValueType.FLOAT,
// ValueType.STRING, ValueType.BOOLEAN, ValueType.NULL, ValueType.ARRAY
```

## Funciones temporales

FlyQL admite llamadas a funciones temporales como valores en el lado derecho de expresiones de comparación. Estas permiten filtros relativos al tiempo sin necesidad de codificar marcas de tiempo.

AST: `value_type: "function"`, `value` es un objeto `FunctionCall`.

### `ago(duration)`

Devuelve una marca de tiempo relativa al momento actual. Admite duraciones compuestas.

<FlyqlCode code={`timestamp > ago(1h)`} />
<FlyqlCode code={`created_at >= ago(7d)`} />
<FlyqlCode code={`updated_at > ago(1h30m)`} />

**Unidades de duración:** `s` (segundos), `m` (minutos), `h` (horas), `d` (días), `w` (semanas)

### `now()`

Devuelve la marca de tiempo actual.

<FlyqlCode code={`expires_at < now()`} />

### `today()`

Devuelve la fecha de hoy. Acepta un argumento opcional de zona horaria IANA.

<FlyqlCode code={`date = today()`} />
<FlyqlCode code={`date = today('Europe/Berlin')`} />

### `startOf(unit)`

Devuelve el inicio del día, la semana (lunes) o el mes actual. Acepta una zona horaria opcional.

<FlyqlCode code={`created_at > startOf('day')`} />
<FlyqlCode code={`created_at > startOf('week')`} />
<FlyqlCode code={`created_at > startOf('month', 'Asia/Tokyo')`} />

**Unidades válidas:** `'day'`, `'week'`, `'month'`

### Restricciones de operadores

Las funciones temporales solo pueden usarse con operadores de comparación: `=`, `!=`, `>`, `>=`, `<`, `<=`.

Usar `~` o `!~` con una función temporal produce un error de parseo. Los operadores como `in`, `has`, `like` e `ilike` utilizan rutas de análisis separadas y no son compatibles con valores de función.

### Referencia a columna sin paréntesis

Los nombres de función sin paréntesis se tratan como referencias a columnas, no como llamadas a funciones:

<FlyqlCode code={`field = ago`} />

Esto produce `value_type: "column"`, no una llamada a función.

## Parameter

Los marcadores de parámetros (`$name` para los nombrados y `$1` para los posicionales) sustituyen valores que se resolverán en tiempo de ejecución mediante `bindParams()`. Pueden aparecer en cualquier lugar donde sea válido un valor literal: dentro de listas IN, argumentos de funciones temporales y argumentos de transformers.

<FlyqlCode code={`status = $code`} />
<FlyqlCode code={`level in [$primary, 'warn']`} />
<FlyqlCode code={`created > ago($duration)`} />

AST: `value_type: "parameter"`, el valor es un objeto `Parameter` con los campos `name` y `positional`.

Consulta [Parámetros](/es/syntax/parameters/) para ver la referencia completa.

## Notas de migración

Si actualizas desde una versión anterior de FlyQL:

- **Breaking:** en Go se eliminaron `ValueTypeString` y `ValueTypeNumber`, sustituidos por `types.ValueType` con 7 valores
- **Breaking:** el campo `value_type` del AST cambió de `"string"`/`"number"` a `"integer"`/`"float"`/`"bigint"`/`"string"`/`"boolean"`/`"null"`
- **Breaking:** se eliminó la restricción de listas de tipo mixto; ahora los arrays heterogéneos son válidos
- **Nuevo:** <FlyqlToken type="boolean">true</FlyqlToken>, <FlyqlToken type="boolean">false</FlyqlToken> y <FlyqlToken type="null">null</FlyqlToken> se parsean como literales tipados, no como strings
- **Nuevo:** el array `values_types` en expresiones de listas `IN` proporciona el tipo de cada elemento
- **Nuevo:** el módulo `flyql.types` exporta las constantes de tipo canónicas
- **Nuevo:** valores de funciones temporales (`ago()`, `now()`, `today()`, `startOf()`) con `value_type: "function"`
- **Nuevo:** marcadores de parámetros (`$name`, `$1`) con `value_type: "parameter"`, resueltos por `bindParams()` en tiempo de ejecución
