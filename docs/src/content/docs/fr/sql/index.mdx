Les generateurs FlyQL produisent un fragment de clause WHERE sous forme de chaine SQL brute avec toutes les valeurs embarquees (echappees). La meme requete FlyQL genere un SQL different selon le dialecte cible.

## Dialectes supportes

- [ClickHouse](/fr/sql/clickhouse/) — pas de guillemets sur les colonnes, `match()` pour les regex, fonctions `JSONExtract` pour les chaines JSON
- [PostgreSQL](/fr/sql/postgresql/) — identifiants entre guillemets doubles, `~` pour les regex, operateurs JSONB
- [StarRocks](/fr/sql/starrocks/) — identifiants entre backticks, `regexp()` pour les regex, `parse_json()` pour les chaines JSON

## Differences principales

| Fonctionnalite | ClickHouse | PostgreSQL | StarRocks |
|---------|-----------|-----------|-----------|
| Guillemets colonnes | Aucun | `"name"` | `` `name` `` |
| Regex | `match(col, 'pat')` | `"col" ~ 'pat'` | `` regexp(`col`, 'pat') `` |
| Regex inverse | `NOT match(col, 'pat')` | `"col" !~ 'pat'` | `` NOT regexp(`col`, 'pat') `` |
| Joker | `col LIKE 'val%'` | `"col" LIKE 'val%'` | `` `col` LIKE 'val%' `` |
| Operateurs booleens | `AND` / `OR` | `AND` / `OR` | `AND` / `OR` |
| `IN []` vide | `0` | `FALSE` | `0` |
| `NOT IN []` vide | `1` | `TRUE` | `1` |
| <FlyqlToken type="operator">has</FlyqlToken> sur String | `position(col, 'v') > 0` | `position('v' in "col") > 0` | `` INSTR(`col`, 'v') > 0 `` |
| <FlyqlToken type="operator">has</FlyqlToken> sur Array | `has(col, 'v')` | `'v' = ANY("col")` | `` array_contains(`col`, 'v') `` |
| <FlyqlToken type="operator">has</FlyqlToken> sur Map | `mapContains(col, 'k')` | `"col" ? 'k'` | `` array_contains(map_keys(`col`), 'k') `` |
| <FlyqlToken type="operator">has</FlyqlToken> sur JSON | `JSONHas(toJSONString(col), 'k')` | `"col" ? 'k'` | `json_exists(...)` |

## Acces JSON — La plus grande difference

Chaque dialecte gere les colonnes JSON imbriquees de maniere tres differente :

**Entree FlyQL :** <FlyqlCode code={`data.user.name = 'john'`} inline />

**ClickHouse** (colonne JSON native) :
```sql
data.`user`.`name` = 'john'
```

**ClickHouse** (JSON stocke comme chaine) :
```sql
multiIf(JSONType(data, 'user', 'name') = 'String',
  equals(JSONExtractString(data, 'user', 'name'), 'john'), 0)
```

**PostgreSQL** (JSONB) :
```sql
(jsonb_typeof("data"->'user'->'name') = 'string'
  AND "data"->'user'->>'name' = 'john')
```

**StarRocks** (JSON natif) :
```sql
`data`->'\"user\"'->'\"name\"' = 'john'
```

Consultez la page de chaque dialecte pour des exemples complets.

Les indices de tableau FlyQL sont uniformement en base 0. Les trois dialectes utilisent des tableaux base 1 au niveau SQL, les generateurs ajoutent donc 1 a l'index flyql avant de l'emettre. La meme expression FlyQL produit des chaines SQL differentes par dialecte (a cause de la mise entre guillemets des identifiants et de la syntaxe des fonctions), mais s'evalue au meme resultat sur les memes donnees sous-jacentes.

## Format de sortie

Les generateurs retournent un fragment de clause WHERE — pas de mot-cle `WHERE`, pas de parametres substitues. Toutes les valeurs sont echappees et embarquees directement :

```
message = 'hello' and count > 10
```

Et non :

```
WHERE message = ? AND count > ?
```
