## Pertencimento a Lista

Use <FlyqlToken type="operator">in</FlyqlToken> para verificar se um valor está em uma lista, e <FlyqlToken type="operator">not in</FlyqlToken> para verificar se não está:

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['prod', 'staging']`} />

## Regras de Sintaxe

- Os valores são colocados entre colchetes <FlyqlToken type="operator">[ ]</FlyqlToken> e separados por vírgulas
- Todos os valores em uma lista devem ser do mesmo tipo (todos strings ou todos números)
- Valores de string devem estar entre aspas: <FlyqlCode code={`['a', 'b', 'c']`} inline />
- Valores numéricos não usam aspas: <FlyqlCode code={`[1, 2, 3]`} inline />
- Espaços ao redor de vírgulas e colchetes são permitidos

### Listas de Strings

<FlyqlCode code={`method in ['GET', 'POST', 'PUT']`} />
<FlyqlCode code={`env not in ['dev', 'test', 'local']`} />

Tanto aspas simples quanto duplas funcionam para valores de string.

### Listas Numéricas

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`port not in [80, 443]`} />

### Listas Vazias

Listas vazias são permitidas:

<FlyqlCode code={`status in []`} />

Uma lista <FlyqlToken type="operator">in</FlyqlToken> vazia é sempre falsa — nada corresponde a uma lista vazia.

<FlyqlCode code={`status not in []`} />

Uma lista <FlyqlToken type="operator">not in</FlyqlToken> vazia é sempre verdadeira — tudo está "fora" de uma lista vazia.

## Tipos Mistos

As listas suportam valores heterogêneos — você pode misturar strings, números, booleanos e null:

<FlyqlCode code={`field in [200, 'ok', true, null]`} />

Cada elemento carrega seu próprio tipo no array `valuesTypes` da expressão analisada (por exemplo, `["int", "string", "bool", "null"]`). Para listas vazias, `valuesTypes` é `null`.

Quando um gerador SQL processa uma lista, ele valida que os tipos dos valores da lista são compatíveis com o tipo da coluna de destino. Por exemplo, passar valores de string para uma coluna numérica gerará um erro de validação durante a geração do SQL.

## Consistência de Tipos em Listas IN

O parser **sempre** aceita listas heterogêneas — a validação de tipos vive nos geradores SQL, não no parser. Cada gerador (ClickHouse, PostgreSQL, StarRocks) aplica a mesma lógica de salto em duas camadas antes de lançar um erro.

### Camada 1 — verificação no local da chamada

Antes de invocar o validador, o gerador verifica três condições:

1. A coluna tem um **tipo declarado** (`column.flyqlType()` não está vazio e não é `Unknown`).
2. A chave **não é segmentada** (um único segmento como `category`, não `meta.category`).
3. A lista é **homogênea** — todos os elementos têm o mesmo tipo de valor (`len(set(values_types)) == 1` em Python; `new Set(valuesTypes).size === 1` em JavaScript; equivalente em Go).

Se **alguma** dessas condições for falsa, o validador é completamente ignorado e o SQL é gerado tal qual. Listas heterogêneas passam. Listas contra colunas sem tipo passam. Listas contra chaves segmentadas (caminhos JSON, índices de array) passam.

### Camada 2 — verificação de elementos

Quando a verificação no local da chamada é satisfeita, o validador itera sobre cada elemento e lança no primeiro valor incompatível, segundo este mapa de compatibilidade:

| Tipo de coluna | Tipos de elemento permitidos |
|----------------|------------------------------|
| `String`       | `string`                     |
| `Int`          | `int`, `float`               |
| `Float`        | `int`, `float`               |
| `Bool`         | `bool`, `int`                |
| `Date`         | `string`                     |

Outros tipos de coluna (`Array`, `Map`, `JSON`, `JSONString`, `Struct`, `Duration`, `Unknown`) não estão no mapa e contornam silenciosamente a verificação de elementos.

### O texto do erro

Quando a validação falha, as três implementações lançam a mensagem de erro **idêntica**:

<FlyqlCode code={`type mismatch in IN list: {column_type} column cannot contain {value_type} values`} />

Concretamente:

- Python: `FlyqlError("type mismatch in IN list: int column cannot contain string values")`
- Go: `errors.New("type mismatch in IN list: int column cannot contain string values")`
- JavaScript: `new Error("type mismatch in IN list: int column cannot contain string values")`

A subcadeia `type mismatch in IN list:` é estável nas três linguagens e nos três dialetos SQL, sendo seguro fazer grep por ela.

Para uma demonstração end-to-end executável dos três casos (correspondência homogênea, bypass heterogêneo, erro de tipo errado homogêneo), veja [Receitas — Consistência de Tipos em Listas IN](/pt-br/syntax/recipes/#consistência-de-tipos-em-listas-in).

## Parâmetros em Listas

Placeholders de parâmetros podem aparecer dentro de listas IN, misturados livremente com valores literais:

<FlyqlCode code={`status in [$x, $y, 200]`} />
<FlyqlCode code={`env not in [$current, 'dev']`} />

Os objetos `Parameter` permanecem no array `values` até que `bindParams()` os resolva. Veja [Parâmetros](/pt-br/syntax/parameters/) para a referência completa.

## Combinado com Outros Operadores

Os operadores de lista combinam com operadores booleanos como qualquer outra condição:

<FlyqlCode code={`status in [200, 201] and method in ['GET', 'POST']`} />
<FlyqlCode code={`level = 'error' or status not in [200, 201, 204]`} />
<FlyqlCode code={`(status in [500, 502, 503] or level = 'critical') and env='prod'`} />

## Exemplos

<FlyqlCode code={`status in [200, 201, 204]`} />
<FlyqlCode code={`env not in ['dev', 'test', 'staging']`} />
<FlyqlCode code={`method in ['GET', 'POST'] and status not in [401, 403, 404]`} />
