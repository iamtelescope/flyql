FlyQL различает **календарные дни** (`Type.Date`) и **моменты времени** (`Type.DateTime`) в схемах колонок и использует это различие для приведения строковых литералов ISO-8601 в matcher. SQL-генераторы, наоборот, передают строковые литералы как есть — их разбирает база данных.

## Календарный день vs. момент

`Type.Date` — только календарный день. Сравнения усекаются до `YYYYMMDD`. Поглощает «сырые» DB-типы `date` / `date32`.

`Type.DateTime` — точка на временной шкале с разрешением в миллисекунду. Сравнения выполняются в UTC-миллисекундах. Поглощает `datetime`, `datetime64`, `timestamp`, `timestamptz`, `year`.

Объявление семантического типа важно для matcher: колонка `Date`, сравниваемая с `'2026-04-06T15:30:00Z'`, перед сравнением усекает значение записи до компонент Y/M/D, тогда как колонка `DateTime` сравнивает до миллисекунды.

Если колонка, объявленная как `Type.Date`, получает значение «datetime-формы» (нативный `datetime` / `time.Time` / `Date`, либо ISO-строка с временным компонентом), FlyQL один раз на колонку выдаёт `UserWarning` (Python) / строку лога (Go) / `console.warn` (JS), предлагая миграцию на `Type.DateTime`.

## Дата как строковый литерал

Сравните колонку date/datetime со строковым литералом в формате ISO-8601. Генераторы передают литерал как есть; база данных разбирает его во время выполнения.

<FlyqlCode code={`due_date < '2023-12-31'`} />
<FlyqlCode code={`created_at >= '2023-12-31T00:00:00Z'`} />
<FlyqlCode code={`event_time >= '2023-01-01' and event_time <= '2023-12-31'`} />

В matcher объявленные схемой колонки `Date` / `DateTime` запускают автоматический разбор как литералов ISO-8601, так и значений записей. Для типичного случая никакой явной функции преобразования не требуется.

### Принимаемые формы iso8601

Семейство `iso8601` в matcher — гибкое:

- `YYYY-MM-DD` — чистая календарная дата (для Date-колонок, либо полночь в tz колонки для DateTime-колонок).
- `YYYY-MM-DDTHH:MM:SS` — наивный datetime.
- `YYYY-MM-DD HH:MM:SS` — наивный datetime с пробелом-разделителем (нормализуется перед разбором).
- `YYYY-MM-DDTHH:MM:SS.ffffff` — необязательный суб-секундный хвост (только ms-разрешение — см. «Точность» ниже).
- Любая из форм выше с суффиксом смещения `Z` или `±HH:MM`.

### Метаданные колонки для наивных литералов

Объявление колонки `DateTime` может нести опциональные метаданные:

- `tz` — имя часового пояса IANA, используемое для интерпретации наивных ISO-строк. Откатывается на `default_timezone` Evaluator, затем на `"UTC"`.
- `unit` — `"ms"` (по умолчанию), `"s"` или `"ns"`. Применяется к числовым значениям записи; нечисловые значения её игнорируют.

```json
{
  "ts":        { "type": "datetime", "tz": "Europe/Moscow", "unit": "s" },
  "event_day": { "type": "date" }
}
```

:::note[Строки должны быть в кавычках]
Литералы дат — это строки, и их **обязательно** заключать в кавычки. Без кавычек FlyQL интерпретирует значение как ссылку на колонку. <FlyqlCode code={`due_date < 2023-12-31`} inline /> разобрался бы как арифметика над колонкой с именем `2023`, а не как литерал даты. См. [Значения и типы — String](/ru/syntax/values/#string).
:::

## Временные функции

Для относительных запросов о времени — «за последний час», «с начала недели», «с сегодня» — FlyQL предоставляет встроенные временные функции: <FlyqlCode code={`ago(...)`} inline />, <FlyqlCode code={`now()`} inline />, <FlyqlCode code={`today()`} inline /> и <FlyqlCode code={`startOf(...)`} inline />.

<FlyqlCode code={`created_at > ago(1h)`} />
<FlyqlCode code={`updated_at >= ago(7d)`} />
<FlyqlCode code={`expires_at < now()`} />
<FlyqlCode code={`event_time > startOf('day')`} />
<FlyqlCode code={`scheduled_for >= today('Europe/Berlin')`} />

Полная грамматика (единицы, правила порядка, аргументы часового пояса) описана в разделе [Значения и типы — Временные функции](/ru/syntax/values/#temporal-functions).

## Летнее время (DST)

Когда наивный datetime-литерал (без смещения, без `Z`) интерпретируется в часовом поясе с DST, FlyQL выбирает однозначное разрешение, заданное спецификацией:

- **Откат (неоднозначные настенные часы)** — выбирается более раннее вхождение (`fold=0` в семантике Python). Пример: `'2026-11-01 01:30:00'` в `America/New_York` разрешается в 05:30 UTC (EDT), а не в 06:30 UTC (EST).
- **Опережение (несуществующие настенные часы)** — значение отвергается. Matcher возвращает для такой записи `false` (пропускает). Пример: `'2026-03-08 02:30:00'` в `America/New_York` не существует.

Записи, попавшие в «провал» опережения, трактуются как не поддающиеся разбору, а не молча нормализуются вперёд.

## Поддерживаемые типы значений записи (Python)

- `datetime.datetime` — с таймзоной или наивный (наивный использует `tz` колонки или дефолт Evaluator).
- `datetime.date` — только календарный день.
- `pandas.Timestamp` — работает, так как является подклассом `datetime`.
- Строки ISO-8601 — см. принимаемые формы выше.
- `int` / `float` — по умолчанию epoch-миллисекунды; задайте у колонки `unit="s"` для epoch-секунд или `"ns"` для наносекунд.

Сегодня не поддерживаются:

- `numpy.datetime64` — конвертируйте перед передачей в matcher: `pd.Timestamp(value).to_pydatetime()` или `value.astype("datetime64[ms]").astype(int)`.

## Точность

Matcher FlyQL сравнивает временные значения с **миллисекундной гранулярностью**. Суб-миллисекундная точность в исходных данных усекается перед сравнением:

- Python — `int(dt.timestamp() * 1000)` отбрасывает микросекунды.
- Go — `UnixMilli()` отбрасывает наносекунды.
- JavaScript — `Date` нативно хранит ms.

Пользователи, которым нужна наносекундная точность, должны предварительно привести записи к bigint-наносекундам и объявить `unit="ns"`. JS-ветка использует `Number`, который выше `2^53` теряет точность — путь с BigInt для наносекунд зарезервирован для следующей спецификации.

## Числа без схемы

Числовые значения записей без схемы колонки трактуются как **epoch-миллисекунды**. Никакой автодетект-эвристики — запись `{"ts": 1712434800}` (по общепринятой конвенции — epoch-секунды) интерпретируется как `1970-01-20`, а не как `2024`. Пользователи с данными в epoch-секундах ОБЯЗАНЫ объявить `unit="s"`:

```json
{ "ts": { "type": "datetime", "unit": "s" } }
```

Это строгость по замыслу: автодетект-эвристика (например, `value < 10^10 → секунды`) спотыкается на границе и на валидных «старых» датах, поэтому мы предпочитаем явные объявления тихим догадкам.

## Когда что использовать

| Намерение | Используйте | Пример |
|-----------|-------------|--------|
| Сравнить с фиксированной календарной датой или меткой времени | Строковый литерал | <FlyqlCode code={`due_date < '2023-12-31'`} inline /> |
| Сравнить со скользящим временным окном | <FlyqlCode code={`ago()`} inline /> | <FlyqlCode code={`created_at > ago(1h)`} inline /> |
| Сравнить с текущим моментом | <FlyqlCode code={`now()`} inline /> | <FlyqlCode code={`expires_at < now()`} inline /> |
| Сравнить с сегодняшним днём (с часовым поясом) | <FlyqlCode code={`today()`} inline /> | <FlyqlCode code={`scheduled_for = today('UTC')`} inline /> |
| Сравнить с началом дня / недели / месяца | <FlyqlCode code={`startOf()`} inline /> | <FlyqlCode code={`event_time > startOf('week')`} inline /> |

Строковые литералы остаются неизменными — одна и та же строка запроса каждый день возвращает один и тот же SQL. Временные функции вычисляются во время выполнения, поэтому один и тот же запрос FlyQL каждый раз даёт другое временное окно.
