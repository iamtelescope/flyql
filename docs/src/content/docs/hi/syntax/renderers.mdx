रेंडरर एलियास किए गए SELECT कॉलम में डिस्प्ले मेटाडेटा जोड़ते हैं। ये flyql द्वारा पार्स और मान्य किए जाते हैं लेकिन **SQL या मैचर आउटपुट को कभी प्रभावित नहीं करते**। [ट्रांसफ़ॉर्मर](/syntax/transformers/) के विपरीत, जो डेटाबेस से लौटाए गए डेटा को बदलते हैं, रेंडरर यह वर्णन करते हैं कि आपके एप्लिकेशन को मान कैसे प्रदर्शित करना चाहिए — व्याख्या पूरी तरह आपकी ज़िम्मेदारी है।

## पोस्ट-एलियास पाइप सिंटैक्स

रेंडरर चेन एलियास के **बाद** आती है, पाइप (`|`) से अलग:

```
url as link|href("http://{{value}}")
```

एलियास आवश्यक है। बिना एलियास वाला कॉलम रेंडरर नहीं ले सकता।

## चेन किए गए रेंडरर

अतिरिक्त पाइप के साथ कई रेंडरर जोड़ें:

```
url as link|href("http://{{value}}")|badge("info")
```

रेंडरर बाएँ से दाएँ एक क्रमबद्ध सूची में पार्स किए जाते हैं। चेन शब्दार्थ पूरी तरह आपके एप्लिकेशन पर निर्भर हैं — flyql रेंडरर पर कोई इनपुट/आउटपुट प्रकार प्रणाली लागू नहीं करता।

## रेंडरर और ट्रांसफ़ॉर्मर एक साथ

ट्रांसफ़ॉर्मर एलियास से पहले चलते हैं; रेंडरर बाद में:

```
url|upper as u|href("https://{{value}}")
```

ट्रांसफ़ॉर्मर `upper` जनरेट किए गए SQL को प्रभावित करता है। रेंडरर `href` नहीं करता।

## बिना आर्गुमेंट वाले रेंडरर

बिना आर्गुमेंट वाले रेंडरर उसी तरह पार्स होते हैं जैसे बिना आर्गुमेंट वाले ट्रांसफ़ॉर्मर:

```
url as link|plain
```

## रेंडरर सक्षम करना

रेंडरर पार्सिंग एक पार्सर क्षमता फ़्लैग के माध्यम से ऑप्ट-इन है:

**Python**
```python
from flyql.columns import parse
cols = parse('url as link|href("/{{value}}")',
             capabilities={"transformers": True, "renderers": True})
```

**Go**
```go
import "github.com/iamtelescope/flyql/golang/columns"
caps := columns.Capabilities{Transformers: true, Renderers: true}
cols, _ := columns.Parse(`url as link|href("/{{value}}")`, caps)
```

**JavaScript**
```javascript
import { parse } from 'flyql/columns'
const cols = parse('url as link|href("/{{value}}")',
    { transformers: true, renderers: true })
```

जब `renderers` `false` (डिफ़ॉल्ट) हो, तो पोस्ट-एलियास `|` errno `11` और संदेश `"renderers are not enabled"` के साथ `ParserError` उत्पन्न करता है।

## खाली डिफ़ॉल्ट रजिस्ट्री

[`TransformerRegistry.default_registry()`](/advanced/custom-transformers/) के विपरीत, जो चार अंतर्निहित ट्रांसफ़ॉर्मर (`upper`, `lower`, `len`, `split`) के साथ आती है, `RendererRegistry.default_registry()` **डिज़ाइन द्वारा खाली** है। flyql कोर आपकी रेंडरिंग तकनीक (HTML, markdown, React, Vue, आदि) के बारे में कोई धारणा नहीं बनाता। वैलिडेटर द्वारा पहचाने जाने से पहले आपको [अपने रेंडरर पंजीकृत](/advanced/custom-renderers/) करने होंगे।

इसका मतलब है कि रेंडरर को वैलिडेटर या एडिटर में उपयोगी बनाने के लिए आपको एक रजिस्ट्री प्रदान करनी होगी; अन्यथा हर संदर्भित रेंडरर `unknown_renderer` डायग्नोस्टिक उत्पन्न करेगा।

## रेंडरर बनाम ट्रांसफ़ॉर्मर

| विषय                      | ट्रांसफ़ॉर्मर        | रेंडरर               |
| ------------------------- | -------------------- | -------------------- |
| सिंटैक्स में स्थिति       | एलियास से पहले       | एलियास के **बाद**    |
| जनरेट किए गए SQL को प्रभावित करता है | हाँ | **नहीं**             |
| मैचर आउटपुट को प्रभावित करता है | हाँ | **नहीं**             |
| डिफ़ॉल्ट रजिस्ट्री       | 4 अंतर्निहित         | खाली                 |
| इनपुट/आउटपुट प्रकार प्रणाली | हाँ (चेन टाइपिंग) | नहीं                 |
| व्याख्या                  | flyql निष्पादित करता है | आपका ऐप निष्पादित करता है |

कस्टम रेंडरर पंजीकरण और मान्यकरण हुक के लिए [कस्टम रेंडरर](/advanced/custom-renderers/) देखें।
